# Opcode-трасса 3-limb Montgomery/CIOS умножения MNT4-753

## 1. Что измеряется

В документе фиксируется стоимость функции `BigIntMNT.montMul3`. Функция
умножает два элемента поля MNT4-753, представленных тремя 256-битными словами,
и выполняет Montgomery-редукцию по схеме CIOS.

Production-код находится в:

```text
arithmetic/mnt4_3limb/src/BigIntMNT.sol
```

## 2. Почему недостаточно границы `18 * 22`

Абстрактная схема содержит девять произведений `a_i * b_j` и девять
произведений `m_i * q_j` редукции. EVM не имеет инструкции `MULHI`, поэтому
полное 512-битное произведение восстанавливается через:

```yul
lo := mul(u, v)
mm := mulmod(u, v, not(0))
hi := sub(sub(mm, lo), lt(mm, lo))
```

Стоимость этой связки равна `22 gas`. Но произведение `18 * 22 = 396 gas`
учитывает только восстановление полных произведений. Оно не учитывает
накопление частичных сумм, переносы, сдвиги окна CIOS, финальную
нормализацию и механику стековой EVM.

## 3. Уже выполненная низкоуровневая оптимизация

Три произведения `m_i * q_0` не вычисляются полностью. После выбора:

```text
m_i = t_0 * (-q^{-1}) mod 2^256
```

младшее слово обнуляется по модулю `2^256` и затем отбрасывается. Поэтому в
оптимизированном графе остаются:

| Операция | Количество |
|---|---:|
| `MULMOD` для старших половин | 18 |
| Полезные младшие `MUL` | 15 |
| `MUL` для коэффициентов `m_i` | 3 |

Итого динамическая трасса содержит `18 MUL` и `18 MULMOD`.

## 4. Динамическая трасса

Для внешнего stack-API вызова `montMul3` использован локальный Anvil с
`--steps-tracing`. Вызов выполнен на full-width словах поля.

Полная трасса внешнего вызова:

| Показатель | Значение |
|---|---:|
| Исполненные EVM-инструкции | 1,012 |
| Gas без intrinsic transaction gas | 3,316 |
| Развернутое CIOS-тело | 2,843 |
| ABI-диспетчер, декодирование и возврат | 473 |

Opcode-состав развернутого CIOS-тела:

| Группа | Количество | Gas |
|---|---:|---:|
| `MUL` | 18 | 90 |
| `MULMOD` | 18 | 144 |
| `ADD` | 63 | 189 |
| `SUB` | 36 | 108 |
| `LT` | 61 | 183 |
| `JUMPI` | 19 | 190 |
| `MLOAD`, `MSTORE`, `CODECOPY` | `40 + 20 + 20` | 300 |
| `PUSH`, `DUP`, `SWAP` и остальные | - | 1,639 |
| **Итого** | - | **2,843** |

Числа относятся к одному конкретному full-width вызову. Количество
исполненных ветвей может немного меняться в зависимости от переносов и
финального сравнения с модулем, но структура стоимости сохраняется.

## 5. Измерение в библиотечном hot path

Внутренний Foundry-бенчмарк вызывает `montMul3` непосредственно в цикле и не
платит за внешний ABI-переход на каждой итерации:

```bash
cd arithmetic/mnt4_3limb
forge test --match-test testGasBench_montMul3_internal -vv
forge test --match-test testGasBench_montMul3_external_stack -vv
```

Результат:

| Режим | Gas/op |
|---|---:|
| Внутренний hot path | 2,959 |
| Внешний stack-ABI вызов в Foundry-цикле | 3,976 |
| Одиночный внешний вызов по opcode-трассе | 3,316 |

Внешние измерения отличаются методикой: Foundry-цикл дополнительно учитывает
стоимость повторных внешних вызовов из тестового контракта.

## 6. Корректная формулировка результата

`2,959 gas/op` не объявляется абсолютной нижней границей среди всех
теоретически возможных EVM-bytecode программ. Для такого утверждения
потребовалась бы отдельная задача супероптимизации байткода.

Подтвержден другой, практически достаточный вывод:

1. Production-код полностью развернут и не материализует промежуточные
   массивы.
2. Из него уже удалены три бесполезных младших произведения `m_i * q_0`.
3. Основную остаточную стоимость создают обязательные переносы и обслуживание
   одновременно живых 256-битных слов на стековой машине EVM.
4. Реализованные альтернативы Barrett, FIOS, Comba/SOS, branchless reduction,
   branchless final subtraction и skip-dead-`t0` не дали улучшения хотя бы на
   10%.

Следовательно, `2,959 gas/op` является лучшим подтвержденным практическим
результатом в исследованном классе реализаций. Существенный неиспользованный
алгоритмический резерв не обнаружен.
