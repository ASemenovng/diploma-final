## 2. Главный разрыв: заявлен DEEP-FRI, но в gas-модели он удален

В разделе 5.4 определен DEEP-полином:

\[
D(X)=
\sum_{j=0}^{31}\alpha^j
\frac{C_j^{\mathrm{tr}}(X)-C_j^{\mathrm{tr}}(z)}{X-z}
+
\sum_{j=0}^{31}\alpha^{32+j}
\frac{C_j^{\mathrm{tr}}(X)-C_j^{\mathrm{tr}}(\omega z)}{X-\omega z}
+
\sum_{j=0}^{7}\alpha^{64+j}
\frac{C_j^{\mathrm{qt}}(X)-C_j^{\mathrm{qt}}(z)}{X-z}.
\]

Но в разделе 9.5 после оценки стоимости предлагается не вычислять
\(D(x)\), а проверять обычный батчированный полином:

\[
B(X)=\sum_j\eta^j C_j(X).
\]

Это уже не DEEP-FRI. После такой замены нельзя использовать DEEP-FRI
soundness theorem.

Нужно выбрать один вариант.

### Вариант A. Реальный DEEP-FRI

Нужно сохранить DEEP-полином и вывести оптимальную формулу его проверки.
Исследовать возможность предварительного batching:

\[
T_\alpha(X)=\sum_{j=0}^{31}\alpha^j C_j^{\mathrm{tr}}(X),
\]

\[
Q_\alpha(X)=\sum_{j=0}^{7}\alpha^{64+j}C_j^{\mathrm{qt}}(X),
\]

чтобы вычислять \(D(x)\) через несколько батчированных значений, а не
72 независимых дроби. Для выбранной формулы:

- доказать эквивалентность исходному DEEP-полиному;
- описать committed oracles;
- описать openings;
- посчитать операции;
- обновить transcript;
- обновить `proof.bin`;
- обновить soundness;
- обновить gas.

### Вариант B. Обычный FRI

Удалить ссылки на DEEP-FRI и выполнить soundness-анализ обычного FRI без
использования улучшенной оценки DEEP-FRI.

Нужно выбрать production-кандидат и обосновать выбор.

## 3. Арифметическая ошибка: quotient по \(Z\) нельзя вычислять verifier-ом

В разделе 4.3 написано:

\[
J_i(\zeta)=
\frac{A(\zeta)B(\zeta)-C(\zeta)}{\zeta^4-13}.
\]

Если verifier сам вычисляет \(J_i(\zeta)\) после получения \(\zeta\), проверка

\[
A(\zeta)B(\zeta)-C(\zeta)
=
J_i(\zeta)(\zeta^4-13)
\]

становится тождественно истинной для любых \(A,B,C\), если
\(\zeta^4-13\ne0\). Такая проверка не доказывает корректность арифметики в
\(\mathbb F_{q^4}\).

Нужно выбрать корректный вариант.

### Вариант A. Коммитить quotient-коэффициенты

Для каждого короткого отношения:

\[
A(Z)B(Z)-C(Z)=J(Z)(Z^4-13)
\]

коэффициенты \(J(Z)\) должны быть witness-столбцами, зафиксированными до
получения \(\zeta\). Тогда verifier раскрывает \(J(\zeta)\) из committed
столбцов.

### Вариант B. Проверять координатные формулы

Можно отказаться от случайной подстановки \(Z=\zeta\) и записать явные четыре
координатных равенства редуцированного умножения в
\(\mathbb F_q[Z]/(Z^4-13)\). Тогда отдельные \(J_i\) не нужны.

Нужно сравнить оба варианта по:

- числу столбцов;
- степени constraints;
- размеру proof;
- calldata;
- runtime gas;
- сложности Rust backend.

После сравнения выбрать один.

## 4. Prepared divisor table пока описана некорректно

В версии 2.0 сначала корректно отмечено, что отдельная линия зависит от
фиксированных коэффициентов и переменных координат \(P\). Но затем весь
блочный множитель:

\[
G_t^Q(P;Z)=
\prod_{h=0}^4
\left(
D_{5t+h}^Q(P;Z)
\cdot
[A_{5t+h}^Q(P;Z)]^{[\delta_{5t+h}\ne0]}
\right)^{2^{4-h}}
\]

заменяется линейным выражением:

\[
G_t^Q(P;\zeta)=
\sum_{k=0}^3
(\lambda_{t,k}^Q x_P+\nu_{t,k}^Q)\zeta^k.
\]

В общем случае это неверно: произведение и степени нескольких line
evaluations не остаются линейными по \(x_P,y_P\).

Нужно вывести реальную MNT4-753 конструкцию:

1. взять точные prepared projective line coefficients из arkworks MNT4;
2. сопоставить их sparse line format текущего прямого verifier-а;
3. определить, какие коэффициенты зависят только от \(Q,S\);
4. определить точную формулу evaluation в точках \(P,-R\);
5. определить, как объединяются пять шагов блока;
6. определить набор мономов по \(x_P,y_P\), возникающий после объединения;
7. выбрать оптимальный способ:
   - коммитить коэффициенты блочного делителя;
   - вводить дополнительные промежуточные столбцы;
   - либо использовать иной корректный формат;
8. пересчитать число fixed columns, размер `root_fixed` tree, proof и gas.

Нельзя использовать приблизительный affine slope формат с делениями, если
production backend и текущий hot path основаны на projective prepared lines.

## 5. Fixed columns и OOD evaluation не связаны с root_fixed

Merkle root таблицы значений на LDE-домене подтверждает opening в точке домена,
но OOD challenge:

\[
z\notin D_{\mathrm{LDE}}
\]

не является индексом Merkle-таблицы.

В OOD constraint check нужны значения fixed columns в точке \(z\). Версия 2.0
не объясняет, как verifier получает их и почему они соответствуют
`root_fixed`.

Нужно выбрать и формализовать:

- явные формулы selectors в точке \(z\);
- канонические коэффициенты fixed polynomials в bytecode;
- DEEP-binding fixed columns;
- отдельный PCS-opening;
- либо иной корректный путь.

Для каждого fixed column указать способ OOD evaluation и стоимость.

## 6. Proof openings неполны для проверки первого FRI fold

Чтобы проверить первый fold:

\[
p_{\mathrm{next}}(x^2)=
\frac{p(x)+p(-x)}{2}
+
\rho\frac{p(x)-p(-x)}{2x},
\]

verifier должен получить связанные с commitments значения исходного
DEEP-полинома в \(x\) и \(-x\).

Если \(D(x)\) вычисляется из trace/quotient openings, требуются openings
исходных committed таблиц как минимум в обеих точках \(x\) и \(-x\).

В текущем `proof.bin` на query передается только один:

```text
trace_values_at_idx
quotient_values_at_idx
fixed_values_at_idx
```

и один `fri_value (sibling в fold)`, который сам по себе не связан с
`root_trace` и `root_quotient`.

Нужно:

- определить точный слой 0;
- определить, существует ли отдельный `root_deep`;
- определить openings для \(x\) и \(-x\);
- доказать consistency между trace/quotient openings и первым FRI layer;
- обновить transcript;
- обновить proof layout;
- пересчитать proof bytes и gas.

## 7. Quotient tree описан противоречиво

После подстановки \(Z=\zeta\) composition quotient:

\[
Q(X)=
\frac{\mathcal N_{\mathrm{mix}}(X)}{X^{128}-1}
\]

является полиномом над \(\mathbb F_q\). Его сегменты
\(Q_0(X),Q_1(X)\) должны содержать по одному элементу \(\mathbb F_q\) на
строку.

Но в версии 2.0 leaf quotient содержит:

```text
Q0_0 ... Q0_3
Q1_0 ... Q1_3
```

то есть восемь координат. Нужно определить, какая модель выбрана:

- quotient строится после \(\zeta\) как скалярный полином над
  \(\mathbb F_q\);
- либо quotient сохраняет координаты расширенного поля.

После выбора переписать формулы, degree bounds, листы и газ.

## 8. Граничные constraints также содержат неподтвержденные quotient values

В формулах \(\mathcal B_3,\mathcal B_4,\mathcal B_5\) используются:

\[
J_{\mathrm{last}}(\zeta),
\qquad
J_{\mathrm{tail}}(\zeta),
\qquad
J_{\mathrm{frob}}(\zeta).
\]

Они не включены в committed witness columns. Это та же проблема, что в
разделе 3.

Нужно включить их в выбранную корректную схему:

- committed quotient coefficients;
- либо координатные constraints.

## 9. Селекторы и множество \(\mathcal K\) не специфицированы точно

Версия 2.0 использует:

```text
S_step

S_tail
S_frob
S_init
S_final
S_pad
S_nu for nu in K
```

Но:

- leaf fixed имеет приблизительный размер `≈18 × 96`;
- множество \(\mathcal K\) не перечислено;
- число selectors не совпадает с приблизительным размером листа;
- chain проверки \(c^\nu\) отсутствует.

Нужно:

1. вычислить точное множество \(\mathcal K\) по реальным 377 signed digits;
2. привести таблицу \(\kappa_t\) по блокам;
3. описать committed либо публичные значения \(c^\nu\);
4. доказать их связь с \(c\);
5. перечислить все selector columns;
6. указать точный размер fixed leaf.

## 10. Генерация challenges не имеет строгой верхней границы

Для rejection sampling ожидаемое число попыток действительно равно примерно:

\[
\frac{2^{753}}q\approx1.13081.
\]

Но утверждение `Worst-case on-chain: <= 5 итераций` неверно. Вероятность более
пяти попыток мала, но не равна нулю.

Нужно выбрать:

- фиксированный cap с явной вероятностью отказа корректного proof;
- получение challenge из ограниченного подмножества без rejection sampling;
- иной детерминированный метод.

Включить вероятность отказа в модель и явно описать Solidity-поведение.

## 11. Query indices должны быть уникальными либо строго проанализированными

В версии 2.0 дубликаты не фильтруются. При 64 индексах из 2048 вероятность
хотя бы одного совпадения близка к \(1-e^{-64\cdot63/(2\cdot2048)}\), то есть
около \(0.63\), а не гарантированный один дубликат.

Нужно:

- детерминированно получать 64 уникальных индекса;
- либо доказать soundness с распределением числа уникальных индексов;
- обновить transcript и gas.

## 12. Soundness budget пока не достигает заявленного порога

В версии 2.0 получено:

\[
\varepsilon_{\mathrm{total}}
\le
2^{-127.99\ldots}.
\]

Это строго больше \(2^{-128}\), а затем ошибочно названо достижением порога.

Нужно:

- зафиксировать корректный критерий: например не менее 128 бит
  криптографической стойкости в принятой модели;
- либо выбрать параметры с формальным запасом;
- не складывать эвристические оценки с точными без пояснений;
- учесть ошибки challenge generation и duplicate indices;
- привести ссылку на конкретную теорему DEEP-FRI и корректно подставить
  параметры.

## 13. Gas-модель пока нельзя считать воспроизводимой

Итог \(\approx24.7\)M gas преждевременен, потому что не учтены исправления выше.

Дополнительно:

1. `root_fixed` leaf size приблизителен;
2. используется отдельный auth path на каждый query вместо заявленного
   multiproof;
3. proof layout не содержит openings для \(x\) и \(-x\);
4. расчет не включает корректный DEEP layer 0;
5. OOD fixed evaluations не посчитаны;
6. chain проверки \(c^\nu\) не посчитана;
7. `Fq inversion ≈ 10K` не подтверждена реальным benchmark;
8. Merkle hashing не учитывает domain byte, index encoding и memory expansion;
9. calldata содержит не только криптографически случайные ненулевые байты;
10. curve membership и canonical checks требуют реального benchmark;
11. фиксированный размер `575620` изменится после исправления layout.

Нужно предоставить Rust-утилиту, которая:

- строит точный layout;
- генерирует fixture;
- печатает число байтов каждого сегмента;
- считает нулевые и ненулевые calldata bytes;
- считает точное число Merkle siblings;
- считает число операций verifier-а.
