# Задание для deep agent: G2/twist и полноценная pairing-инстанциация для lollipop-305

## 1. Зачем нужно исследование

В дипломном проекте исследуется, можно ли заменить MNT4-753 более легкой для EVM кривой из семейства ePrint 2024/1627 и тем самым получить меньшую стоимость арифметики, цикла Миллера и проверки уравнения сопряжений.

Текущая MNT4-753 реализация показывает, что даже после оптимизаций вычисление остается дорогим:

| Режим MNT4-753 | Gas |
|---|---:|
| Полное fixed-Q on-chain вычисление | около `259M` |
| Prepared sparse blob/code-shards, одно сопряжение | около `79.5M--80.2M` |
| Article640 residue equation, code-shards | около `93.7M` |

Идея lollipop-направления: проверить, дает ли меньшее поле практический выигрыш. Для исследовательского кандидата `lollipop-305-158` уже получено, что базовая арифметика дешевле MNT4-753 примерно в `2.8x--3.3x` на уровне `Fp/Fp2/Fp4` после low-level оптимизаций. Но полный verifier нельзя завершить без строгой математической фиксации второй pairing-группы.

Нужно решить именно этот блокер.

## 3. Текущие параметры, которые нельзя смешивать

Из ePrint 2024/1627 Appendix A Example 1 и авторских Magma/GP-скриптов используется параметр:

```text

```

Два 305-битных простых числа:

```text
p = x^2 - x + 1
  = 64064760444466402482617092084437280876782408929523650941985296571943203113725143542535221603

q = x^2 + 1
  = 64064760444466402482617092084437280876782408937527697446377084679579090118008868997013766277
```

Подгрупповой порядок для ordinary stick-curve:

```text

```

Проверено:

```text
bitlen(p) = 305
bitlen(q) = 305
bitlen(r) = 158
p^4 = 1 mod r
p^1 != 1 mod r
p^2 != 1 mod r
```

Для stick-curve используется:

```text
E/Fp: y^2 = x^3 + a*x + b

a = 11875228336988574493882712067711066361723405878662955689469453989104183434229646197784790728
b = 44530775641606776770556944911003809865281631340093296710155954546346967839351469608314023792
```

Башня расширений в текущем коде:

```text
Fp2 = Fp[u] / (u^2 + 1),      u^2 = -1
Fp4 = Fp2[v] / (v^2 - u),     v^2 = u
```

Важно: в статье также встречаются другие объекты lollipop/cycle construction, например supersingular cycle-компоненты. Агент должен явно разделить роли этих кривых и не смешивать их со stick-curve `E/Fp`, используемой в текущем verifier-направлении.

## 4. Главный исследовательский вопрос

Нужно определить, существует ли и как именно задается корректная невырожденная pairing-инстанциация для выбранной `lollipop-305-158` постановки:

```text
G1 = <P> subset E(Fp)[r]
G2 = <Q> subset E(Fp4)[r]
```

или через twist-модель:

```text
G2 subset E_twist(...)[r]
```

с явным embedding/twist map в `E(Fp4)`.

Для найденных генераторов должно выполняться:

```text
P != O,
Q != O,
rP = O,
rQ = O,
e(P,Q) != 1.
```

Если такой `G2` для выбранной постановки не может быть построен или не следует из статьи, нужно не подбирать произвольные точки, а строго объяснить блокер.

## 5. Что именно нужно сделать

### 5.1. Разобрать математику статьи ePrint 2024/1627

Нужно ответить:

1. Какие именно кривые входят в пример `lollipop-305-158`?
2. Сколько там кривых фактически используется и каковы их роли?
3. Какая из них является stick-curve для pairing-friendly проверки?
4. Какая кривая/подгруппа должна использоваться как `G1`?
5. Какая кривая/подгруппа должна использоваться как `G2`?
6. Нужно ли для pairing verifier вычислять один Miller loop или несколько Miller loop по разным кривым lollipop-конструкции?
7. Что является embedding degree для выбранного pairing, и почему?

### 5.2. Построить или опровергнуть `G2`

Нужно зафиксировать один из двух результатов.

**Результат A: G2 найден.** Тогда дать:

- точное поле, над которым живет `G2`;
- twist equation, если используется twist;
- twist/embedding map в `E(Fp4)`;
- cofactor `h2`;
- алгоритм получения `Q`;
- координаты генератора `Q`;
- доказательство `rQ=O`;
- проверку `Q != O`;
- проверку `e(P,Q) != 1`.

**Результат B: G2 не найден или постановка некорректна.** Тогда дать:

- точную причину;
- какое предположение ломается;
- почему текущий smoke-подход с embedded base-field point недостаточен;
- что именно нужно изменить: кривую, поле расширения, pairing type, параметры или весь lollipop-кандидат.

### 5.3. Зафиксировать pairing algorithm

Если `G2` найден, нужно описать:

- используется Tate, ate или другой pairing;
- loop scalar;
- bit/NAF schedule;
- line-function formulas;
- denominator handling;
- final exponent:

```text
E_final = (p^k - 1) / r
```

- как именно проверяется reduced pairing:

```text
Y = f_loop(P,Q)^E_final,
Y^r = 1,
Y != 1 для выбранных генераторов.
```

Если используется ate-pairing, нужно доказать, почему loop scalar корректен для данной кривой, а не просто использовать `x-1` по аналогии.

### 5.4. Подготовить воспроизводимый скрипт

Нужен Sage, Magma или PARI/GP скрипт, который:

1. строит `Fp`, `Fp2`, `Fp4`;
2. строит curve/twist;
3. строит `G1` и `G2`;
4. проверяет subgroup membership;
5. считает pairing;
6. проверяет невырожденность;
7. печатает JSON test vector.

Скрипт должен быть достаточно конкретным, чтобы по нему можно было обновить Rust backend без математических догадок.

## 6. Что не считается приемлемым ответом

Нельзя:

- просто взять `Q in E(Fp)` и вложить его в `E(Fp4)` без доказательства невырожденности;
- использовать smoke-result, где reduced pairing может быть `1`, как полноценный pairing;
- писать “нужен twist” без координат, формул и проверяемого скрипта;
- утверждать production-ready безопасность для `lollipop-305-158`;
- подменять задачу общей теорией без конкретных параметров `lollipop-305-158`.

## 7. Что делать после результата агента

Если агент выдает результат A, то следующий этап реализации в проекте:

1. обновить `lollipop305_research/rust_backend`;
2. добавить generator `P`, generator `Q`, twist map;
3. построить full prepared Miller witness;
4. реализовать Solidity/Yul hot-path verifier по аналогии с `article640_mnt4_verifier`;
5. измерить:
   - Miller core;
   - Miller + direct final exponentiation;
   - Miller + article640 residue final exponentiation;
   - calldata/cache size;
   - сравнение с MNT4-753.

Если агент выдает результат B, то следующий этап диплома:

1. оставить lollipop-305 как arithmetic research result;
2. явно написать, что арифметика дешевле, но полный pairing verifier заблокирован отсутствием зафиксированной `G2/twist` спецификации для выбранной постановки;
3. указать, что `lollipop-305-158` все равно не production-secure, а более безопасные lollipop-кандидаты имеют больший размер поля и могут потерять gas-выигрыш.

## 8. Ограничение по безопасности

`lollipop-305-158` не является production-кандидатом. Его цель в проекте — исследовательская: проверить, дает ли меньшее поле реальное снижение gas и стоит ли дальше рассматривать более крупные lollipop-кривые.

Агент должен отдельно указать, как изменится стоимость при переходе к production-secure параметрам из статьи: какие размеры полей потребуются и почему арифметика там может снова стать дорогой.

## 9. Definition of Done для исследования

Исследование считается завершенным только если есть один из двух строгих итогов.

**DoD A: полная математическая спецификация.** Есть `P`, `Q`, twist/embedding, loop scalar, final exponent, проверяемый Sage/Magma/GP скрипт, JSON vectors, и проверка `e(P,Q) != 1`.

**DoD B: формальный stop-result.** Доказано, почему для текущей постановки нельзя корректно завершить pairing verifier без смены параметров/кривой/модели `G2`, и указано, что именно нужно изменить.
