# Оценки сложности алгоритмов MNT4-753

## 1. Цель документа

Этот документ закрывает замечание о том, что для реализованных алгоритмов нужны оценки сложности: не только итоговые gas-измерения, но и объяснение, из каких базовых операций складывается стоимость.

Документ дополняет `ARITHMETIC_ALGORITHM_STUDY_RU.md`. Если тот отчет отвечает на вопрос "какой вариант арифметики выбран и почему", то этот отчет отвечает на вопрос "как из стоимости базовой арифметики получается стоимость цикла Миллера, финальной экспоненты, полной on-chain реализации и residue-verifier".

Покрываются:

- базовые операции EVM-реализации поля MNT4-753;
- операции в башне расширений `Fq2/Fq4`;
- Montgomery/Barrett/CIOS/FIOS/Comba как варианты базового умножения;
- sparse/prepared line arithmetic;
- цикл Миллера;
- финальная экспонента;
- direct residue verifier по идее `On Proving Pairings`;
- полиномиальная проверка цикла Миллера как дальнейшее направление;
- связь gas-оценок с constraints-оценками для proof-подходов.

Основные источники чисел внутри проекта:

## 2. Уровни модели сложности

Стоимость вычисления сопряжения удобно описывать не сразу в gas, а слоями.

```text
EVM word operations
  -> arithmetic in Fp
  -> arithmetic in Fp2
  -> arithmetic in Fp4
  -> line evaluation and sparse line multiplication
  -> Miller loop
  -> final exponentiation / residue relation
  -> full verifier scenario
```

Обозначения:

| Обозначение | Смысл |
|---|---|
| `M1` | одно умножение в базовом поле `Fp` |
| `S1` | одно возведение в квадрат в `Fp` |
| `A1` | сложение или вычитание в `Fp` |
| `M2` | одно умножение в `Fp2` |
| `S2` | одно возведение в квадрат в `Fp2` |
| `M4` | одно умножение в `Fp4` |
| `S4` | одно возведение в квадрат в `Fp4` |
| `L_dbl` | применение prepared doubling-линии к аккумулятору Миллера |
| `L_add` | применение prepared addition-линии к аккумулятору Миллера |
| `L` | число doubling-раундов signed ate-loop |
| `H` | число ненулевых addition-шагов signed ate-loop |

Для MNT4-753 в проекте используются:

```text
L = 376
H = 124
```

В некоторых местах для основной signed-записи отдельно встречается `123` ненулевых NAF-шагов и один корректирующий шаг для отрицательного loop count. В итоговых грубых оценках используется `H = 124`, чтобы не занизить стоимость.

## 3. Стоимость базовой арифметики

Элемент `Fp` имеет размер около 753 бит и хранится в трех 256-битных словах:

```text
x = x0 + x1 * 2^256 + x2 * 2^512.
```

Поэтому одно умножение в `Fp` не является одной EVM-инструкцией `MUL`. Оно требует произведений слов, переносов и модульной редукции.

Измеренные значения:

| Вариант | Операция | Gas/op | Роль в проекте |
|---|---:|---:|---|
| Montgomery/CIOS | `Fp mul` | 2,959 | основной путь |
| Montgomery/CIOS | `Fp sqr` | 2,947 | основной путь |
| Barrett | `Fp mul` | 46,998 | экспериментальный вариант |
| Barrett | `Fp sqr` | 47,172 | экспериментальный вариант |
| FIOS | `Fp mul` | 18,509 | экспериментальный вариант |
| FIOS | `Fp sqr` | 18,543 | экспериментальный вариант |
| Comba/SOS | `Fp mul` | 18,301 | экспериментальный вариант |
| Comba/SOS | `Fp sqr` | 18,334 | экспериментальный вариант |

Вывод: для длинных цепочек умножений в EVM выбранный путь Montgomery/CIOS является наиболее дешевым среди реализованных вариантов.

### 3.1. Montgomery/CIOS

Пусть `B = 2^256`, `n = 3`, `R = B^n = 2^768`. Montgomery-представление элемента `a`:

```text
a_tilde = a * R mod p.
```

Если `a_tilde` и `b_tilde` уже находятся в Montgomery-домене, то

```text
MontMul(a_tilde, b_tilde) = a_tilde * b_tilde * R^{-1} mod p = a*b*R mod p.
```

То есть результат остается в Montgomery-домене. Для сопряжения это важно, потому что цикл Миллера и финальная экспонента состоят из длинных цепочек умножений. Вход в Montgomery-домен выполняется редко, а основная часть вычисления остается в нем.

CIOS-организация объединяет накопление частичных произведений и редукцию. Для трехсловного поля это особенно выгодно, потому что не нужно полностью материализовать 6-словное произведение перед редукцией.

### 3.2. Barrett

Barrett-редукция использует предвычисленное приближение обратного модуля:

```text
mu = floor(B^(2n) / p).
```

После произведения оценивается частное, затем выполняется коррекция остатка. Для одиночной редукции это может быть приемлемо, но в цепочке из тысяч умножений Barrett каждый раз заново несет стоимость оценки частного и корректировок. Измерения показывают, что в EVM для данного поля это намного дороже Montgomery/CIOS.

### 3.3. FIOS и Comba/SOS

FIOS интегрирует редукцию тоньше, чем CIOS, но в EVM это увеличивает управляющую логику и работу с переносами. Comba/SOS сначала строит полное произведение, затем редуцирует его. Для обычного процессора это может быть выгодно при хорошем использовании регистров, но в EVM материализация промежуточного массива и memory pressure делают вариант дороже.

Итоговая практическая иерархия для `Fp mul`:

```text
Montgomery/CIOS  <<  Comba/SOS ~= FIOS  <<  Barrett
2,959 gas            ~18,3k gas            ~47k gas
```

## 4. Стоимость арифметики расширений

Башня расширений в проекте:

```text
Fp2 = Fp[u] / (u^2 - 13)
Fp4 = Fp2[v] / (v^2 - u).
```

Элемент `Fp2` имеет вид:

```text
a = a0 + a1*u.
```

Элемент `Fp4` имеет вид:

```text
x = x0 + x1*v,  x0,x1 in Fp2.
```

Измеренные значения:

| Операция | Gas/op |
|---|---:|
| `Fp2 mul` | 11,877 |
| `Fp2 sqr` | 10,592 |
| `Fp4 mul` | 42,764 |
| `Fp4 sqr` | 36,606 |

### 4.1. Karatsuba для `Fp2`

Прямое умножение:

```text
(a0 + a1*u)(b0 + b1*u)
= (a0*b0 + 13*a1*b1) + (a0*b1 + a1*b0)u.
```

Наивно требуется 4 умножения в `Fp`. Karatsuba-форма использует 3 умножения:

```text
t0 = a0*b0
t1 = a1*b1
t2 = (a0+a1)(b0+b1)
res0 = t0 + 13*t1
res1 = t2 - t0 - t1.
```

Структурная оценка:

```text
M2 ~= 3*M1 + cheap_mul_by_13 + O(A1).
```

Почему измеренный `Fp2 mul = 11,877 gas`, а не ровно `3 * 2,959 = 8,877 gas`: добавляются сложения, редукции, загрузки и записи 6-словных элементов, работа со scratch memory и вызовы вспомогательных функций.

### 4.2. Karatsuba для `Fp4`

Для

```text
x = x0 + x1*v,
y = y0 + y1*v,
v^2 = u
```

используется:

```text
t0 = x0*y0
t1 = x1*y1
t2 = (x0+x1)(y0+y1)
res0 = t0 + u*t1
res1 = t2 - t0 - t1.
```

Структурная оценка:

```text
M4 ~= 3*M2 + cheap_mul_by_u + O(A2).
```

Измерение:

```text
M4 = 42,764 gas.
```

Это выше грубого `3*M2 = 35,631 gas`, потому что добавляются операции над компонентами `Fp2`, умножение на нерезидент, память и редукции.

### 4.3. Cheap non-residue multiplication

Так как нерезиденты имеют специальный вид, умножение на них не выполняется как generic field multiplication.

| Операция | Generic gas/op | Specialized gas/op | Улучшение |
|---|---:|---:|---:|
| `mulBy13` в `Fp` | 2,961 | 1,481 | около 2.0x |
| `mulByU` в `Fp2` | 12,587 | 1,731 | около 7.3x |
| `mulByV` в `Fp4` | 42,253 | 1,859 | около 22.7x |

Это важная часть модели сложности: если считать умножение на `13`, `u` или `v` как обычное умножение в расширении, оценка будет сильно завышена. В production-path эти операции вынесены в специализированные helpers.

## 5. Sparse line arithmetic

В цикле Миллера line-функция после подстановки точки `P` не является произвольным плотным элементом `Fp4`. Она имеет вид:

```text
ell(P) = ell0 + ell1 * v,
```

где `ell0, ell1 in Fp2`.

Для accumulator

```text
f = f0 + f1*v
```

умножение на линию:

```text
f * ell = (f0*ell0 + u*f1*ell1) + (f0*ell1 + f1*ell0)*v.
```

Через Karatsuba:

```text
t0 = f0*ell0
t1 = f1*ell1
t2 = (f0+f1)(ell0+ell1)
res0 = t0 + u*t1
res1 = t2 - t0 - t1.
```

Структурная оценка sparse line multiplication:

```text
L_sparse ~= 3*M2 + cheap_mul_by_u + O(A2).
```

То есть по числу умножений она близка к `Fp4 mul`, но дешевле по загрузке данных, потому что линия хранится в sparse-формате и не требует полного dense `Fp4`-представления.

В проекте также проверен aggressive fused-вариант для шага:

```text
f <- f^2 * ell(P).
```

Он не материализует промежуточный `Fp4` для `f^2`, но дает только около `20k gas` экономии на полном single pairing path. Это означает, что основная стоимость определяется не memory traffic, а количеством дорогих умножений в `Fp2/Fp4`.

## 6. Оценка цикла Миллера

Для single prepared Miller loop используется схема:

```text
f <- 1
for each bit/round:
    f <- f^2
    f <- f * ell_dbl(P)
    if bit != 0:
        f <- f * ell_add(P)
```

Для pairing equation с двумя парами, например

```text
e(P,Q) * e(-R,S) = 1,
```

общий accumulator обновляется так:

```text
f <- f^2
f <- f * ell_Q,dbl(P)
f <- f * ell_S,dbl(-R)
if bit != 0:
    f <- f * ell_Q,add(P)
    f <- f * ell_S,add(-R)
```

### 6.1. Грубая нижняя gas-оценка для двухпарного core

Используем измеренные стоимости:

```text
S4 = 36,606 gas
M4 = 42,764 gas
L = 376
H = 123
```

Первое слагаемое: обязательное возведение accumulator в квадрат на каждом раунде:

```text
376 * S4 = 376 * 36,606 = 13,763,856 gas.
```

Второе слагаемое: две doubling-линии на каждый раунд:

```text
2 * 376 * M4 = 2 * 376 * 42,764 = 32,157,728 gas.
```

Третье слагаемое: две addition-линии на каждый ненулевой шаг:

```text
2 * 123 * M4 = 2 * 123 * 42,764 = 10,499,864 gas.
```

Итого:

```text
13,763,856 + 32,157,728 + 10,499,864 = 56,421,448 gas.
```

Это не точный прогноз, а нижняя оценка уровня операций. Она считает применение линии как одну операцию уровня `M4`, хотя реальный код дополнительно:

- загружает sparse-коэффициенты;
- восстанавливает значение линии в точке;
- выполняет дополнительные операции в `Fp2`;
- работает с памятью;
- обрабатывает signed loop и correction;
- выполняет проверки и упаковку входов.

Измеренный hot core:

```text
Two-pair Miller core = 87,747,219 gas.
```

Средняя стоимость на один раунд:

```text
87,747,219 / 376 ~= 233,370 gas/round.
```

Вывод: порядок величины согласуется с операционной оценкой. Стоимость не является ошибкой теста: даже prepared sparse-путь исполняет сотни операций в `Fp4/Fp2`.

### 6.2. Нижняя оценка из текста диплома через базовые умножения

В `main-2.tex` используется более абстрактная модель через число базовых умножений `Fp`.

Для prepared Miller:

```text
N_Miller,prep >= 20*L + 17*H.
```

При `L = 376`, `H = 124`:

```text
20*376 + 17*124 = 9,628 базовых умножений Fp.
```

Для полного on-chain режима, где линии генерируются в EVM:

```text
N_Miller,on >= 50*L + 47*H.
```

При `L = 376`, `H = 124`:

```text
50*376 + 47*124 = 24,628 базовых умножений Fp.
```

Эта модель намеренно консервативна. Она доказывает не точную стоимость контракта, а то, что даже нижняя граница уже велика.

## 7. Оценка финальной экспоненты

После цикла Миллера получаем `x in Fp4*`. Для MNT4 со степенью вложения 4 финальная экспонента:

```text
E_FE = (p^4 - 1) / r = (p^2 - 1) * (p^2 + 1) / r.
```

Реализация использует разложение:

```text
z = x^(p^2 - 1) = x^(p^2) * x^(-1)
E_hard = (p^2 + 1) / r = p + w0
y = z^p * z^w0.
```

Модель стоимости:

```text
C_FE = C_inv4
     + C_easy
     + C_frobenius
     + N_sqr(w0)*S4
     + N_mul(w0)*M4
     + C_final_mul.
```

Где:

- `C_inv4` — инверсия в `Fp4`;
- `C_easy` — первая часть `x^(p^2-1)`;
- `C_frobenius` — применение Frobenius map;
- `N_sqr(w0)` и `N_mul(w0)` — число операций fixed addition-chain для `w0`;
- `C_final_mul` — финальное умножение частей.

Измеренная декомпозиция prepared fixed-Q path:

| Стадия | Добавочная стоимость, gas | Интерпретация |
|---|---:|---|
| Инверсия в `Fp4` | 3,024,934 | дорогое обращение элемента целевого поля |
| Первая часть `x^(p^2-1)` | 115,039 | conjugation + multiplication |
| Frobenius-часть `z^p` | 30,727 | почти линейное преобразование |
| `w0`-часть hard part | 16,755,647 | доминирующая fixed-chain часть |
| Финальное умножение/завершение | 187,969 | сборка результата |

Вывод: в финальной экспоненте доминирует `w0`-часть, потому что именно она содержит длинную цепочку операций в `Fp4`.

## 8. Полное on-chain вычисление

Полный on-chain путь выполняет:

```text
1. проверку и подготовку входов;
2. генерацию или загрузку line coefficients;
3. Miller loop;
4. final exponentiation;
5. упаковку результата/digest.
```

Основные измерения:

| Сценарий | Gas | Смысл |
|---|---:|---|
| Полное on-chain reference-вычисление MNT4 pairing digest | 259,719,954 | все вычисление внутри EVM |
| Полное on-chain вычисление из `main-2.tex` | 259,543,110 | актуальная сводная таблица диплома |

| Prepared code-shards | 79,565,941 | потоковая загрузка коэффициентов через code-shards |

Практический вывод:

```text
Даже при оптимизированной арифметике полный on-chain путь остается на уровне десятков/сотен миллионов gas.
```

Главная причина — не один плохой участок кода, а произведение трех факторов:

```text
большое поле 753 бит
* башня Fp -> Fp2 -> Fp4
* сотни раундов Miller loop и длинная final exponentiation.
```

## 9. Direct residue verifier по статье 2024/640

В варианте direct residue verifier проверяется pairing equation:

```text
e(P,Q) * e(-R,S) = 1.
```

Вместо отдельного вычисления полной финальной экспоненты используется witness `c, c^{-1}` и relation, встроенная в измененный Miller loop. Смысл: заменить часть длинного возведения в степень более короткой проверкой отношения.

Измеренные значения:

| Режим | Что делает | Gas |
|---|---|---:|
| core | цикл Миллера без финальной экспоненты | 87,747,219 |
| full equation | цикл Миллера + обычная финальная экспонента | 115,997,313 |
| direct residue equation | цикл Миллера + residue-проверка вместо полной FE | 93,879,746 |

Экономия residue-подхода относительно обычной финальной экспоненты:

```text
115,997,313 - 93,879,746 = 22,117,567 gas.
```

Дополнительная стоимость residue-ветки относительно чистого Miller core:

```text
93,879,746 - 87,747,219 = 6,132,527 gas.
```

Итог:

```text
Residue-оптимизация финальной экспоненты работает, но не решает проблему полностью, потому что Miller core уже стоит 87.7M gas.
```

## 10. Почему одной оптимизации финальной экспоненты недостаточно

Полная стоимость direct verifier раскладывается так:

```text
G_full_equation ~= G_Miller + G_FE
G_residue       ~= G_Miller + G_residue_check
```

Подстановка измерений:

```text
G_Miller = 87,747,219 gas
G_FE contribution ~= 115,997,313 - 87,747,219 = 28,250,094 gas
G_residue contribution ~= 93,879,746 - 87,747,219 = 6,132,527 gas
```

Оптимизация FE снижает завершающую часть примерно с `28.25M` до `6.13M gas`, но общий результат все равно остается около `94M gas`, потому что основная стоимость — цикл Миллера.

Поэтому практическое снижение gas требует не только заменить final exponentiation, но и сжать проверку Miller loop.

## 11. Polynomial check для цикла Миллера

Идея статьи `On Proving Pairings`: не исполнять каждое умножение в расширенном поле on-chain, а проверять много алгебраических отношений через полиномиальную проверку.

Пусть элемент `Fp4` представлен как многочлен степени меньше 4:

```text
a(X) = a0 + a1 X + a2 X^2 + a3 X^3.
```

Умножение в `Fp4` задается отношением:

```text
a(X) * b(X) - c(X) = q(X) * p_ext(X),
```

где `p_ext(X)` — неприводимый многочлен башни расширения.

Вместо полного умножения verifier выбирает случайную точку `alpha` и проверяет:

```text
a(alpha) * b(alpha) - c(alpha) = q(alpha) * p_ext(alpha).
```

Для многих отношений используется объединение через challenge `beta`:

```text
sum_i beta^i * R_i(alpha) = Q(alpha) * Z(alpha).
```

Смысл:

```text
много Fp4-умножений -> одна или несколько проверок отношений в случайной точке.
```

Но для полной реализации нужен механизм commitments/openings, иначе prover может прислать произвольные значения в `alpha`.

## 12. Два варианта commitments/openings

### 12.1. KZG over BN254

Плюс:

```text
короткое доказательство и дешевый on-chain verifier через BN254 precompile.
```

Минус для MNT4-753:

```text
MNT4-элементы имеют 753 бита, а BN254 field около 254 бит.
```

Значит элементы MNT4 приходится раскладывать на limb-ы и проверять как non-native arithmetic. Это переносит стоимость из gas в constraints/proving time.

Уже измеренный ориентир проекта:

| Операция в BN254-circuit | Constraints |
|---|---:|
| Ненативное `Fp`-умножение MNT4-753 | 7,032 |
| Ненативное `Fp2`-умножение MNT4-753 | 44,404 |

Нижняя оценка из текста диплома:

```text
13,536 базовых Fp-умножений * 7,032 constraints = 95,185,152 constraints.
```

Это оценка только по базовым умножениям и без range checks, сложений, hashing, commitments и операций расширений. Поэтому полный non-native replay в BN254 не является практическим путем.

### 12.2. Merkle/FRI

Плюс:

```text
можно оставаться ближе к родному полю вычисления и не завязываться на BN254 KZG.
```

Минус:

```text
растет calldata и hash/opening overhead.
```

Один элемент MNT4-753 занимает около:

```text
ceil(753 / 8) = 95 байт,
```

практически — около `96 байт` при выравнивании. Если opening раскрывает много значений witness-многочленов, каждый слой Merkle proof и каждое значение становятся дорогими для calldata. Поэтому Merkle/FRI может уменьшить арифметику verifier-а, но увеличить стоимость данных.

## 13. Constraints-модель для proof-подходов

Для proof-подходов важна не только gas-стоимость on-chain verifier-а, но и число constraints.

Текущие ориентиры из проекта:

| Компонент | Constraints | Интерпретация |
|---|---:|---|
| BN254 verifier-envelope | 1,538 | компактная обвязка, не полный MNT4 pairing proof |
| Prepared algebraic/residue relation | 10,211 | текущая compact relation |
| MNT-native accounting model | 24,178 | модель будущего MNT-cycle relation слоя |
| Compiled MNT-native relation fragment | 24,181 | скомпилированный фрагмент модели |
| BN254 emulated pairing reference | 1,393,318 | ориентир non-native/emulated подхода |
| Sonobe-like decider reference | около 9,000,000 | ориентир folding-decider слоя |

Важное ограничение:

```text
1,538 или 10,211 constraints нельзя называть стоимостью полного доказательства MNT4 pairing.
```

Это стоимость compact relation / verifier envelope. Полный низкоуровневый replay MNT4-арифметики в BN254-circuit был бы на порядки дороже.

## 15. Границы применимости оценок

Оценки бывают трех типов.

### 15.1. Нижние оценки

Например:

```text
N_Miller,prep >= 20L + 17H.
```

Они показывают, что дешевле определенного уровня быть не может в выбранной модели, но не предсказывают точный gas.

### 15.2. Операционные оценки

Например:

```text
376*S4 + 2*376*M4 + 2*123*M4 = 56,421,448 gas.
```

Они ближе к реальному контракту, но все еще не учитывают весь overhead: память, загрузку коэффициентов, ветвления, ABI, проверки и correction.

### 15.3. Измерения

Например:

```text
Miller core = 87,747,219 gas.
```

Это фактическая стоимость реализации в тестовой среде Foundry. Именно она является главным практическим числом, но без модели сложности она не объясняет, почему стоимость такая высокая.

## 17. Итоговый вывод

Третье замечание руководителя закрывается следующей цепочкой аргументации:

1. Стоимость MNT4-сопряжения разложена на уровни `Fp -> Fp2 -> Fp4 -> Miller -> FE`.
2. Для базовых операций реализованы и измерены альтернативные алгоритмы.
3. Для расширений указаны формулы Karatsuba и специальные умножения на нерезиденты.
4. Для цикла Миллера построена нижняя оценка через `Fp`-умножения и практическая оценка через `Fp4`-операции.
5. Для финальной экспоненты выделены стадии и показано, какая часть доминирует.
6. Для direct residue verifier показано, что оптимизация FE дает экономию, но не устраняет стоимость Miller loop.
7. Для proof-подходов указано, почему полный BN254 non-native replay приводит к десяткам миллионов constraints.

Практический смысл результата:

```text
Проект показывает не только итоговые gas-числа, но и причину этих чисел: стоимость возникает из большого числа операций в Fp4 над 753-битным полем. Поэтому дальнейшее кратное улучшение требует менять протокол проверки Miller loop, а не только оптимизировать отдельные Yul-функции.
```
