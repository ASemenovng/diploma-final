# План полного аудита практической части дипломной работы

## 1. Цель аудита

Цель аудита — независимо проверить, можно ли считать практическую часть дипломной
работы завершенной и защищаемой. Проверка не должна ограничиваться успешным
запуском тестов. Нужно установить:

1. какие заявленные результаты действительно реализованы исполняемым кодом;
2. какие результаты подтверждены аналитической моделью, но еще не реализованы как
   готовый constraint gadget, proof system или production-контракт;
3. какие утверждения подтверждены кросс-проверкой с канонической библиотекой;
4. где остаются ограничения безопасности, применимости или воспроизводимости;
5. существуют ли еще математические, архитектурные или низкоуровневые
   оптимизации, которые могут существенно изменить выводы работы.

Основной принцип:

> Ни один аналитический расчет нельзя выдавать за измерение готовой реализации.
> Ни один исследовательский прототип нельзя выдавать за production-ready
> контракт. При этом отрицательный результат также является полноценным
> результатом работы, если он строго воспроизводим и корректно сформулирован.

## 2. Формат итогового отчета

После выполнения аудита должен быть создан документ:

```text
docs/PRACTICAL_IMPLEMENTATION_FULL_AUDIT_REPORT_RU.md
```

Для каждого проверяемого тезиса в отчете указывается один из статусов:

| Статус | Значение |
|---|---|
| `Закрыто кодом` | Есть исполняемая реализация, тесты и воспроизводимые измерения. |
| `Закрыто аналитически` | Есть формальная модель или вывод, но нет полного исполняемого слоя. |
| `Частично закрыто` | Реализована только часть требуемого контура или есть существенное ограничение. |
| `Не закрыто` | Требуемый результат отсутствует либо текущие доказательства недостаточны. |
| `Требует исправления` | Есть ошибка корректности, тестирования, документации или интерпретации результата. |

Отдельно формируется матрица трассируемости:

```text
docs/PRACTICAL_IMPLEMENTATION_AUDIT_TRACEABILITY_MATRIX_RU.md
```

Строка матрицы:

```text
Тезис -> код -> тест -> внешний эталон -> метрика -> ограничение -> статус.
```

## 3. Правила проведения проверки

1. Сначала проводится аудит без изменения production-кода.
2. Любой найденный разрыв фиксируется отдельно: ошибка, недостающий тест,
   недостающая реализация, неточная формулировка или возможная оптимизация.
3. Исправления выполняются только после формирования первичного отчета.
4. После исправлений повторяется соответствующая часть аудита.
5. Gas измеряется отдельно как:
   - execution gas функции;
   - test-call gas;
   - calldata gas;
   - deployment gas;
   - стоимость data-contract/code-shards deployment, если они используются.
6. Constraints разделяются на:
   - реально измеренные constraints собранной схемы;
   - воспроизводимый operation/constraint accounting;
   - внешние сравнительные ориентиры.
7. Для внешних сравнений используются первичные источники: статьи, официальная
   документация и исходный код библиотек.

особенно внимательно:
