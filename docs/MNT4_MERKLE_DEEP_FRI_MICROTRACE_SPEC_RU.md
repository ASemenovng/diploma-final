# MNT4-753: спецификация Merkle/DEEP-FRI-проверки цикла Миллера по пошаговой микротрассе

## 1. Статус документа

Этот документ задает полностью определенную спецификацию экспериментального модуля, который проверяет цикл Миллера MNT4-753 без его полного исполнения в EVM. Модуль предназначен для ответа на прикладной вопрос:

> дает ли замена прямого on-chain цикла Миллера на неинтерактивную Merkle/DEEP-FRI-проверку принципиальный выигрыш по gas?

Первая реализация является исследовательским модулем для воспроизводимого измерения gas, а не готовым production-verifier-ом. Криптографический вывод о рекомендуемом числе запросов допускается только после генерации и проверки `security_report.json`. Это ограничение не мешает реализации: оба измерительных профиля и все проверяемые отношения в документе определены полностью.

Спецификация не изменяет математическое утверждение текущего `article640_mnt4`-модуля. Проверяется равенство сопряжений

\[
e(P,Q)\cdot e(-R,S)=1,
\]

где \(P,R\in G_1\) передаются пользователем, а \(Q,S\in G_2\) фиксируются при подготовке verifier-а.

Документ является самостоятельной основой для реализации. В нем зафиксированы:

1. точное проверяемое утверждение;
2. формат пошаговой трассы;
3. набор операций трассы;
4. арифметические ограничения;
5. формат фиксированной таблицы линий;
6. построение quotient polynomial;
7. неинтерактивный transcript;
8. Merkle- и DEEP-FRI-проверка;
9. бинарный формат доказательства;
10. тесты и критерии готовности;
11. расчетная модель calldata и gas.

Важно: описанная схема является адаптацией идей randomized checking и DEEP-FRI к конкретному MNT4-753 циклу Миллера. Из статьи `On Proving Pairings` используются: замена финальной экспоненты residue-check-отношением, подготовленные линии для фиксированного второго аргумента и идея случайной проверки арифметических отношений. Merkle/DEEP-FRI-оболочка пошаговой микротрассы является отдельной неинтерактивной инженерной конструкцией для EVM-эксперимента, а не дословным воспроизведением алгоритма статьи.

### 1.1. Связь с предыдущей спецификацией

Документ:

```text
docs/MNT4_MERKLE_FRI_POLYNOMIAL_MILLER_SPEC_RU.md
```

зафиксировал правильную общую цель: заменить последовательное on-chain исполнение цикла Миллера проверкой полиномиальных отношений, Merkle-коммитментов и FRI-доказательства низкой степени.

Настоящая спецификация реализует ту же идею в более строгом базовом варианте:

```text
одна строка AIR = одна реальная микрооперация residue-рекурсии
```

Старый документ дополнительно предлагал объединять пять итераций цикла Миллера в один блок. Такое сжатие может быть исследовано после базовой реализации, но оно не используется в первом эксперименте: блочная версия требует отдельного строгого вывода подготовленных делителей, промежуточных столбцов и степенных ограничений. Пошаговая микротрасса исключает скрытые переходы и позволяет проверить каждую реальную операцию.

В новой схеме по-прежнему используются полиномы:

1. четыре trace-полинома \(T_0(X),\ldots,T_3(X)\) для координат аккумулятора \(\mathbb F_{q^4}\);
2. семнадцать fixed-полиномов \(K_0(X),\ldots,K_{16}(X)\) для расписания и подготовленных линий;
3. два сегмента quotient polynomial \(Q_0(X),Q_1(X)\);
4. DEEP polynomial \(D_{\mathrm{deep}}(X)\);
5. последовательность FRI-полиномов после folding.

Они фиксируются Merkle-корнями:

```text
rootFixed
rootTrace
rootQuot
rootDeep
rootFri1
...
rootFri7
```

Merkle-дерево само по себе не доказывает корректность цикла Миллера. Оно только связывает prover-а с конкретными таблицами значений. Корректность обеспечивается совокупностью трех проверок:

1. quotient relation проверяет выполнение AIR-отношений;
2. Merkle-multiproof связывает раскрытые значения с зафиксированными таблицами;
3. DEEP-FRI проверяет, что таблицы являются низкостепенными продолжениями полиномов.

## 2. Проверяемое утверждение

### 2.1. Публичные данные

Verifier получает:

```text
P = (Px, Py) in G1
R = (Rx, Ry) in G1
c, cInv in Fq4
proof: bytes
```

Точки \(Q,S\in G_2\), параметры кривой, таблица подготовленных линий и параметры DEEP-FRI фиксируются конфигурацией verifier-а.

### 2.2. Свидетельство для финальной экспоненты

Проверяется уравнение

\[
e(P,Q)\cdot e(-R,S)=1.
\]

Пусть \(F\in \mathbb F_{q^4}^{*}\) — результат объединенного цикла Миллера для двух пар \((P,Q)\) и \((-R,S)\). Вместо полной финальной экспоненты используется свидетельство \(c\), связывающее результат цикла Миллера с проверяемым отношением. Текущая оптимизированная MNT4-753 реализация сводит on-chain проверку к рекурсии:

\[
F_0=c^{-1},
\]

\[
F_{i+1}
=
\operatorname{Step}_i(F_i;P,R,c,c^{-1}),
\]

\[
F_{\mathrm{final}}=1.
\]

Функции \(\operatorname{Step}_i\) полностью задаются подписанными цифрами ate-параметра, фиксированными подготовленными линиями для \(Q,S\), а также завершающим множителем \(\operatorname{Frob}_{q}(c^{-1})\).

До запуска proof verifier контракт обязан проверить:

\[
c\cdot c^{-1}=1.
\]

### 2.3. Почему residue-рекурсия корректна

Обозначим через:

\[
r
\]

порядок целевой подгруппы сопряжения, а через:

\[
h=\frac{q^4-1}{r}
\]

показатель финальной экспоненты.

Пусть:

\[
t_1,\ldots,t_{376}\in\{-1,0,1\}
\]

— signed-digit цифры ate-цикла после старшей цифры. В residue-рекурсии показатель степени \(c\) изменяется независимо от умножений на линии. Его можно вычислить отдельно:

\[
\kappa_0=-1,
\]

\[
\kappa_{i+1}=
\begin{cases}
2\kappa_i-1, & t_i=1,\\
2\kappa_i, & t_i=0,\\
2\kappa_i+1, & t_i=-1.
\end{cases}
\]

После основного цикла добавляется множитель:

\[
\operatorname{Frob}_{q}(c^{-1})=c^{-q}.
\]

Следовательно:

\[
\kappa_{\mathrm{final}}=\kappa_{376}-q.
\]

Для зафиксированного MNT4-753 ate-параметра прямое целочисленное вычисление дает:

\[
\kappa_{\mathrm{final}}=-r.
\]

Это равенство должно быть проверено отдельным Rust unit test без арифметики по модулю.

Пусть:

\[
M(P,Q,-R,S)
\]

— результат обычной объединенной рекурсии Миллера: с теми же возведениями аккумулятора в квадрат и теми же умножениями на значения подготовленных линий, но без множителей \(c\), \(c^{-1}\) и \(\operatorname{Frob}_{q}(c^{-1})\). Тогда финальное состояние микротрассы равно:

\[
F_{\mathrm{final}}
=
M(P,Q,-R,S)\cdot c^{-r}.
\]

Условие:

\[
F_{\mathrm{final}}=1
\]

эквивалентно:

\[
c^r=M(P,Q,-R,S).
\]

Отсюда следует:

\[
M(P,Q,-R,S)^h
=
c^{rh}
=
c^{q^4-1}
=
1.
\]

То есть принятие микротрассы гарантирует, что полная финальная экспонента объединенного Miller-результата равна единице.

Верна и обратная импликация. Для MNT4-753:

\[
\gcd(r,h)=1.
\]

Если:

\[
M(P,Q,-R,S)^h=1,
\]

то \(M(P,Q,-R,S)\) принадлежит подгруппе порядка \(h\). Возведение в степень \(r\) является автоморфизмом этой подгруппы, поскольку \(\gcd(r,h)=1\). Поэтому существует единственное:

\[
c=M(P,Q,-R,S)^{r^{-1}\bmod h}
\]

такое, что:

\[
c^r=M(P,Q,-R,S).
\]

Таким образом, residue-рекурсия не ослабляет исходную проверку уравнения сопряжений: она заменяет длинную финальную экспоненту эквивалентным отношением со свидетельством \(c\).

### 2.4. API контракта

Рекомендуемый внешний интерфейс:

```solidity
function verifyEquationMicrotrace(
    G1Point calldata p,
    G1Point calldata r,
    Fq4 calldata c,
    Fq4 calldata cInv,
    bytes calldata proof
) external view returns (bool);
```

Verifier возвращает `true` только при одновременном выполнении условий:

1. координаты \(P,R,c,c^{-1}\) канонически закодированы;
2. \(P,R\) принадлежат \(G_1\);
3. \(c\cdot c^{-1}=1\);
4. фиксированная таблица линий связана с доверенным `rootFixed`;
5. Merkle-пути корректны;
6. quotient polynomial согласован с AIR;
7. DEEP-FRI-проверка принята;
8. начальное состояние трассы равно \(c^{-1}\);
9. финальное состояние трассы равно \(1\).

Для MNT4-753 проверка уравнения кривой достаточна для принадлежности \(G_1\), поскольку кофактор \(G_1\) равен единице.

Точки \(P,R\) не могут быть точкой на бесконечности. Внешний API использует только аффинные координаты и отклоняет выделенную нулевую кодировку \((0,0)\).

### 2.5. Доверенная граница конфигурации

Модуль является verifier-ом для двух заранее фиксированных точек \(Q,S\in G_2\). При генерации конфигурации Rust backend обязан проверить:

1. \(Q,S\) не являются точкой на бесконечности;
2. \(Q,S\) лежат на кривой расширения;
3. \(Q,S\) принадлежат требуемой подгруппе;
4. таблица подготовленных линий воспроизводится из \(Q,S\);
5. `rootFixed` соответствует LDE этой таблицы.

Контракт хранит `configDigest` и `rootFixed`. Поэтому вызывающая сторона не может заменить \(Q,S\) или их кэш. Если требуется другая пара фиксированных точек, создается новая конфигурация verifier-а.

## 3. Поля и представление элементов

### 3.1. Базовое поле

Все арифметические ограничения вычисляются над базовым полем MNT4-753:

\[
\mathbb F_q.
\]

Один элемент \(\mathbb F_q\) сериализуется тремя 256-битными словами EVM, то есть занимает \(96\) байт.

### 3.2. Башня расширений

Используется башня:

\[
\mathbb F_{q^2}=\mathbb F_q[u]/(u^2-13),
\]

\[
\mathbb F_{q^4}=\mathbb F_{q^2}[v]/(v^2-u).
\]

Элемент \(\mathbb F_{q^4}\) представляется как

\[
A=a_0+a_1Z+a_2Z^2+a_3Z^3,
\qquad Z^4=13.
\]

Такое плоское представление удобно для AIR: один аккумулятор \(\mathbb F_{q^4}\) задается четырьмя столбцами \(\mathbb F_q\).

### 3.3. Формулы умножения и возведения в квадрат

Для

\[
A=a_0+a_1Z+a_2Z^2+a_3Z^3,
\qquad
B=b_0+b_1Z+b_2Z^2+b_3Z^3
\]

произведение \(C=A\cdot B\) имеет координаты:

\[
\begin{aligned}
c_0 &= a_0b_0+13(a_1b_3+a_2b_2+a_3b_1),\\
c_1 &= a_0b_1+a_1b_0+13(a_2b_3+a_3b_2),\\
c_2 &= a_0b_2+a_1b_1+a_2b_0+13a_3b_3,\\
c_3 &= a_0b_3+a_1b_2+a_2b_1+a_3b_0.
\end{aligned}
\]

Для \(S=A^2\):

\[
\begin{aligned}
s_0 &= a_0^2+13(2a_1a_3+a_2^2),\\
s_1 &= 2a_0a_1+26a_2a_3,\\
s_2 &= 2a_0a_2+a_1^2+13a_3^2,\\
s_3 &= 2a_0a_3+2a_1a_2.
\end{aligned}
\]

Эти формулы должны использоваться и в Rust backend, и в Solidity verifier-е.

### 3.4. Отображение Фробениуса

Для элемента:

\[
A=a_0+a_1u\in\mathbb F_{q^2}
\]

отображение Фробениуса имеет вид:

\[
\operatorname{Frob}_{q}^{(2)}(A)
=
a_0+\chi_2a_1u,
\]

где:

\[
\chi_2=u^{q-1}\in\mathbb F_q.
\]

Для элемента:

\[
B=b_0+b_1v\in\mathbb F_{q^4},
\qquad
b_0,b_1\in\mathbb F_{q^2}
\]

используется:

\[
\operatorname{Frob}_{q}^{(4)}(B)
=
\operatorname{Frob}_{q}^{(2)}(b_0)
+
\chi_4\operatorname{Frob}_{q}^{(2)}(b_1)v,
\]

где:

\[
\chi_4=v^{q-1}\in\mathbb F_{q^2}.
\]

Коэффициенты \(\chi_2,\chi_4\) являются фиксированными параметрами поля расширения. Они включаются в `fixed_config.json`, воспроизводятся из параметров MNT4-753 в Rust backend и сверяются с константами текущего Solidity/Yul-модуля. Значение:

\[
\operatorname{Frob}_{q}(c^{-1})
\]

вычисляется verifier-ом из публичного \(c^{-1}\), а не принимается как доверенный вход.

## 4. Нормализованный формат подготовленных линий

### 4.1. Зачем нужен фиксированный кэш

Точки \(Q,S\in G_2\) фиксированы. Поэтому коэффициенты линий ate-цикла вычисляются один раз off-chain. Контракт не получает произвольный кэш от пользователя: он хранит доверенный корень Merkle-дерева `rootFixed`.

Проверяемое доказательство раскрывает только строки фиксированной таблицы, необходимые для случайных запросов DEEP-FRI. Каждая раскрытая строка подтверждается Merkle-путем до `rootFixed`.

### 4.2. Единый формат строки линии

Каждая подготовленная линия кодируется тремя элементами \(\mathbb F_{q^2}\):

\[
(k_0,k_1,k_2)\in\mathbb F_{q^2}^3.
\]

Каждый \(k_i\) занимает две координаты \(\mathbb F_q\). Поэтому строка линии занимает шесть элементов \(\mathbb F_q\).

### 4.3. Удвоение

Исходные arkworks-коэффициенты удвоения:

\[
(c_h,c_{4c},c_j,c_l).
\]

Сначала вычисляются:

\[
d_0=c_l-c_{4c},
\qquad
d_1=c_j,
\qquad
d_2=c_h.
\]

В фиксированную таблицу записываются:

\[
k_0=d_0,
\qquad
k_1=-u\,d_1,
\qquad
k_2=u\,d_2.
\]

Для точки \(T=(x,y)\in G_1\):

\[
\ell_0=k_0+k_1x,
\qquad
\ell_1=k_2y.
\]

Значение линии в \(\mathbb F_{q^4}\):

\[
\ell(T)=\ell_0+\ell_1v.
\]

Эквивалентность текущему sparse hot path следует прямой подстановкой:

\[
k_0+k_1x
=
(c_l-c_{4c})-u\,c_jx,
\]

\[
k_2y
=
u\,c_hy.
\]

### 4.4. Сложение

Исходные arkworks-коэффициенты сложения:

\[
(c_{l1},c_{rz}).
\]

Обозначим:

\[
a_0=c_{rz},
\qquad
a_1=c_{l1}.
\]

Для фиксированной точки \(Q\) или \(S\) заранее известны координаты:

\[
x_{\mathrm{tw}},
\qquad
y_{\mathrm{tw}}.
\]

В фиксированную таблицу записываются:

\[
k_0=u\,a_0,
\]

\[
k_1=-(y_{\mathrm{tw}}a_0-x_{\mathrm{tw}}a_1),
\]

\[
k_2=-a_1.
\]

Для точки \(T=(x,y)\in G_1\):

\[
\ell_0=k_0y,
\qquad
\ell_1=k_1+k_2x,
\]

\[
\ell(T)=\ell_0+\ell_1v.
\]

Для второй пары используется \(T=-R=(R_x,-R_y)\).

Для signed-digit цифры \(+1\) в нормализацию входит \(y_{\mathrm{tw}}\) фиксированной точки. Для цифры \(-1\) используется \(-y_{\mathrm{tw}}\). В отрицательном tail снова используется \(y_{\mathrm{tw}}\). Знак выбирается при построении фиксированной таблицы; verifier не принимает его как отдельный witness.

Эквивалентность текущему sparse hot path:

\[
k_0y=u\,a_0y,
\]

\[
\begin{aligned}
k_1+k_2x
&=
-(y_{\mathrm{tw}}a_0-x_{\mathrm{tw}}a_1)-a_1x\\
&=
-\left(y_{\mathrm{tw}}a_0+(x-x_{\mathrm{tw}})a_1\right).
\end{aligned}
\]

### 4.5. Проверка предварительной обработки

Rust backend обязан построить нормализованную таблицу из канонических arkworks-коэффициентов и проверить:

1. совпадение значений линий с arkworks для тестовых точек;
2. совпадение значений с текущим Solidity/Yul hot path;
3. детерминированность `rootFixed`;
4. корректное число строк каждого типа.

### 4.6. Какие значения входят в `rootFixed`

`rootFixed` связывает не исходную таблицу из \(2048\) строк, а ее низкостепенное продолжение на домен \(D\). Предварительная обработка выполняется в следующем порядке:

1. строится расписание из раздела 5;
2. для каждой строки основного домена \(H\) записываются девять transition selectors, два boundary selectors и шесть координат \((k_0,k_1,k_2)\);
3. если строка не использует линию, шесть координат линии равны нулю;
4. каждый из семнадцати столбцов интерполируется многочленом степени меньше \(N\);
5. каждый многочлен вычисляется на LDE-домене \(D\);
6. LDE-строки фиксируются Merkle-деревом `rootFixed`.

Таким образом, раскрытие fixed leaf в случайной точке \(x\in D\) содержит:

```text
selectors(x): 11 Fq
line(x):       6 Fq
```

и относится к значениям интерполированных столбцов в точке \(x\), а не к одной конкретной строке исходного ate-цикла. Исходная таблица и ее LDE сохраняются как отдельные воспроизводимые артефакты:

```text
fixed_table_h.bin
fixed_table_lde.bin
```

## 5. Пошаговая микротрасса

### 5.1. Расписание операций

ate-параметр MNT4-753 представлен \(377\) signed-digit цифрами. Основной цикл содержит \(376\) итераций:

```text
positive digits: 66
negative digits: 57
non-zero digits: 123
zero digits:     253
```

На каждой итерации выполняются:

1. `SQR`: возведение аккумулятора в квадрат;
2. `DBL_P`: умножение на линию удвоения для \((P,Q)\);
3. `DBL_R`: умножение на линию удвоения для \((-R,S)\).

Для каждой ненулевой signed-digit цифры дополнительно выполняются:

4. `ADD_P`: умножение на линию сложения для \((P,Q)\);
5. `ADD_R`: умножение на линию сложения для \((-R,S)\);
6. `MUL_CINV`, если цифра равна \(+1\), либо `MUL_C`, если цифра равна \(-1\).

После основного цикла добавляется отрицательный tail:

7. `ADD_P`;
8. `ADD_R`;
9. `MUL_FROB_CINV`.

Число проверяемых операций:

\[
376\cdot 3+123\cdot 3+3=1500.
\]

### 5.2. Размер трассового домена

Выбирается:

\[
N=2048.
\]

Строки трассы:

```text
0..1499   реальные операции;
1500      финальное состояние F = 1;
1500..2046 HOLD-переходы;
2047      STOP-строка без перехода.
```

Число `HOLD`-переходов:

\[
2046-1500+1=547.
\]

Последняя строка не содержит перехода, поскольку циклический сдвиг вернул бы трассу к начальному состоянию \(c^{-1}\), а не к единице.

### 5.3. Столбцы witness-трассы

Witness-трасса содержит только текущий аккумулятор:

\[
F(X)=f_0(X)+f_1(X)Z+f_2(X)Z^2+f_3(X)Z^3.
\]

Итого:

```text
4 witness columns over Fq
```

Состояние в строке \(r\) — значение аккумулятора до выполнения операции строки \(r\).

### 5.4. Фиксированные столбцы

Фиксированная таблица содержит:

```text
s_sqr
s_dbl_p
s_dbl_r
s_add_p
s_add_r
s_mul_c

```

Итого:

```text
17 fixed columns over Fq
```

В строке `STOP` все transition selectors равны нулю.

## 6. AIR: точные ограничения переходов

### 6.1. Общая форма

Пусть:

\[
F=F(X),
\qquad
F'=F(\omega X).
\]

Для каждой строки выполняется ровно одно активное отношение перехода.

### 6.2. Возведение в квадрат

При `s_sqr = 1`:

\[
F'=\operatorname{Sqr}(F).
\]

Координаты \(\operatorname{Sqr}\) заданы формулами раздела 3.3.

### 6.3. Умножение на линию удвоения

Для `DBL_P`:

\[
L_P=(k_0+k_1P_x)+(k_2P_y)v,
\]

\[
F'=F\cdot L_P.
\]

Для `DBL_R`:

\[
L_R=(k_0+k_1R_x)-(k_2R_y)v,
\]

\[
F'=F\cdot L_R.
\]

Знак меняется, поскольку вычисляется линия в точке \(-R\).

### 6.4. Умножение на линию сложения

Для `ADD_P`:

\[
L_P=(k_0P_y)+(k_1+k_2P_x)v,
\]

\[
F'=F\cdot L_P.
\]

Для `ADD_R`:

\[
L_R=-(k_0R_y)+(k_1+k_2R_x)v,
\]

\[
F'=F\cdot L_R.
\]

### 6.5. Умножение на свидетельство финальной экспоненты

Для `MUL_C`:

\[
F'=F\cdot c.
\]

Для `MUL_CINV`:

\[
F'=F\cdot c^{-1}.
\]

Для `MUL_FROB_CINV`:

\[
F'=F\cdot\operatorname{Frob}_{q}(c^{-1}).
\]

### 6.6. Дополнение трассы

Для `HOLD`:

\[
F'=F.
\]

### 6.7. Граничные условия

В строке `s_init`:

\[
F=c^{-1}.
\]

В строке `s_final`:

\[
F=1.
\]

## 7. Quotient polynomial

### 7.1. Трассовый домен

Пусть:

\[
H=\{1,\omega,\ldots,\omega^{N-1}\},
\qquad N=2048,
\]

где \(\omega\) — примитивный корень порядка \(N\).

Обнуляющий многочлен:

\[
Z_H(X)=X^N-1.
\]

### 7.2. Объединение ограничений

После фиксации `rootTrace` verifier получает challenge:

\[
\beta\in\mathbb F_q.
\]

Все координатные ограничения объединяются:

\[
N_\beta(X)=\sum_j \beta^j C_j(X).
\]

Порядок \(C_j\) фиксирован:

```text
for opcode in [
    SQR,
    DBL_P,
    DBL_R,
    ADD_P,
    ADD_R,
    MUL_C,
    MUL_CINV,
    MUL_FROB_CINV,
    HOLD
]:
    for coordinate in [0, 1, 2, 3]:
        append transitionConstraint(opcode, coordinate)

for coordinate in [0, 1, 2, 3]:
    append initBoundaryConstraint(coordinate)

for coordinate in [0, 1, 2, 3]:
    append finalBoundaryConstraint(coordinate)
```

Итого:

\[
9\cdot4+4+4=44
\]

ограничения. Строка `STOP` имеет нулевые transition selectors и не создает wrap-переход.

Для корректной трассы:

\[
Z_H(X)\mid N_\beta(X).
\]

Quotient polynomial:

\[
Q(X)=\frac{N_\beta(X)}{Z_H(X)}.
\]

### 7.3. Оценка степени

Максимальную степень имеют line-переходы:

\[
\deg C_j\le 3(N-1)=6141.
\]

Следовательно:

\[
\deg Q\le 6141-2048=4093<4096=2N.
\]

Quotient разбивается на два сегмента степени меньше \(N\):

\[
Q(X)=Q_0(X)+X^NQ_1(X),
\]

\[
\deg Q_0<N,
\qquad
\deg Q_1<N.
\]

## 8. LDE и Merkle-деревья

### 8.1. Расширенный домен

Выбирается blowup factor:

\[
B=16.
\]

Размер LDE-домена:

\[
M=N\cdot B=32768.
\]

Пусть:

\[
D=\{\gamma,\gamma\eta,\ldots,\gamma\eta^{M-1}\},
\]

где \(\eta\) имеет порядок \(M\), а \(\eta^{16}=\omega\). Множитель \(\gamma\) выбирается так, чтобы \(D\cap H=\varnothing\).

### 8.2. Порядок листьев

Листья Merkle-деревьев хранятся в bit-reversed порядке. Тогда точки:

\[
x,
\qquad
-x
\]

располагаются рядом и раскрываются одним парным Merkle-путем.

Если точка домена имеет показатель \(i\), ее физическая позиция в дереве:

\[
\operatorname{leafPosition}
=
\operatorname{bitReverse}(i,\log_2 M).
\]

Для показателей \(i\) и \(i+M/2\) физические позиции отличаются только младшим битом и образуют одну соседнюю пару. В `leafHash` из раздела 8.4 передается именно физическая позиция листа.

### 8.3. Деревья

Используются четыре типа деревьев:

| Дерево | Содержимое листа | Размер листа |
|---|---:|---:|
| `rootTrace` | 4 значения \(\mathbb F_q\) | 384 байта |
| `rootFixed` | 17 значений \(\mathbb F_q\) | 1632 байта |
| `rootQuot` | 2 значения \(\mathbb F_q\) | 192 байта |
| `rootDeep` | 1 значение \(\mathbb F_q\) | 96 байт |

`rootFixed` вычисляется при подготовке verifier-а и хранится в контракте.

### 8.4. Хеширование Merkle-деревьев

Все листья содержат элементы \(\mathbb F_q\) в каноническом big-endian представлении. Для каждого дерева задается однобайтовый `treeTag`:

```text
0x01 TRACE
0x02 FIXED
0x03 QUOTIENT
0x04 DEEP
0x11 FRI_LEVEL_1
...
0x17 FRI_LEVEL_7
```

Хеш листа:

```text
leafHash = keccak256(
    0x00 ||
    treeTag ||
    u32be(leafIndex) ||
    u32be(payloadLength) ||
    payload
)
```

Хеш внутренней вершины:

```text
nodeHash = keccak256(
    0x01 ||
    treeTag ||
    u8(level) ||
    leftChild ||
    rightChild
)
```

Здесь `level = 0` соответствует родителю листьев. Порядок `leftChild`, `rightChild` определяется bit-reversed индексом листа. Rust backend и Solidity verifier должны использовать буквально одинаковую функцию.

### 8.5. Фиксация конфигурации

Конфигурация verifier-а связывается одним digest:

```text
configDigest = keccak256(
    0xE0 ||
    u16be(version) ||
    U768(q) ||
    U768(r) ||
    u32be(N) ||
    u32be(M) ||
    u8(blowupFactor) ||
    Fq(omega) ||
    Fq(eta) ||
    Fq(gamma) ||
    encodeG2(Q) ||
    encodeG2(S) ||
    rootFixed
)
```

Здесь `U768(x)` означает фиксированную 96-байтовую big-endian кодировку целого числа, а `Fq(x)` — такую же кодировку канонического элемента поля. Параметры конфигурации и `rootFixed` задаются при развертывании verifier-а и не принимаются от пользователя во время проверки.

### 8.6. Проверка параметров домена

Rust backend обязан проверить:

\[
M\mid(q-1),
\]

\[
\eta^M=1,
\qquad
\eta^{M/2}\ne1,
\]

\[
\omega=\eta^{16},
\]

\[
\omega^N=1,
\qquad
\omega^{N/2}\ne1,
\]

\[
\gamma^M\ne1.
\]

Последнее условие гарантирует:

\[
D\cap H=\varnothing.
\]

Параметры домена являются частью `configDigest`. Для чистового контракта они генерируются один раз, проверяются Rust-тестом и встраиваются как фиксированные константы. Контракт не принимает их от вызывающей стороны.

Используется единственный детерминированный способ построения:

\[
g=17,
\]

\[
\eta=g^{(q-1)/M},
\]

\[
\omega=\eta^{16},
\]

\[
\gamma=g.
\]

Значение \(g=17\) — канонический генератор поля MNT4-753, зафиксированный в arkworks. Solidity verifier хранит:

```text
gamma
etaPow2[0..14]
omega
```

где:

\[
\operatorname{etaPow2}[j]=\eta^{2^j}.
\]

Это позволяет вычислять:

\[
x=\gamma\eta^{\operatorname{index}}
\]

не более чем за пятнадцать условных умножений в \(\mathbb F_q\).

## 9. DEEP-комбинация

### 9.1. Внешняя точка

После фиксации `rootQuot` transcript формирует:

\[
z\in\mathbb F_q,
\]

причем:

\[
z\notin H,
\qquad
z\notin D.
\]

Если полученное значение принадлежит запрещенному множеству, применяется детерминированный повтор с увеличенным счетчиком.

При воспроизведении transcript Solidity verifier явно проверяет:

\[
z^N\ne 1,
\]

\[
z^M\ne \gamma^M.
\]

Первое условие означает \(z\notin H\), второе — \(z\notin D\). Если хотя бы одно условие нарушено, verifier увеличивает счетчик и повторяет вывод challenge. Поскольку \(\omega D=D\), из \(z\notin D\) также следует:

\[
\omega z\notin D.
\]

### 9.2. OOD-значения

Доказательство содержит:

```text
trace(z):        4 Fq
trace(omega*z):  4 Fq
fixed(z):       17 Fq
quotient(z):     2 Fq
```

Итого:

```text
27 Fq elements
```

Verifier проверяет:

\[
Q(z)\cdot Z_H(z)=N_\beta(z),
\]

где:

\[
Q(z)=Q_0(z)+z^NQ_1(z).
\]

### 9.3. DEEP polynomial

После фиксации OOD-значений transcript формирует:

\[
\alpha\in\mathbb F_q.
\]

Пусть \(T_i\) — четыре трассовых столбца, \(K_i\) — семнадцать фиксированных столбцов, \(Q_i\) — два quotient-сегмента. Строится:

\[
\begin{aligned}
D_{\mathrm{deep}}(X)
={}&
\sum_{i=0}^{3}\alpha^i
\frac{T_i(X)-T_i(z)}{X-z}\\
&+
\sum_{i=0}^{16}\alpha^{4+i}
\frac{K_i(X)-K_i(z)}{X-z}\\
&+
\sum_{i=0}^{1}\alpha^{21+i}
\frac{Q_i(X)-Q_i(z)}{X-z}\\
&+
\sum_{i=0}^{3}\alpha^{23+i}
\frac{T_i(X)-T_i(\omega z)}{X-\omega z}.
\end{aligned}
\]

Всего объединяются:

```text
27 terms
```

Prover вычисляет значения \(D_{\mathrm{deep}}\) на LDE-домене и фиксирует `rootDeep`.

## 10. Неинтерактивный transcript

### 10.1. Fiat-Shamir

Схема не требует интерактивного verifier-а. Все challenges получаются через Fiat-Shamir-преобразование:

```text
initialState = keccak256("MNT4-MICROTRACE-DEEP-FRI-V1")

absorb(label, payload):
    transcriptState = keccak256(
        0xA0 ||
        transcriptState ||
        u16be(label.length) ||
        label ||
        u32be(payload.length) ||
        payload
    )

challenge(label, counter):
    digest = keccak256(
        0xC0 ||
        transcriptState ||
        u16be(label.length) ||
        label ||
        u32be(counter)
    )
```

MNT4-753 base field больше \(2^{256}\), поэтому 256-битный digest можно однозначно встроить в \(\mathbb F_q\) без modular reduction.

Все `payload` сериализуются в том же порядке, в котором перечислены в разделе 10.2. Не допускается конкатенация переменных массивов без префикса длины.

### 10.2. Порядок фиксации

Порядок transcript обязателен:

```text
1. absorb("public", version || profile || configDigest || rootFixed || P || R || c || cInv)
2. absorb("trace-root", rootTrace)
3. beta <- challenge("beta")
4. absorb("quotient-root", rootQuot)
5. z <- challengeOutsideDomains("z")
6. absorb("ood", OOD bundle)
7. alpha <- challenge("alpha")
8. absorb("deep-root", rootDeep)
9. rho0 <- challenge("rho-0")
10. absorb("fri-root-1", rootFri1)
11. rho1 <- challenge("rho-1")
...
16. absorb("fri-root-7", rootFri7)
17. rho7 <- challenge("rho-7")
18. absorb("fri-final", finalPolynomialCoefficients)
19. querySeed <- challenge("query-seed")
20. derive unique query indices without replacement
```

Порядок нельзя менять без увеличения `version`.

Query indices выводятся повторным хешированием:

```text
candidate = u256be(
    keccak256(0xD0 || querySeed || u32be(counter))
) mod (M / 2)
```

Повторы отбрасываются. После получения требуемого числа уникальных индексов массив сортируется по возрастанию. Ограничение:

\[
0\le \operatorname{index}<M/2
\]

фиксирует пару:

\[
\operatorname{index},
\qquad
\operatorname{index}+M/2.
\]

## 11. DEEP-FRI folding

### 11.1. Первый раунд

Для пары значений:

\[
p(x),
\qquad
p(-x)
\]

вычисляются:

\[
p_{\mathrm{even}}(x^2)=\frac{p(x)+p(-x)}{2},
\]

\[
p_{\mathrm{odd}}(x^2)=\frac{p(x)-p(-x)}{2x}.
\]

После challenge \(\rho\):

\[
p_{\mathrm{next}}(x^2)
=
p_{\mathrm{even}}(x^2)+
\rho\,p_{\mathrm{odd}}(x^2).
\]

### 11.2. Как verifier выполняет деления

В EVM инверсия 753-битного элемента существенно дороже умножения. Поэтому proof содержит для каждого запроса:

\[
x^{-1},
\]

\[
(x-z)^{-1},
\qquad
(x-\omega z)^{-1},
\]

\[
(-x-z)^{-1},
\qquad
(-x-\omega z)^{-1}.
\]

Значение:

\[
x=\gamma\eta^{\operatorname{index}}
\]

verifier выводит самостоятельно из конфигурации и индекса запроса. Степень вычисляется бинарным методом по заранее зафиксированным степеням \(\eta^{2^j}\).

Verifier не доверяет этим значениям и проверяет пять равенств:

\[
x\cdot x^{-1}=1,
\]

\[
(x-z)\cdot(x-z)^{-1}=1,
\]

\[
(x-\omega z)\cdot(x-\omega z)^{-1}=1,
\]

\[
(-x-z)\cdot(-x-z)^{-1}=1,
\]

\[
(-x-\omega z)\cdot(-x-\omega z)^{-1}=1.
\]

После первого FRI-раунда текущая точка заменяется на \(x^2\). Ее инверсия получается без нового witness:

\[
(x^2)^{-1}=(x^{-1})^2.
\]

Поэтому для всех последующих раундов достаточно последовательно возводить текущую инверсию в квадрат. Деление на \(2x\) выполняется умножением на:

\[
\frac{x^{-1}}{2}.
\]

### 11.3. Проверка одного FRI-запроса

Пусть исходный индекс запроса:

\[
i_0\in\{0,\ldots,M/2-1\}.
\]

На нулевом уровне:

\[
n_0=M,
\qquad
x_0=\gamma\eta^{i_0}.
\]

Verifier выполняет следующие действия:

1. раскрывает trace-, fixed- и quotient-листья для показателей \(i_0\) и \(i_0+M/2\);
2. проверяет их парные Merkle-пути;
3. вычисляет \(D_{\mathrm{deep}}(x_0)\) и \(D_{\mathrm{deep}}(-x_0)\);
4. по `deepPathAbovePair` проверяет, что вычисленные значения входят в `rootDeep`;
5. выполняет folding с challenge \(\rho_0\) и получает значение следующего уровня.

Для уровня:

\[
\ell\in\{1,\ldots,7\}
\]

используются:

\[
n_\ell=\frac{M}{2^\ell},
\]

\[
i_\ell=i_0\bmod n_\ell,
\]

\[
j_\ell=i_\ell\bmod \frac{n_\ell}{2}.
\]

Текущая точка:

\[
x_\ell=x_0^{2^\ell}.
\]

Если:

\[
i_\ell=j_\ell,
\]

то уже вычисленное значение относится к \(x_\ell\), а переданный sibling — к \(-x_\ell\). Если:

\[
i_\ell=j_\ell+\frac{n_\ell}{2},
\]

порядок обратный. Verifier обязан восстановить каноническую пару:

\[
\bigl(p_\ell(x_\ell),p_\ell(-x_\ell)\bigr),
\]

проверить ее путь до `rootFri` соответствующего уровня и выполнить следующий fold.

Физическая позиция листа уровня \(\ell\):

\[
\operatorname{bitReverse}(i_\ell,\log_2 n_\ell).
\]

После восьмого folding-раунда verifier получает:

\[
p_8(x_0^{2^8}).
\]

Это значение сравнивается с вычислением переданного финального многочлена:

\[
p_{\mathrm{final}}(x_0^{2^8})
=
\sum_{j=0}^{7}a_j\left(x_0^{2^8}\right)^j.
\]

### 11.4. Число раундов

Выполняются восемь folding-раундов. После них размер домена:

\[
\frac{32768}{2^8}=128.
\]

Финальный многочлен имеет степень меньше:

\[
\frac{2048}{2^8}=8.
\]

Доказательство передает восемь коэффициентов финального многочлена.

### 11.5. Параметры профилей

Один и тот же verifier поддерживает два профиля:

| Профиль | Число запросов | Назначение |
|---|---:|---|
| `benchmark-32q` | 32 | измерение gas и принципиального выигрыша |
| `conservative-128q` | 128 | консервативный профиль для дальнейшей оценки |

Спецификация не приписывает профилям неподтвержденную точную криптографическую стойкость. Rust backend обязан формировать `security_report.json`, в котором выбранная теорема DEEP-FRI применяется к фактическим параметрам домена, степени, blowup factor, числу раундов и числу запросов.

## 12. Бинарный формат proof

### 12.1. Общие правила

Все целые числа кодируются big-endian. Все элементы \(\mathbb F_q\) занимают ровно \(96\) байт и должны быть меньше модуля \(q\).

Merkle-деревья и `proof.bin` всегда используют обычные канонические остатки:

\[
0\le a<q.
\]

Montgomery-представление является внутренней деталью Solidity/Yul арифметики. Solidity parser проверяет каноничность входа и после этого переводит значение в Montgomery-представление. При вычислении Merkle-хеша используется исходное каноническое представление.

Координаты составных объектов сериализуются в порядке:

```text
Fq2(a0 + a1*u):        a0 || a1
Fq4(b0 + b1*v):        encodeFq2(b0) || encodeFq2(b1)
G1(x, y):              encodeFq(x) || encodeFq(y)
G2(x, y):              encodeFq2(x) || encodeFq2(y)
```

Parser обязан потребить proof целиком. Любые хвостовые байты приводят к отклонению.

### 12.2. Заголовок

```text
u32 magic
u16 version
u8  profile
u8  queryCount
u32 domainSize
u32 traceSize

bytes32 rootTrace
bytes32 rootQuot
bytes32 rootDeep
bytes32 rootFri1
...
bytes32 rootFri7

Fq[4]  traceAtZ
Fq[4]  traceAtOmegaZ

```

Фиксированные значения:

```text
magic   = 0x4d344446  // ASCII "M4DF"
version = 0x0001
profile = 0x01       // benchmark-32q
       or 0x02       // conservative-128q
```

Verifier отклоняет неизвестные версии, профили и несоответствие `queryCount` выбранному профилю.

Размер заголовка:

\[
16+10\cdot 32+35\cdot 96=3696\ \text{байт}.
\]

### 12.3. Вспомогательные значения запросов

Индексы запросов не передаются в `proof.bin`: verifier самостоятельно выводит их из `querySeed`, сортирует и удаляет повторы по правилу раздела 10.2.

Для каждого индекса в отсортированном массиве proof передает:

```text
Fq xInv
Fq xMinusZInv
Fq xMinusOmegaZInv
Fq negXMinusZInv
Fq negXMinusOmegaZInv
```

Размер вспомогательных значений одного запроса:

\[
5\cdot 96=480\ \text{байт}.
\]

### 12.4. Почему используются Merkle-multiproof

Передавать независимый Merkle-путь для каждого запроса корректно, но неэкономично: верхние части путей многократно повторяются. Чтобы измерение gas не содержало заведомо устранимый перерасход calldata, production-формат использует детерминированный компактный multiproof.

Для дерева с \(L=2^d\) листьями и отсортированным множеством открываемых физических позиций \(O_0\) verifier строит ожидаемый frontier:

```text
known = O_0
for level in 0..d-1:
    next = empty set
    for index in known in ascending order:
        sibling = index xor 1
        if sibling not in known:
            append (level, sibling) to frontier
        insert (index >> 1) into next
    known = unique(next)
```

Доказательство передает хеши frontier-вершин ровно в этом порядке. Индексы frontier-вершин передавать не нужно: verifier выводит их самостоятельно. Лишний, пропущенный или переставленный хеш приводит к отклонению proof.

### 12.5. Наборы открываемых листьев

Для `rootTrace`, `rootFixed`, `rootQuot` и `rootDeep` verifier использует один и тот же базовый набор:

\[
O_{\mathrm{base}}
=
\bigcup_{i_0}
\left\{
\operatorname{bitReverse}(i_0,\log_2M),
\operatorname{bitReverse}(i_0+M/2,\log_2M)
\right\}.
\]

Для `rootFri` уровня \(\ell\in\{1,\ldots,7\}\):

\[
n_\ell=\frac{M}{2^\ell},
\]

\[
i_\ell=i_0\bmod n_\ell,
\]

\[
O_{\mathrm{fri},\ell}
=
\bigcup_{i_0}
\left\{
\operatorname{bitReverse}(i_\ell,\log_2n_\ell),
\operatorname{bitReverse}\left(
(i_\ell+n_\ell/2)\bmod n_\ell,
\log_2n_\ell
\right)
\right\}.
\]

Во всех множествах позиции сортируются по возрастанию и удаляются повторы.

### 12.6. Формат секций multiproof

После вспомогательных значений запросов следуют одиннадцать секций в фиксированном порядке:

```text
TRACE
FIXED
QUOTIENT
DEEP
FRI_LEVEL_1
...
FRI_LEVEL_7
```

Для секции с передаваемыми листьями:

```text
u16 openedLeafCount
payload[openedLeafCount]
u16 frontierHashCount
bytes32 frontierHashes[frontierHashCount]
```

Verifier требует:

1. `openedLeafCount` совпадает с числом ожидаемых уникальных позиций;
2. листья следуют в порядке возрастания ожидаемых физических позиций;
3. `frontierHashCount` совпадает с длиной frontier, выведенного алгоритмом раздела 12.4;
4. восстановленный корень равен ожидаемому корню.

Размер payload одного листа:

| Секция | Payload |
|---|---:|
| `TRACE` | \(4\cdot96=384\) байта |
| `FIXED` | \(17\cdot96=1632\) байта |
| `QUOTIENT` | \(2\cdot96=192\) байта |
| `FRI_LEVEL_1..7` | \(96\) байт |

Для секции `DEEP` leaf payload не передается: verifier вычисляет значения \(D_{\mathrm{deep}}(x)\) самостоятельно из раскрытий `TRACE`, `FIXED`, `QUOTIENT` и OOD bundle. Секция имеет вид:

```text
u16 frontierHashCount
bytes32 frontierHashes[frontierHashCount]
```

### 12.7. Итоговый размер proof

Точный размер multiproof зависит от Fiat-Shamir-запросов, поскольку пересечения Merkle-путей зависят от полученных индексов. Rust backend обязан сохранять точное значение в `metrics.json`.

Обозначим:

\[
\lambda
\]

число запросов,

\[
L_t,L_f,L_q,L_1,\ldots,L_7
\]

число уникальных открываемых листьев соответствующих секций,

\[
B_t,B_f,B_q,B_d,B_1,\ldots,B_7
\]

число frontier-хешей.

Тогда:

\[
\begin{aligned}
|\pi|
={}&3696+480\lambda+4\cdot 10+2\\
&+384L_t+1632L_f+192L_q\\
&+96\sum_{\ell=1}^{7}L_\ell\\
&+32\left(B_t+B_f+B_q+B_d+\sum_{\ell=1}^{7}B_\ell\right).
\end{aligned}
\]

Слагаемое:

\[
4\cdot10+2=42
\]

соответствует двум счетчикам в десяти секциях с payload и одному счетчику секции `DEEP`.

Для контроля реализации используется также простая верхняя оценка через независимые пути:

\[
|\pi|_{\mathrm{independent}}
=
3696+\lambda\cdot9604.
\]

| Профиль | Верхняя оценка независимых путей |
|---|---:|
| `benchmark-32q` | 311024 байта |
| `conservative-128q` | 1233008 байт |

Фактический компактный multiproof обязан быть не больше соответствующей верхней оценки.

Публичные входы \(P,R,c,c^{-1}\) дополнительно занимают:

\[
12\cdot96=1152\ \text{байта}.
\]

### 12.8. Оценка calldata gas

При консервативной оценке \(16\) gas за ненулевой байт независимые пути дают верхнюю оценку:

| Профиль | Proof + public inputs |
|---|---:|
| `benchmark-32q` | 4994816 gas |
| `conservative-128q` | 19746560 gas |

Фактический calldata gas компактного multiproof измеряется по реальному `proof.bin`. Execution gas измеряется отдельно Foundry-тестами.

## 13. Корректность и модель безопасности

### 13.1. Полнота

Если:

1. \(P,R\in G_1\);
2. \(c\cdot c^{-1}=1\);
3. off-chain backend построил каноническую residue-рекурсию;
4. фиксированная таблица линий построена из канонических \(Q,S\);
5. witness-трасса удовлетворяет всем переходам;
6. quotient polynomial вычислен корректно;
7. Merkle-пути и DEEP-FRI proof сформированы корректно;

то verifier возвращает `true`.

### 13.2. Связывание фиксированного кэша

Пользователь не может подменить подготовленные линии без нарушения хотя бы одного условия:

1. Merkle-путь раскрытой строки ведет к `rootFixed`;
2. OOD-значения фиксированных столбцов включены в \(D_{\mathrm{deep}}\);
3. DEEP-FRI связывает случайно проверенные значения с низкостепенным продолжением тех же столбцов.

### 13.3. Связывание трассы и quotient polynomial

Пользователь не может независимо выбрать трассу, quotient polynomial и OOD bundle:

1. `rootTrace` фиксируется до \(\beta\);
2. `rootQuot` фиксируется до \(z\);
3. OOD bundle фиксируется до \(\alpha\);
4. `rootDeep` фиксируется до FRI challenges;
5. запросы выводятся только после фиксации финального многочлена.

### 13.4. Компоненты soundness

Итоговая оценка soundness должна учитывать:

1. collision resistance `keccak256`;
2. случайное объединение AIR-ограничений challenge-ом \(\beta\);
3. проверку quotient relation в точке \(z\);
4. DEEP-FRI proximity bound;
5. выбор запросов без возвращения;
6. размер множества Fiat-Shamir challenges \(2^{256}\).

Нельзя заменять эту оценку упрощенной формулой без ссылки на конкретную теорему DEEP-FRI. Реализация Rust backend обязана генерировать машиночитаемый отчет:

```text
security_report.json
```

с параметрами и итоговой численной оценкой.

### 15.2. Разделение ответственности

`MNT4MerkleDeepFriVerifier.sol`:

1. проверяет входы;
2. читает proof;
3. воспроизводит transcript;
4. проверяет OOD relation;
5. проверяет запросы;
6. возвращает `true` или `false`.

1. выполняет арифметику \(\mathbb F_q\);

1. проверяет парные Merkle-пути;

2. выводит challenges;
3. выводит уникальные query indices.

## 17. Критерии готовности

Модуль считается реализованным только при одновременном выполнении условий:

1. Rust backend строит фиксированную таблицу и `rootFixed`.
2. Нормализованные линии совпадают с arkworks и текущим Solidity/Yul hot path.
3. Расписание содержит ровно `1500` реальных микроопераций.
4. Witness-трасса имеет размер `2048`.
5. Quotient divisibility проверена Rust-тестом.
6. Proof сериализуется в фиксированном формате.
7. Компактный Merkle-multiproof передает каждый требуемый frontier-хеш не более одного раза на дерево.
8. Solidity parser не копирует proof целиком в memory.
9. Solidity verifier принимает положительные fixtures.
10. Каждый негативный тест отклоняется.
11. Оба профиля измерены через Foundry.
12. Метрики Rust backend сохранены.
13. `security_report.json` сформирован для каждого профиля.
14. Итоговый отчет сравнивает gas с текущим `article640 fixed shards`.
15. В отчете явно отделены calldata gas и execution gas.
16. Если выигрыш отсутствует, это фиксируется как отрицательный экспериментальный результат без ослабления модели безопасности.

## 18. Что именно проверяет эксперимент

Эксперимент не пытается заранее доказать, что Merkle/DEEP-FRI будет дешевле. Он проверяет это измерением.

Прямой `article640 fixed shards` verifier выполняет все \(1500\) микроопераций в EVM. Новый verifier:

1. выполняет эти операции off-chain;
2. фиксирует трассу Merkle-корнем;
3. проверяет quotient relation в случайной точке;
4. проверяет низкую степень через DEEP-FRI;
5. раскрывает только случайно выбранные строки.

Ожидаемый выигрыш по execution gas может быть частично или полностью съеден:

1. большим calldata;
2. 753-битной арифметикой DEEP-проверки;
3. Merkle-хешированием;
4. количеством запросов, необходимым для soundness.

Именно поэтому реализация и реальный gas-report необходимы: только они показывают, является ли Merkle/DEEP-FRI-путь полезным для MNT4-753 в EVM.
