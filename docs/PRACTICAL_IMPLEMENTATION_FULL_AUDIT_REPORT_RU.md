# Полный аудит практической части дипломной работы

Дата проверки: 1 июня 2026 года.

## 1. Итоговый вердикт

Практическая часть содержит большой объем воспроизводимого кода и дает
защищаемый исследовательский результат:

- реализована и проверена арифметика MNT4-753 и MNT6-753;
- построен полный on-chain baseline для MNT4-753;
- реализован оптимизированный MNT4-верификатор уравнения сопряжений по мотивам
  ePrint 2024/640;
- экспериментально подтверждено, что оптимизация финальной экспоненты снижает
  стоимость, но не устраняет доминирующую стоимость цикла Миллера;
- исследованы KZG и Merkle/FRI как способы сжатия проверки цикла Миллера;
- реализован воспроизводимый Merkle/FRI cost model и сохранен исполняемый
  DEEP-FRI отрицательный эксперимент;
- реализованы арифметические прототипы MNT4/MNT6-цикла и lollipop-305.

Однако исходный сильный тезис «готова production-библиотека, которая для
произвольного недоверенного кэша дешево заменяет MNT-сопряжение в EVM» пока
нельзя заявлять. Основные причины:

1. Самый дешевый MNT4 fixed-shards verifier использует доверенную регистрацию
   подготовленного кэша при развертывании контракта.
2. Публичные MNT4/MNT6 Article640 API не проверяют принадлежность входных точек
   кривой и нужным подгруппам.
3. MNT6-контур еще не доведен до симметричного MNT4 fixed-shards
   pairing-equation verifier-а.
4. MNT-cycle constraints-модуль является моделью количества умножений, а не
   полностью скомпилированной системой ограничений.
5. Merkle/FRI-направление честно завершено как аналитический и отрицательный
   эксперимент, а не как production-замена on-chain цикла Миллера.

Следовательно, работа готова к представлению как всестороннее исследование
границ применимости MNT-арифметики в EVM и как набор проверенных прототипов.
Перед заявлением production-готовности нужны доработки из раздела 9.

## 5. Найденные ошибки и разрывы

### P1. Доверенная регистрация кэша в MNT4 fixed-shards

`MNT4Article640FixedShardsVerifier` фиксирует адреса data-контрактов в
конструкторе и затем читает линии только из них. Это защищает от подмены кэша
пользователем во время вызова, но не доказывает, что при развертывании были
зарегистрированы именно корректные линии для Q и S.

Commitment-режим имеет ту же границу: хеш связывает вызов с зарегистрированным
blob, но сам по себе не доказывает корректность генерации линий.

**Как исправить:** выбрать и явно документировать одну модель.

1. Для fixed-Q сервиса: считать регистрацию доверенной конфигурацией, добавить
   deployment-скрипт, который воспроизводит Rust fixture и проверяет хеши.
2. Для универсального trustless API: добавить отдельное доказательство
   корректности `Q -> lines -> commitment`. Оно неизбежно потребует новых
   gas/constraints измерений.

### P1. Нет проверки точек на кривой и в подгруппе

Публичные Article640-функции принимают G1-точки, но не отклоняют точки вне
кривой и вне требуемой подгруппы. Для precompile-like API это обязательная
проверка безопасности.

**Как исправить:** добавить `isOnCurve` и subgroup policy для каждого публичного
API либо документировать строгую внешнюю предпосылку и не называть API
универсальным публичным verifier-ом. После исправления заново измерить gas.

### P1. MNT6 Article640-контур не симметричен MNT4

MNT6 реализует packed Miller loop, FE и residue-фрагменты. Но production API не
проверяет итоговое уравнение сопряжений в fixed-shards режиме. Тест residue
проверяет только ненулевой digest.

**Как исправить:** добавить fixed-shards контракт, интегрированный bool-verifier
уравнения сопряжений, отрицательные тесты подмены P/R/c/cache и gas-report.

### P1. Ошибка счетчиков MNT-cycle модели

В `mnt_cycle_full` использованы счетчики `377/124` для обеих кривых. Для
исполняемых циклов нужны:

| Кривая | Удвоения | Сложения |
|---|---:|---:|
| MNT4 | 376 | 124 |
| MNT6 | 376 | 123 |

После исправления текущей формулы модель дает:

| Слой | Было | Должно быть |
|---|---:|---:|
| MNT4 prepared relation | 24 178 | 24 126 |
| MNT6 prepared relation | 49 120 | 48 942 |

Кроме того, это ручная модель multiplication constraints, а не полностью
скомпилированная constraint system.

**Как исправить:** поправить счетчики, обновить отчеты и везде называть метрики
модельной оценкой. Для более сильного тезиса реализовать реальные gadgets и
получить число ограничений компиляцией.

### P1. Merkle/FRI не реализован как production-замена цикла Миллера

Основной каталог содержит корректно размеченную ordinary-FRI модель стоимости.
В Article640 реализован Merkle opening layer, но не полный low-degree FRI
verifier. Архивный DEEP-FRI прототип полезен как отрицательный эксперимент, но
не закрывает production soundness.

**Как исправить:** не заявлять полноценную Merkle/FRI замену. Для продолжения
ветки сначала численно выбрать soundness-профиль по конкретной теореме, затем
реализовать оптимизированный verifier. Текущая cost model показывает, что это
направление не является приоритетным.

### P2. Наивный Tate baseline частичный

Наивный baseline содержит фрагмент финальной экспоненты и экстраполяцию, а не
полное неоптимизированное Tate-сопряжение.

**Как исправить:** либо переименовать модуль и таблицы в «наивная модель
стоимости», либо реализовать полный медленный baseline и запускать его отдельно.

### P2. MNT6 streaming code-shards hot path не финализирован

Потоковая ветка создает временные структуры и копирует данные. Ее газовые тесты
отключены. Поэтому нельзя утверждать, что MNT6 достиг MNT4-уровня
низкоуровневой оптимизации.

**Как исправить:** оставить только packed pointer API, добавить реальные
fixed-shards тесты и сравнить blob/memory/code-shards режимы.

### P2. Корневой скрипт не запускает Merkle/FRI cost model

`scripts/run_all.sh` не включает ordinary-FRI cost model. Backend-скрипты также
не выполняют автоматическое сравнение с tracked fixture-ами.

**Как исправить:** добавить отдельный `run_audit_all.sh` или расширить
`run_all.sh`; автоматически регенерировать fixture во временную директорию и
выполнять `cmp`.

### P2. Документация не полностью синхронизирована

В документах остались абсолютные пути старого репозитория, прежние имена
Merkle/FRI модулей и старые числа MNT-cycle модели.

**Как исправить:** выполнить редакторский проход после исправления счетчиков,
оставить относительные пути и единую таблицу статусов.

## 7. Сравнение с Sonobe

Официальный README Sonobe указывает ориентир порядка 9 млн ограничений для
CycleFold decider и время порядка трех минут на указанной авторами машине.
Локальные MNT-cycle числа 24 126 и 48 942 нельзя напрямую делить на 9 млн:
они описывают подготовленные relation-фрагменты, а Sonobe-число относится к
целому decider-контуру.

Корректная формулировка результата:

> MNT4/MNT6 cycle-native арифметика дает перспективный малый строительный блок
> для будущего folding. Прямое количественное превосходство над полным Sonobe
> decider еще не доказано, потому что в проекте не собран полный folding-контур.

## 8. Оставшиеся возможности оптимизации

### MNT4

Для прямого исполнения в EVM исследованы Montgomery/CIOS, Barrett, FIOS,
Comba/SOS, branchless reduction, lazy reduction, Karatsuba, sparse lines,
fused hot path, blob и code-shards режимы. Очевидного технического улучшения
более чем на 10% без смены архитектуры не найдено.

### MNT6

Еще возможна доводка streaming code-shards, pointer-swap и all-stack Yul для
Fq3/Fq6. Это может дать проценты или десятки процентов, но не порядок.

### Архитектурная граница

Для существенного выигрыша нужно менять не отдельную Yul-функцию, а способ
проверки цикла Миллера: precompile, доказательство корректности кэша,
folding-native relation или компактная полиномиальная схема с приемлемым PCS.

## 9. План исправлений перед финальной фиксацией

### Обязательные исправления текста и статуса

1. Исправить счетчики MNT-cycle и числа `24 126`, `48 942`.
2. Явно назвать MNT4 fixed-shards режимом доверенной предрегистрации кэша.
3. Не называть MNT6 контур завершенным Article640 verifier-ом.
4. Не называть Merkle/FRI ветку production-реализацией.
5. Не сравнивать relation-фрагменты напрямую с полным Sonobe decider.
6. Переименовать наивный Tate baseline либо реализовать его полностью.

### Обязательные исправления для production-like API

1. Добавить проверки входных точек.
2. Реализовать MNT6 fixed-shards pairing-equation bool-verifier.
3. Добавить автоматические fixture cross-check скрипты.

### Отдельная будущая исследовательская работа

1. Скомпилировать полноценные MNT-cycle constraint gadgets.
2. Если Merkle/FRI направление продолжается, сначала закрыть численный
   soundness-профиль и только затем оптимизировать Solidity verifier.
3. Для универсального недоверенного кэша реализовать доказательство
   `Q -> lines -> commitment`.
