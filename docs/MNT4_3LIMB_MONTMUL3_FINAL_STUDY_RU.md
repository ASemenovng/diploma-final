# 3-limb умножение MNT4-753 в EVM

## 1. Что реализовано

Базовое поле MNT4-753 имеет модуль `p` длиной 753 бита. Один элемент поля не помещается в одно слово EVM, поэтому хранится тремя 256-битными словами:

```text
x = x0 + 2^256*x1 + 2^512*x2,    0 <= x < p.
```

В основном коде используется Montgomery-представление. `R = 2^768`, обычное значение `x` хранится как

```text
x_M = x * R mod p.
```

Основная функция умножения:

```text
montMul3(a_M, b_M) = a_M * b_M * R^{-1} mod p = a*b*R mod p.
```
Код находится в:

```text
arithmetic/mnt4_3limb/src/BigIntMNT.sol
```

Основная функция:

```text
BigIntMNT.montMul3
```

## 2. Почему выбран Montgomery/CIOS

В EVM нет нативной 753-битной арифметики и нет инструкции `mulhi`, которая возвращала бы старшую половину 512-битного произведения. Поэтому каждое 256x256 -> 512 умножение приходится восстанавливать через:

```text
lo = mul(x, y)
mm = mulmod(x, y, 2^256 - 1)
hi = mm - lo - carry
```

Montgomery-редукция позволяет заменить деление по модулю `p` на последовательность умножений на предвычисленную константу

```text
magic = -p^{-1} mod 2^256.
```

CIOS означает `Coarsely Integrated Operand Scanning`. Смысл схемы: не строить сначала полное 6-словное произведение `a*b`, а сразу совмещать накопление произведений и Montgomery-редукцию.

Для трех слов это особенно удобно:

```text
для i = 0,1,2:
    добавить a_i * b к аккумулятору
    выбрать m = t_0 * magic mod 2^256
    добавить m * p к аккумулятору
    отбросить младшее слово t_0
```

После каждого шага младшее слово становится нулевым по модулю `2^256`, поэтому его можно удалить сдвигом окна аккумулятора.

## 3. Псевдокод алгоритма

Ниже приведен псевдокод алгоритма, он описывает ту логику, которая реализована в `montMul3`.

```text
Вход:
    A = (a0, a1, a2)  -- Montgomery-значение aR mod p
    B = (b0, b1, b2)  -- Montgomery-значение bR mod p

Константы:
    p = (p0, p1, p2)
    magic = -p0^{-1} mod 2^256

Вспомогательная операция:
    mul512(x, y):
        вернуть lo, hi такие что x*y = lo + 2^256*hi

Алгоритм:
    t = (0, 0, 0, 0)     -- 4-словный аккумулятор

    для i = 0..2:
        u = a_i

        -- 1. Добавить строку произведения u * B.
        t[0..3] += u * (b0, b1, b2)
        то есть:
            t += u*b0
            t += 2^256 * u*b1
            t += 2^512 * u*b2
        все переносы обрабатываются явно по словам

        -- 2. Выбрать Montgomery-коэффициент.
        m = t[0] * magic mod 2^256

        -- 3. Добавить m*p так, чтобы младшее слово стало нулевым.
        t[0..3] += m * (p0, p1, p2)

        -- 4. Отбросить младшее слово.
        -- Оно равно 0 mod 2^256 по построению.
        t = (t[1], t[2], t[3], 0)

    -- После трех шагов получен кандидат r < 2p.
    r = (t[0], t[1], t[2])

    если r >= p:
        r = r - p

    вернуть r
```

## 4. Корректность

На каждом шаге выбирается:

```text
m = t_0 * (-p_0^{-1}) mod 2^256.
```

Тогда младшее слово суммы

```text
t + m*p
```

равно нулю по модулю `2^256`, потому что:

```text
t_0 + m*p_0
= t_0 - t_0*p_0^{-1}*p_0
= 0 mod 2^256.
```

Значит младшее слово можно отбросить. После трех таких шагов деление на `R = 2^768` выполнено без обычного деления:

```text
REDC(T) = T * R^{-1} mod p.
```

Так как входы уже находятся в Montgomery-представлении:

```text
T = (aR)*(bR) = abR^2,
REDC(T) = abR mod p.
```

То результат остается в том же представлении.

## 5. Стоимость в базовых операциях

Для одного `montMul3` неизбежны:

```text
9 произведений a_i*b_j
9 произведений m_i*p_j
```

Итого:

```text
18 произведений 256x256 -> 512.
```

Одно такое произведение не выполняется одной инструкцией - нужно вычислить младшую и старшую половины:

```text
lo = MUL(x, y)
mm = MULMOD(x, y, 2^256 - 1)
hi = mm - lo - carry
```

Кроме самих `mul`/`mulmod`, требуются:

- сложения слов;
- проверки переносов через `lt`;
- коррекции старших слов;
- сдвиг CIOS-окна;
- финальное условное вычитание `p`;
- `dup/swap`-операции из-за давления на стек.

## 6. Измеренная стоимость

Для реализации получены значения:

| Операция | Gas/op |
|---|---:|
| `BigIntMNT.montMul3`, internal hot path | 2,959 |

Были реализованы и измерены несколько вариантов:

| Подход | Результат |
|---|---:|
| Montgomery/CIOS | 2,959 gas/op |
| Barrett | около 46,998 gas/op |
| FIOS | около 18,509 gas/op |
| Comba/SOS | около 18,301 gas/op |

| Стандартное умножение REDC | около 5,643 gas/op |

## 8. Дальнейшая оптимизация

Существенное снижение больше чем на 10% не найдено по следующим причинам.

1. Число произведений уже минимально: `9` произведений входов и `9` произведений редукции.
2. CIOS уже совмещает умножение и редукцию, поэтому нет отдельного массива `a*b`, который можно было бы убрать.
3. Код развернут вручную в Yul; нет циклов, динамических массивов и лишних memory-структур в hot path.
4. Альтернативы Barrett/FIOS/Comba оказались существенно дороже.

## 9. Альтернатива через MODEXP

Отдельно проверялась идея считать умножение так:

```text
4ab = (a+b)^2 - (a-b)^2.
```

Тогда можно получить:

```text
s1 = (a+b)^2 mod p
s2 = |a-b|^2 mod p
diff = (s1 - s2) mod p
ab = diff / 4 mod p
```

Квадраты можно считать через `MODEXP`.

Измерения:

| Операция | Prague gas/op | Osaka gas/op |
|---|---:|---:|
| умножение через два MODEXP | 2,375 | 2,975 |
| квадрат через один MODEXP | 578 | 878 |

Квадрат получается дешевым, но весь pairing не получится сделать существенно дешевле

### 9.1. Нельзя заменить только `montSqr3`

Production-код работает в Montgomery-представлении:

```text
x_M = xR mod p.
```

Текущий square вычисляет:

```text
montSqr3(x_M) = x_M^2 R^{-1} = x^2 R mod p.
```

То есть результат остается Montgomery-элементом.

MODEXP-квадрат вычисляет другое:

```text
sqrMod3(x) = x^2 mod p.
```

Если подать в MODEXP не обычный `x`, а Montgomery-значение `x_M = xR`, то
получится:

```text
MODEXP(x_M, 2) = x^2 R^2 mod p,
```

а production-коду нужен `x^2 R mod p`. Чтобы исправить представление, нужно
дополнительно выполнить Montgomery-редукцию:

```text
x^2 R^2  ->  x^2 R.
```

Эта редукция по стоимости сопоставима с `montMul3(..., 1)`, то есть снова
требует полного 3-limb Montgomery-умножения. Поэтому локальная замена

```text
montSqr3 -> MODEXP square
```

не работает: экономия одного дешевого square теряется на приведении результата
обратно в Montgomery-представление.

### 9.2. Отказ от Montgomery

Можно рассмотреть вариант: хранить все элементы поля в
обычном представлении и заменить базовые операции так:

```text
mul(a,b) = ((a+b)^2 - (a-b)^2)/4 mod p,
sqr(a)   = a^2 mod p.
```

Тогда конверсии между Montgomery и canonical не нужны. Но появляются другие ограничения.

Во-первых, общее умножение не становится дешевле. После EIP-7883/Osaka
полноразмерное умножение через два MODEXP стоит:

```text
mulMod3_MODEXP ~= 2,975 gas/op.
```

Текущий production-вариант стоит:

```text
montMul3 ~= 2,959 gas/op.
```

При этом умножение через MODEXP все равно содержит редукции:

```text
s1 = (a+b)^2 mod p,
s2 = |a-b|^2 mod p,
diff = (s1-s2) mod p,
r = diff * 4^{-1} mod p.
```

Во-вторых, основные умножения в расширениях почти не выигрывают. Например,
умножение в `Fq2` по Karatsuba требует три умножения в `Fq`:

```text
(a0 + a1*u)(b0 + b1*u)
= (a0b0 + beta*a1b1) + ((a0+a1)(b0+b1)-a0b0-a1b1)u.
```

Следовательно нижняя часть стоимости:

```text
Fq2.mul_Montgomery >= 3 * 2,959 = 8,877 gas,
Fq2.mul_MODEXP     >= 3 * 2,975 = 8,925 gas.
```

Для `Fq4.mul` ситуация аналогична: оно строится из нескольких `Fq2.mul`, а значит из
многих общих `Fq.mul`, где MODEXP не дает выигрыша.

В-третьих, дешевле становятся только square-операции. Это полезно, но их доля
в итоговом verifier-е ограничена. В MNT4 цикл Миллера использует общий accumulator:

```text
f <- f^2 * line_Q(P) * line_S(-R).
```

То есть на раунд есть один квадрат аккумулятора в `Fq4` и несколько
умножений на line-значения. Для MNT4 число раундов порядка `376`.

Даже если сделать предположение, что все `Fq4.square`
в цикле Миллера стали бесплатными, верхняя граница экономии равна:

```text
376 * Fq4.square_gas ~= 376 * 36,606 = 13,763,856 gas.
```

Текущий fixed-shards residue verifier стоит:

```text
93,734,789 gas.
```

Даже при бесплатных `Fq4.square` получилась бы оценка:

```text
93,734,789 - 13,763,856 = 79,970,933 gas.
```


Поэтому полный отказ от Montgomery в пользу canonical arithmetic + MODEXP
не снижает существенно стоимость pairing verifier-а, потому что основной вклад дают не изолированные
квадраты в базовом поле, а большое число общих умножений в `Fq2/Fq4` и умножений на значения линий.
