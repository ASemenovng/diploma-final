# 3-limb умножение MNT4-753 в EVM

## 1. Что именно реализовано

Базовое поле MNT4-753 имеет модуль `p` длиной 753 бита. Один элемент поля не помещается в одно слово EVM, поэтому хранится тремя 256-битными словами:

```text
x = x0 + 2^256*x1 + 2^512*x2,    0 <= x < p.
```

В production-коде используется Montgomery-представление. Если `R = 2^768`, то обычное значение `x` хранится как

```text
x_M = x * R mod p.
```

Основная функция умножения:

```text
montMul3(a_M, b_M) = a_M * b_M * R^{-1} mod p = a*b*R mod p.
```

То есть результат остается в Montgomery-представлении. Это важно: вся длинная цепочка арифметики в цикле Миллера работает без постоянных переводов туда-обратно.

Код находится в:

```text
arithmetic/mnt4_3limb/src/BigIntMNT.sol
```

Основная функция:

```text
BigIntMNT.montMul3
```

## 2. Почему выбран Montgomery/CIOS

В EVM нет нативной 753-битной арифметики и нет инструкции `mulhi`, которая возвращала бы старшую половину 512-битного произведения. Поэтому каждое 256x256 -> 512 умножение приходится восстанавливать через:

```text
lo = mul(x, y)
mm = mulmod(x, y, 2^256 - 1)
hi = mm - lo - carry
```

Montgomery-редукция позволяет заменить деление по модулю `p` на последовательность умножений на предвычисленную константу

```text
magic = -p^{-1} mod 2^256.
```

CIOS означает `Coarsely Integrated Operand Scanning`. Смысл схемы: не строить сначала полное 6-словное произведение `a*b`, а сразу совмещать накопление произведений и Montgomery-редукцию.

Для трех слов это особенно удобно:

```text
для i = 0,1,2:
    добавить a_i * b к аккумулятору
    выбрать m = t_0 * magic mod 2^256
    добавить m * p к аккумулятору
    отбросить младшее слово t_0
```

После каждого шага младшее слово становится нулевым по модулю `2^256`, поэтому его можно удалить сдвигом окна аккумулятора.

## 3. Псевдокод production-алгоритма

Ниже приведен псевдокод без Yul-деталей. Он описывает ровно ту логику, которая реализована в `montMul3`.

```text
Вход:
    A = (a0, a1, a2)  -- Montgomery-значение aR mod p
    B = (b0, b1, b2)  -- Montgomery-значение bR mod p

Константы:
    p = (p0, p1, p2)
    magic = -p0^{-1} mod 2^256

Вспомогательная операция:
    mul512(x, y):
        вернуть lo, hi такие что x*y = lo + 2^256*hi

Алгоритм:
    t = (0, 0, 0, 0)     -- 4-словный рабочий аккумулятор

    для i = 0..2:
        u = a_i

        -- 1. Добавить строку schoolbook-произведения u * B.
        t[0..3] += u * (b0, b1, b2)
        то есть:
            t += u*b0
            t += 2^256 * u*b1
            t += 2^512 * u*b2
        все переносы обрабатываются явно по словам

        -- 2. Выбрать Montgomery-коэффициент.
        m = t[0] * magic mod 2^256

        -- 3. Добавить m*p так, чтобы младшее слово стало нулевым.
        t[0..3] += m * (p0, p1, p2)

        -- 4. Отбросить младшее слово.
        -- Оно равно 0 mod 2^256 по построению.
        t = (t[1], t[2], t[3], 0)

    -- После трех шагов получен кандидат r < 2p.
    r = (t[0], t[1], t[2])

    если r >= p:
        r = r - p

    вернуть r
```

## 4. Почему алгоритм корректен

На каждом шаге выбирается:

```text
m = t_0 * (-p_0^{-1}) mod 2^256.
```

Тогда младшее слово суммы

```text
t + m*p
```

равно нулю по модулю `2^256`, потому что:

```text
t_0 + m*p_0
= t_0 - t_0*p_0^{-1}*p_0
= 0 mod 2^256.
```

Значит младшее слово можно отбросить. После трех таких шагов деление на `R = 2^768` выполнено без обычного деления:

```text
REDC(T) = T * R^{-1} mod p.
```

Так как входы уже находятся в Montgomery-представлении:

```text
T = (aR)*(bR) = abR^2,
REDC(T) = abR mod p.
```

То есть результат остается в том же представлении.

## 5. Стоимость в базовых операциях

Для одного `montMul3` неизбежны:

```text
9 произведений a_i*b_j
9 произведений m_i*p_j
```

Итого:

```text
18 широких произведений 256x256 -> 512.
```

В EVM одно такое широкое произведение не является одной инструкцией. Нужно вычислить младшую и старшую половины:

```text
lo = MUL(x, y)
mm = MULMOD(x, y, 2^256 - 1)
hi = mm - lo - carry
```

Кроме самих `mul`/`mulmod`, требуются:

- сложения слов;
- проверки переносов через `lt`;
- коррекции старших слов;
- сдвиг CIOS-окна;
- финальное условное вычитание `p`;
- `dup/swap`-операции из-за давления на стек.

Поэтому нижняя оценка только по “идеальным” арифметическим opcode слишком оптимистична. Она не учитывает, что EVM не умеет держать 753-битный accumulator как один регистр.

## 6. Измеренная стоимость

Для production-реализации получены значения:

| Операция | Gas/op |
|---|---:|
| `BigIntMNT.montMul3`, internal hot path | 2,959 |
| `BigIntMNT.montSqr3`, internal hot path | 2,947 |
| `BigIntMNT.montMul3`, external stack ABI | 3,316 |

Отдельная opcode-трасса для развернутого CIOS-тела показывает примерно:

```text
2,843 gas
```

Разница между `2,843` и `2,959` объясняется тем, что Foundry-бенчмарк измеряет не только чистое тело, но и небольшие расходы цикла, вызова и окружения.

## 7. Какие альтернативы проверялись

Были реализованы и измерены несколько вариантов:

| Подход | Результат |
|---|---:|
| Montgomery/CIOS | 2,959 gas/op |
| Barrett | около 46,998 gas/op |
| FIOS | около 18,509 gas/op |
| Comba/SOS | около 18,301 gas/op |
| Square-specialized Comba | около 18,420 gas/op |
| Независимый canonical schoolbook+REDC вариант | около 5,643 gas/op |

Вывод из сравнений: CIOS выигрывает, потому что не материализует полное 6-словное произведение и сразу удаляет младшие слова через Montgomery-редукцию. В EVM это важнее, чем теоретически более красивое разделение “сначала произведение, потом редукция”.

## 8. Почему дальнейшая оптимизация почти не остается

Существенное снижение больше чем на 10% не найдено по следующим причинам.

1. Число широких произведений уже минимально для обычного 3-limb Montgomery-умножения: `9` произведений входов и `9` произведений редукции.
2. CIOS уже совмещает умножение и редукцию, поэтому нет отдельного 6-словного массива `a*b`, который можно было бы убрать.
3. Код развернут вручную в Yul; нет циклов, динамических массивов и лишних memory-структур в hot path.
4. Альтернативы Barrett/FIOS/Comba были реализованы рядом и оказались существенно дороже.
5. Специализация квадрата через Comba дала небольшой выигрыш относительно Comba, но осталась намного хуже CIOS.
6. Главные оставшиеся расходы связаны не с выбранным алгоритмом, а с моделью EVM: отсутствием `mulhi`, явными переносами и ограниченным стеком.

Поэтому текущая реализация не является математической нижней границей для всех возможных виртуальных машин, но является лучшим найденным практическим вариантом для текущего EVM/Yul-контекста проекта.

## 9. Альтернатива через MODEXP

Отдельно проверялась идея считать умножение через тождество:

```text
4ab = (a+b)^2 - (a-b)^2.
```

Тогда можно получить:

```text
s1 = (a+b)^2 mod p
s2 = |a-b|^2 mod p
diff = (s1 - s2) mod p
ab = diff / 4 mod p
```

Квадраты можно считать через precompile `MODEXP` по адресу `0x05` с экспонентой `2`.

Измерения:

| Операция | Prague gas/op | Osaka gas/op |
|---|---:|---:|
| canonical умножение через два MODEXP, полноразмерные входы | 2,375 | 2,975 |
| canonical квадрат через один MODEXP, полноразмерные входы | 578 | 878 |

Квадрат действительно получается дешевым. Но общее умножение через два квадрата не дает выигрыша для production-пути:

- production `montMul3` стоит около `2,959 gas/op`;
- MODEXP-умножение после EIP-7883/Osaka стоит около `2,975 gas/op`;
- MODEXP-результат находится в каноническом представлении, а production-код работает в Montgomery-представлении;
- перевод между представлениями требует дополнительных Montgomery-умножений;
- если полностью перейти на canonical-представление, придется пересчитать все константы, prepared cache и заново измерить `Fq2/Fq4/Miller` hot path.

Поэтому MODEXP-квадрат полезен как исследовательская идея и потенциальный путь для отдельной canonical-ветки, но он не является drop-in заменой текущего `montMul3`.

## 10. Итог

Текущая 3-limb реализация:

```text
Montgomery representation + CIOS reduction + полностью развернутый Yul hot path
```

является лучшим подтвержденным вариантом среди реализованных и измеренных подходов. Существенно ускорить ее внутри той же архитектуры не удалось: все проверенные альтернативы либо требуют больше переносов и памяти, либо меняют представление данных и теряют выигрыш на редукции/конвертации.

Практический вывод для всей работы: дальнейшее снижение стоимости pairing нужно искать не в замене одной `Fq`-операции, а на более высоком уровне:

- уменьшать число `Fq`-умножений;
- использовать разреженные линии;
- объединять аккумуляторы;
- использовать prepared/code-shards cache;
- заменять финальную экспоненту relation-проверкой;
- исследовать другие семейства кривых и другие proof/commitment модели.
