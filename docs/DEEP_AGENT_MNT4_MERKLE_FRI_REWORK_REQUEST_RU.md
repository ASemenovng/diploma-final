## 3. Блокирующие математические замечания

### 3.1. Уточнить объект, который фиксируется в таблицах делителей

В предыдущем ответе используются выражения

\[
G_t^Q(P;Z), \qquad G_t^S(-R;Z).
\]

Они зависят не только от фиксированных точек \(Q,S\), но и от публичных
входов \(P,R\). Поэтому их нельзя без пояснения считать фиксированными
столбцами, корни которых заранее зашиты при развертывании verifier-а.

Нужно явно вывести:

1. какие коэффициенты зависят только от \(Q,S\) и ate loop;
2. какие значения зависят от \(P,R\);
3. что именно хранится в канонических fixed tables;
4. что именно раскрывается Merkle proof-ом;
5. какие операции verifier выполняет в sampled row для получения
   \(G_t^Q(P;\zeta)\) и \(G_t^S(-R;\zeta)\);
6. как эти значения связаны с sparse line format текущего прямого verifier-а.

Нужны явные формулы line evaluation для MNT4-753, а не абстрактная ссылка на
prepared divisors.

### 3.2. Исправить ошибку степени прямого блочного quotient по \(Z\)

В предыдущем ответе для прямого отношения

\[
F_{t+5}(Z) -
F_t(Z)^{32}G_t^Q(P;Z)G_t^S(-R;Z)c(Z)^{\kappa_t}
=H_t(Z)(Z^4-13)
\]

утверждается:

\[
\deg H_t\le 2.
\]

Это неверно для прямого произведения без промежуточных редукций: степень левой
части существенно выше шести. Граница \(\deg J_i\le 2\) применима только к
отдельным коротким отношениям после введения промежуточных приведенных
элементов.

Нужно:

- исправить формулировку;
- оставить только корректную цепочку коротких операций;
- вывести степень каждого локального quotient \(J_i(Z)\);
- вывести степень каждого полинома по трассовой переменной \(X\).

### 3.3. Полностью раскрыть граничные отношения

Выражение \(\mathcal N_{\mathrm{aux}}\) в предыдущем ответе оставлено
неопределенным. Это недопустимо.

Нужно явно записать полиномиальные отношения для:

- \(F_0=c^{-1}\);
- проверки \(c\cdot c^{-1}=1\);
- 75 полных блоков по пять шагов;
- одного остаточного шага;
- signed addition steps;
- tail-линий отрицательного ate parameter;
- Frobenius-tail;
- \(F_{\mathrm{final}}=1\);
- padding rows.

Для каждого отношения указать selector, домен, степень и способ проверки.

### 3.4. Исправить quotient по трассовой переменной

В предыдущем ответе используется:

\[
Q_{\mathrm{trace}}(X)=
\frac{\mathcal N_{\mathrm{mix}}(X)}{X^{128}-1}.
\]

Нужно доказать, что числитель действительно обращается в ноль на всех
128 точках трассового домена, включая padding rows. Если активные переходы
занимают только часть строк, нужно явно определить:

- active-row selector;
- transition selector;
- boundary selectors;
- padding policy;
- степень каждого selector;
- влияние selectors на степень composition polynomial.

Если деление на \(X^{128}-1\) не является оптимальным или корректным вариантом,
выбрать другой vanishing polynomial и вывести все степени заново.

## 4. Блокирующие замечания по FRI

### 4.1. Нельзя использовать DEEP-FRI soundness без DEEP-FRI-протокола

Предыдущий ответ использует улучшенную оценку DEEP-FRI, но фактически описывает
обычный FRI folding. Это некорректно.

DEEP-FRI требует отдельной техники Domain Extending for Eliminating
Pretenders: verifier выбирает точку вне evaluation domain и получает
дополнительное значение интерполирующего полинома. Если используется
DEEP-FRI, нужно полностью описать и реализовать его логику:

- outside-domain challenge;
- дополнительные commitments и openings;
- обновленный transcript;
- acceptance predicate;
- точную теорему soundness;
- стоимость calldata и выполнения.

Если реализуется обычный FRI, нельзя использовать DEEP-FRI-оценку.

Исправить ссылку на источник. Авторы DEEP-FRI:

- Eli Ben-Sasson;
- Lior Goldberg;
- Swastik Kopparty;
- Shubhangi Saraf.

Источник:

https://drops.dagstuhl.de/entities/document/10.4230/LIPIcs.ITCS.2020.5

### 4.2. Исправить расписание сверток

В предыдущем ответе одновременно заявлено:

- начальная степень \(<254\);
- семь FRI-раундов;
- финальный полином степени \(<16\).

После семи бинарных сверток начальная граница степени \(<254\) уменьшается
примерно до \(<2\), а не до \(<16\). Если требуется финальный полином степени
\(<16\), нужно остановиться раньше.

Нужно выбрать одно точное расписание и для каждого слоя привести:

| Слой | Размер домена | Граница степени | Root | Challenge | Открытия |
|---|---:|---:|---|---|---|

### 4.3. Исправить batching столбцов с разными degree bounds

В предыдущем ответе один полином

\[
B(X)=Q_{\mathrm{trace}}(X)+\sum_j \eta^j C_j(X)
\]

используется для доказательства малой степени всех столбцов. Но:

- trace-столбцы должны иметь степень \(<128\);
- quotient может иметь степень до \(253\).

Если FRI доказывает только \(\deg B<254\), это не доказывает
\(\deg C_j<128\) для trace-столбцов. Тогда исходная оценка степени composition
polynomial перестает быть обоснованной.

Нужно выбрать и доказать корректный способ:

- отдельные FRI batches для разных degree classes;
- degree-adjustment множители;
- random linear combination после выравнивания степеней;
- иной корректный вариант.

Нужно пересчитать proof size и gas.

### 4.4. Исправить оценку quotient soundness

В предыдущем ответе для проверки

\[
\mathcal N_{\mathrm{mix}}(x)
=
Q_{\mathrm{trace}}(x)(x^{128}-1)
\]

используется оценка:

\[
\Pr[\text{ошибка}]
\le
\frac{381}{|\mathbb F_q|}.
\]

Но \(x\) выбирается из LDE-домена размера 2048, а не равномерно из всего поля.
Эта оценка неприменима.

Нужно вывести корректный ALI/STARK soundness-анализ:

- как proximity к Reed-Solomon code сочетается с локальной проверкой
  constraints;
- какова минимальная доля плохих sampled rows при ложном утверждении;
- сколько запросов требуется;
- как учитываются Merkle commitments и FRI proximity;
- какова точная итоговая ошибка.

### 4.5. Необходимо выбрать один конкретный протокол

В предыдущем ответе смешаны:

- обычный FRI;
- DEEP-FRI;
- STARK-подобный AIR;
- идеи IOP из Section 6 статьи *On Proving Pairings*.

Нужно сравнить варианты, но итоговая спецификация должна выбрать ровно один
production-протокол. Для него привести полную последовательность сообщений
до Fiat--Shamir transform и после него.

## 6. Блокирующие замечания по Merkle PCS

### 6.1. Нужны точные leaf layouts

Недостаточно написать, что используется row-major layout.

Для каждого дерева привести таблицу:

| Tree | Leaf index | Поля листа | Размер листа | Encoding | Padding |
|---|---:|---|---:|---|---|

### 6.2. Нужен точный multiproof format

Нужно описать:

- сортировку индексов;
- дедупликацию;
- порядок siblings;
- восстановление root;
- защиту от duplicate indices;
- защиту от ambiguous encodings;
- worst-case и fixture-specific число siblings.

### 6.3. Fixed roots должны иметь строгую модель доверия

Конструктор, принимающий произвольные roots, сам по себе не гарантирует, что
таблицы соответствуют каноническим точкам \(Q,S\).

Нужно выбрать один вариант:

1. roots зашиты в bytecode;
2. verifier развертывается фабрикой с проверяемой процедурой;
3. существует one-time proof корректности таблиц;
4. явно фиксируется предположение доверия к конкретному deployment.

Для дипломной production-модели предпочтительно использовать канонические
roots в bytecode либо проверяемую фабрику.

## 8. Блокирующие замечания по gas-модели

Предыдущая оценка \(\approx 28\)M gas является предварительной гипотезой, а не
доказанным результатом.

Проблемы:

1. число Merkle hash steps указано приблизительно;
2. размер multiproof указан приблизительно;
3. стоимость хеширования крупных row-major leaves не посчитана точно;
4. стоимость деления в FRI fold оценена эвристически;
5. не выбрана оптимизация: заранее фиксированные inverses domain points,
   batch inversion либо иной вариант;
6. стоимость canonical checks для 753-битных элементов не посчитана;
7. стоимость curve membership не выведена;
8. не учтен исправленный batching по degree classes;
9. не учтена возможная DEEP-FRI логика;
10. calldata посчитана по приблизительному proof size.

Нужно:

- дать точное число операций;
- использовать реальные измерения существующей 3-limb EVM-арифметики;
- отдельно посчитать calldata worst-case и fixture-specific;
- предоставить Rust-скрипт, который строит proof layout и печатает размер;
- предоставить таблицу по профилям \(d=1,4,5,6\);
- честно указать, если результат превышает прямые \(93\,879\,746\) gas.
