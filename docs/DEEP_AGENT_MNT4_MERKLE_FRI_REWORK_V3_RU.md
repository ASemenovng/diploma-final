# Третья доработка формализации MNT4 Merkle/DEEP-FRI verifier-а

## 1. Контекст и итог аудита версии 3.0

Проверен документ `MNT4_MERKLE_FRI_FULL_FORMALIZATION (2).md`.

Версия 3.0 исправляет часть замечаний к предыдущему исследованию:

- quotient после композиции теперь корректно рассматривается как скалярный
  полином над `Fq`;
- вместо небезопасного вычисления quotient-по-`Z` самим verifier-ом выбраны
  координатные формулы арифметики `Fq4`;
- добавлены openings в точках `x` и `-x`;
- явно введены OOD evaluations;
- сформулирован неинтерактивный Fiat--Shamir transcript;
- добавлено отдельное обсуждение множителей `c^kappa`.

Однако документ все еще нельзя использовать как спецификацию для реализации.
Главный риск не косметический: текущий AIR не доказывает реальную рекурсию
цикла Миллера MNT4-753 в том формате, в котором ее вычисляют `arkworks` и
чистовая Solidity/Yul-реализация проекта.

Исправленный ответ должен быть самостоятельной версией 4.0, полностью
заменяющей версию 3.0. В нем не должно остаться формулировок вида
`приблизительно`, `упрощенно`, `можно выбрать`, если от выбора зависит
корректность, формат proof или gas.

## 2. Блокирующая ошибка: AIR блока не доказывает все умножения Миллера

В версии 3.0 правильно записано:

\[
F_{t+5}
=
F_t^{32}
\cdot
\prod_{h=0}^{4} M_{5t+h}^{2^{4-h}}.
\]

Но в трассе присутствуют только:

- `f`: 4 координаты;
- `U1..U5`: 20 координат для пяти возведений в квадрат;
- `VQ`, `VS`: 8 координат;
- итого 32 witness-столбца.

После этого документ проверяет:

\[
V_Q=U_5\cdot G_t^Q(P),
\qquad
V_S=V_Q\cdot G_t^S(-R).
\]

Проблема: `G_t^Q(P)` и `G_t^S(-R)` являются произведениями линий и их
степеней. Они не являются одной подготовленной линией и не могут быть
получены verifier-ом из трех коэффициентов без воспроизведения всех
внутренних операций блока.

Нужно выбрать один корректный вариант и довести его до конца.

### Вариант A. Пошаговая трасса внутри блока

Для каждого из пяти шагов блока добавить committed intermediate states:

1. возведение аккумулятора в квадрат;
2. умножение на doubling line для `Q`;
3. умножение на doubling line для `S`;
4. условное умножение на addition line для `Q`;
5. условное умножение на addition line для `S`;
6. умножение на соответствующий множитель `c`, `c^{-1}` либо `1`.

Для каждого фактически выполняемого умножения записать четыре координатных
constraints над `Fq`.

### Вариант B. Корректный агрегированный делитель блока

Можно коммитить агрегированные блочные множители, но тогда необходимо:

1. явно определить их как witness-полиномы;
2. связать их с per-step prepared lines отдельными constraints;
3. включить эти witness-полиномы в trace commitment;
4. включить openings и OOD evaluations в `proof.bin`;
5. пересчитать degree bounds, proof size и gas.

Простое объявление `G_t^Q(P)` фиксированным столбцом недопустимо: это значение
зависит от публичной переменной точки `P`.

Нужно сравнить оба варианта и выбрать тот, который дает корректный и более
дешевый production-кандидат.

## 3. Использован неверный формат prepared-линий MNT4

Версия 3.0 использует обобщенную Jacobian-линию:

```text
PreparedLine = (c0, c1, c2),  c_i in Fq2
```

и далее местами считает одну линию равной трем элементам `Fq`.

Это не соответствует канонической MNT4-реализации.

В `arkworks` для flipped ate Miller loop используются:

```rust
AteDoubleCoefficients {
    c_h:  Fp2,
    c_4c: Fp2,
    c_j:  Fp2,
    c_l:  Fp2,
}

AteAdditionCoefficients {
    c_l1: Fp2,
    c_rz: Fp2,
}
```

Оценка doubling line в точке `P`:

\[
g_{RR}(P)=
\operatorname{Fp4}\left(
-c_{4c}-c_j x_{P,\mathrm{twist}}+c_l,\;
c_h y_{P,\mathrm{twist}}
\right).
\]

Оценка addition line:

\[
g_{RQ}(P)=
\operatorname{Fp4}\left(
c_{rz}y_{P,\mathrm{twist}},\;
-
\left(
y_{Q/\mathrm{twist}}c_{rz}
+
(x_{P,\mathrm{twist}}-x_{Q/\mathrm{twist}})c_{l1}
\right)
\right).
\]

Чистовая Solidity/Yul-реализация проекта сериализует оптимизированный sparse
формат:

- doubling step: `3 * Fq2 = 576` байт;
- addition step: `2 * Fq2 = 384` байт.

Нужно:

1. вывести точное соответствие между коэффициентами `arkworks` и sparse
   форматом чистовой Solidity/Yul-реализации;
2. использовать именно эти формулы в AIR;
3. указать, какие коэффициенты сохраняются, а какие восстанавливаются;
4. пересчитать fixed columns, leaf layouts, proof size и gas;
5. дать Rust-псевдокод генератора таблицы из `ark-mnt4-753`.

Нельзя продолжать с приблизительной Jacobian slope-моделью.

## 4. Противоречие в fixed columns и `root_fixed`

В разделе про AIR заявлены 65 fixed columns: per-step prepared coefficients
для пяти шагов блока. Но в Merkle leaf `root_fixed` сохраняются уже другие
данные:

```text
GQ_0..GQ_3
GS_0..GS_3
selectors
kappa
```

Это разные протоколы. Кроме того, `GQ(P)` и `GS(-R)` зависят от публичных
точек `P`, `R`, поэтому не могут входить в предвычисленный `root_fixed`.

Нужно полностью определить один fixed oracle:

1. точный перечень столбцов;
2. точный порядок сериализации;
3. построение trace-domain и LDE-domain таблиц;
4. leaf layout;
5. Merkle root;
6. способ открытия в query-точках;
7. способ получения значений fixed-полиномов в OOD-точке `z`;
8. способ хранения канонических fixed-данных в EVM.

Merkle opening в точке домена не подтверждает значение fixed-полинома в
OOD-точке. Это нужно связать отдельным строго описанным механизмом.

## 5. Хранение fixed-таблицы в одном контракте невозможно

Версия 3.0 предлагает хранить около `85 KB` fixed-данных в bytecode verifier-а.
Для Ethereum mainnet это невозможно в одном контракте: EIP-170 ограничивает
runtime-код значением `0x6000 = 24,576` байт.

Нужно выбрать реалистичную архитектуру:

- несколько data-contract shards с чтением через `EXTCODECOPY`;
- либо иной проверяемый механизм хранения.

Для выбранного варианта указать:

- число shards;
- байты на shard;
- адресную привязку;
- порядок чтения;
- стоимость deployment;
- runtime gas;
- почему пользователь не может подменить fixed cache.

## 6. Transcript FRI пока циклический

Версия 3.0 определяет `fri_root[0]` как commitment слоя после первого fold, но
в transcript сначала поглощает `fri_root[0]`, а затем получает challenge
`rho_0`:

```text
T <- hash(T || fri_root[0])
rho_0 <- challenge(T)
```

Если слой `fri_root[0]` уже зависит от `rho_0`, возникает цикл.

Нужно дать точную последовательность сообщений. Допустимы два корректных
варианта:

### Вариант A. Коммитить исходный DEEP-слой

```text
commit root_D
derive rho_0
commit root_1
derive rho_1
...
```

### Вариант B. Не коммитить исходный DEEP-слой

Verifier вычисляет `D(x)` и `D(-x)` из trace/quotient openings, получает
`rho_0`, затем проверяет вычисленное значение первого folded layer против
`root_1`.

Для каждого раунда указать:

- какой root передается;
- когда он передается;
- какой challenge выводится после root;
- какие значения открываются;
- какой Merkle path проверяется;
- как вычисляется индекс следующего слоя.

## 7. FRI openings описаны неполно

В текущем `proof.bin` для каждого FRI-слоя передаются sibling и auth path, но
не зафиксировано, где передается и как аутентифицируется значение текущего
folded layer.

Нужно описать query proof покомпонентно:

1. openings trace/quotient/fixed в `x` и `-x`;
2. вычисление `D(x)` и `D(-x)`;
3. вычисление folded value первого слоя;
4. opening этого value против первого FRI root;
5. sibling следующего fold;
6. переход индекса;
7. повторение до final polynomial;
8. final polynomial coefficients и проверка степени.

Для каждой Merkle-аутентификации указать точный leaf index и точный список
siblings. Нельзя оставлять выбор между `xor` и сложением.

## 8. Свидетельство `c^kappa` не включено в реальную трассу

В приложении версии 3.0 сказано, что prover коммитит `c^{kappa_t}` per row и
проверяет addition chain. Но:

- этих столбцов нет среди 32 witness columns;
- их нет в trace leaf;
- их нет в OOD evaluations;
- их нет в proof size;
- множество встречающихся `kappa_t` не вычислено;
- chain constraints не включены в composition polynomial.

Нужно:

1. вычислить точные signed digits `ATE_LOOP_COUNT`;
2. перечислить все `kappa_t`;
3. перечислить множество `K`;
4. определить witness-столбцы степеней `c`;
5. записать точные coordinate constraints addition chain;
6. добавить их в composition polynomial;
7. обновить degree bound;
8. обновить leaf layouts, proof size, gas.

## 9. Challenge generation повреждает полноту

`fieldChallenge` с максимумом 8 попыток дает вероятность отказа честного
proof порядка `2^-21` для всего протокола. Для production verifier-а это
слишком много.

Для MNT4 `q` имеет 753 бита. Поэтому предпочтительно использовать
256-битный вывод Keccak непосредственно как элемент подмножества `Fq`:

\[
\{0,\ldots,2^{256}-1\}\subset \mathbb F_q.
\]

Это исключает rejection sampling и сохраняет достаточную энтропию challenge.
Если выбирается другой способ, нужно доказать:

- отсутствие заметной ошибки полноты;
- отсутствие bias, опасного для soundness;
- точный Solidity-псевдокод.

Для query indices также нужен алгоритм выборки без зависания и без
необработанного worst-case cap.

## 10. Soundness-анализ DEEP-FRI нельзя подставлять эвристически

Версия 3.0 использует:

\[
\varepsilon_{\mathrm{query}}
\le
(2\rho)^R,
\qquad
\varepsilon_{\mathrm{FRI}}
\le
\varepsilon_{\mathrm{query}}^\lambda.
\]

Эта формула не выведена из теоремы DEEP-FRI для выбранных параметров и
конкретного протокола. Также неверно предлагать считать Keccak-256 имеющим
`256-bit collision resistance`: стандартная generic collision security
256-битного hash составляет около 128 бит.

Нужно:

1. указать точную теорему DEEP-FRI и ее условия;
2. проверить применимость к выбранному code rate, числу fold-rounds и final
   polynomial degree;
3. вывести ошибку proximity test без пропусков;
4. отдельно учесть algebraic batching, OOD check, Fiat--Shamir в ROM и Merkle
   binding;
5. сформулировать итоговый уровень безопасности аккуратно: например,
   `не менее примерно 128 бит вычислительной безопасности при стандартном
   предположении о Keccak-256`, если строгая сумма вероятностей немного выше
   `2^-128`.

## 11. Gas-модель `17.1M` пока не воспроизводима

Оценка газа зависит от неверного формата prepared lines и неполной трассы.
Дополнительно:

- OOD fixed evaluation сначала оценивается в `300K`, затем в `50K`;
- стоимость инверсии `Fq` равна `10K` без benchmark;
- стоимость DEEP evaluation сначала выводится как `9.1M`, затем заменяется на
  `8.0M` через неформализованную оптимизацию;
- код Merkle path не учитывает точный proof parser, memory expansion и
  multiproof-варианты;
- стоимость shards не включена;
- addition-chain для `c^kappa` не включена полностью.

Нужно представить три таблицы:

1. **Точная operation model**: количество `Fq.add`, `Fq.sub`, `Fq.mul`,
   `Fq.sqr`, `Fq.inv`, `keccak`, `calldataload`, `EXTCODECOPY`, memory-copy.
2. **Benchmark primitive costs**: результаты Foundry для каждой примитивной
   операции на чистовых библиотеках проекта.
3. **Derived gas model**: произведение количества операций на измеренную
   стоимость плюс calldata и memory.

До реализации допустимо дать расчетную модель, но каждое число должно быть
получено формулой. Нельзя объявлять `17.1M` итоговым gas.

## 12. Тестовый fixture должен строиться без решения дискретного логарифма

Псевдокод:

```rust
let r = compute_r_for_valid_pairing(p, q, s);
```

не является допустимой спецификацией: в общем случае подбор `R` может
потребовать решения задачи дискретного логарифмирования.

Нужно определить воспроизводимые fixture-ы:

- простой положительный fixture: `Q = S`, `P = R`;
- более содержательный fixture по билинейности:
  `Q = [a]G2`, `S = [b]G2`, `P = [b]G1`, `R = [a]G1`, тогда
  `e(P,Q) = e(R,S)`;
- отрицательные fixture-ы: подмена одной координаты, одной trace leaf,
  одного quotient opening, одного fixed opening, одного FRI sibling, одного
  OOD evaluation, одного transcript root.

## 13. Обязательные артефакты версии 4.0

Исправленный документ должен содержать:

1. точную residue-рекурсию MNT4-753;
2. точное отображение `arkworks -> packed sparse Solidity/Yul format`;
3. выбранную модель AIR с полным списком witness и fixed columns;
4. полный composition polynomial;
5. degree bounds;
6. точную модель `root_fixed`, включая OOD evaluation;
7. точный Fiat--Shamir transcript без циклов;
8. точный Merkle/FRI query proof;
9. точный `proof.bin` layout;
10. строгий soundness-анализ с корректной ссылкой на теорему;
11. воспроизводимую gas-модель;
12. Rust-модули и Solidity-модули будущей реализации;
13. полный перечень тестов;
14. список параметров и констант, которые генерирует Rust backend.

## 14. Критерии допуска к реализации

Реализацию можно начинать только если:

- AIR действительно проверяет каждый необходимый переход цикла Миллера;
- prepared sparse format совпадает с `ark-mnt4-753` и чистовой
  Solidity/Yul-реализацией;
- fixed oracle не содержит значений, зависящих от `P` или `R`;
- OOD fixed evaluations криптографически связаны с canonical fixed data;
- FRI transcript не имеет циклических зависимостей;
- для каждого FRI fold описан полный opening proof;
- `c^kappa` включен в trace, constraints и proof layout;
- fixture-ы не требуют решения дискретного логарифма;
- gas-модель является выводимой из operation counts и benchmark-ов;
- все размеры bytecode совместимы с EIP-170 либо разложены по code-shards.
