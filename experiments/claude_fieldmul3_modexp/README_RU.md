# Эксперимент: 3-limb умножение через MODEXP

## Идея

Для нечетного модуля `p` выполняется тождество:

```text
(a + b)^2 - (a - b)^2 = 4 * a * b.
```

Поэтому канонический результат `a * b mod p` можно получить так:

```text
s1 = (a + b)^2 mod p
s2 = |a - b|^2 mod p
diff = (s1 - s2) mod p
r = halve(halve(diff))
```

где модульное полуделение определено формулой:

```text
halve(x) = x / 2,       если x четное;
halve(x) = (x + p) / 2, если x нечетное.
```

Два квадрата вычисляются системным контрактом `MODEXP` по адресу `0x05`.

## EIP-7883 и Osaka

До обновления Fusaka минимальная стоимость одного вызова `MODEXP` составляла
`200 gas`. EIP-7883 увеличил минимум до `500 gas` и изменил общую формулу
тарификации. В терминах инструментов Foundry новый набор правил слоя исполнения
выбирается как hardfork `osaka`.

Для 96-байтных базы и модуля и экспоненты `2` расчетная стоимость ниже минимума.
Поэтому один квадрат стоит:

| Правила EVM | MODEXP gas |
|---|---:|
| Prague, до EIP-7883 | 200 |
| Osaka, после EIP-7883 | 500 |

## Проверка корректности

Эталон находится в `scripts/ref_mul.py` и использует целые числа Python
произвольной точности. Проверены:

- нулевые значения;
- умножение на единицу;
- `(p - 1)^2`;
- полноразмерные слова;
- `64` быстрых случайных вектора;
- `5000` детерминированных случайных векторов;
- отдельный полноразмерный квадрат.

Команды:

```bash
forge test --evm-version prague --match-test 'test_(Zero|One|PMinus|High|Square|Random64)' -vv
forge test --evm-version prague --match-test test_Random5000AgainstPython -vv
```

## Ограничение прямого сравнения

Новая функция и production-функция имеют разную семантику:

```text
FieldMul3Modexp.mulMod3(a, b) = a * b mod p
BigIntMNT.montMul3(aR, bR)    = a * b * R mod p
```

Поэтому нельзя заменить вызов `montMul3` вызовом `mulMod3`: результат будет
находиться в другом представлении.

## Вывод по возможной интеграции

После EIP-7883 полноразмерное общее умножение через два вызова MODEXP стоит
`2 975 gas/op`. Это практически равно текущему Montgomery-умножению
`2 959 gas/op` и не дает самостоятельного выигрыша.

Квадрат через один MODEXP стоит `878 gas/op`, то есть примерно в `3.36` раза
дешевле текущего `montSqr3`. Поэтому имеет смысл исследовать отдельную
canonical-ветку арифметики:

1. хранить элементы поля в каноническом представлении вместо Montgomery;
2. использовать `mulMod3` для общих умножений;
3. использовать `sqrMod3` для квадратов;
4. перевести вызывающие функции с `pure` на `view`;
5. пересчитать константы и prepared line cache;
6. измерить `Fq2`, `Fq4` и полный цикл Миллера.

Такая миграция является отдельным экспериментом. Она не должна заменять
production-код до полного сравнительного прогона.
