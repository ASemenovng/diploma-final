# Дополнительное задание для deep agent: Ehat/Fq2 Tate/Ate verifier для lollipop-305

## 1. Контекст задачи

В проекте исследуется возможность построить более дешевую альтернативу MNT4-753/MNT6-753 для pairing-based proof systems на основе lollipop-конструкций из ePrint 2024/1627. Для MNT4/MNT6 уже реализованы и измерены:

- оптимизированная Solidity/Yul арифметика;
- on-chain вычисление сопряжения;
- prepared-line режим;
- verifier по мотивам ePrint 2024/640 с заменой финальной экспоненты проверкой вида `c^r = F`.

Для направления lollipop-305 сейчас есть три части:

1. `E_stick/Fp`, где целевое поле `Fp4`.
2. `E_cycle/Fp2`, где целевое поле `Fp4`.
3. `Ehat/Fq2`, где целевое поле `Fq6`.

Первые две части уже имеют article640-style verifier в проекте. Третья часть (`Ehat/Fq2`) сейчас реализована только как дорогой fallback через Weil-ratio:

```text
e_p(P,Q) = f_{p,P}(Q) / f_{p,Q}(P)
```

Это математически допустимо как fallback, но не является целевым вариантом. Целевой вариант должен быть ближе к MNT4/MNT6:

```text
prepared lines + Miller loop + residue/relation check
```

Задача агента — строго завершить спецификацию для `Ehat/Fq2`, чтобы ее можно было реализовать в Rust и Solidity/Yul без дополнительных математических догадок.

## 2. Что уже было предложено агентом

В предыдущем исследовании было предложено:

- кривая:

```text
Ehat / Fq2: y^2 = x^3 + (eta + 2), eta^2 = -2
```

- поле:

```text
Fq6 = Fq2[w] / (w^3 - rho)
```

- степень вложения для `p`-подгруппы над `Fq2`:

```text
k = ord_p(q^2) = 3
```

- потенциальный optimal Ate параметр:

```text
s = x - 1
```

- идея для отсутствующего denominator elimination:

```text
F = F_num / F_den,
c^p * F_den = F_num.
```

Это направление выглядит перспективным, но текущий документ пока нельзя использовать как production-ready спецификацию.

## 3. Обнаруженные проблемы в текущем исследовании

### 3.1. Параметр `s = x - 1` требует строгого доказательства

В документе утверждается, что для `Ehat/Fq2` можно использовать optimal Ate loop parameter:

```text
s = x - 1.
```

Обоснование дано через сравнение:

```text
q^2 ≡ x - 1 (mod p).
```

Но для реализации этого недостаточно. Нужно строго доказать, что Miller-функция

```text
f_{s,Q'}(P)
```

после финальной экспоненты действительно задает невырожденное сопряжение, эквивалентное reduced Tate pairing с точностью до ненулевой степени в `mu_p`.

Нужно вывести это из теории Ate/optimal Ate pairings:

- указать базовое поле именно `Fq2`;
- указать порядок подгруппы `p`;
- указать действие Фробениуса на `G2`;
- показать, что выбранный `s` соответствует корректному собственному значению Фробениуса;
- доказать, что получаемая степень не равна нулю по модулю `p`.

Формулировка вида "`s ≡ q^2 mod p`, значит можно" недостаточна.

### 3.2. Псевдокод Miller loop содержит ошибку addition-step

В текущем документе `miller_loop_prepared(P, lines)` написан как цикл:

```text
for each line in lines:
    F_num = F_num^2
    F_den = F_den^2
    F_num = F_num * line_num(P)
    F_den = F_den * line_den(P)
```

Но это неверно для addition-step.

В алгоритме Миллера возведение аккумулятора в квадрат выполняется только на doubling-step. Addition-step добавляет только умножение на дополнительную линию:

```text
doubling:
    f <- f^2 * l_dbl / v_dbl

addition:
    f <- f * l_add / v_add
```

Следовательно, prepared-line формат должен явно различать тип шага:

```text
op = DOUBLE | ADD
```

и verifier должен выполнять разные переходы.

### 3.3. Неверная формула для вычисления residue witness `c`

В документе предложено:

```rust
e = mod_inverse(p, q6 - 1)
c = F^e
```

Это невозможно, потому что:

```text
p | (q^6 - 1).
```

Следовательно, `p` не обратим по модулю `q^6 - 1`.

Корректная логика такая. Пусть:

```text
n = q^6 - 1,
m = n / p.
```

Если `F` лежит в образе отображения `z -> z^p`, то `F^m = 1`. Тогда можно использовать:

```text
u = p^{-1} mod m,
c = F^u,
```

потому что:

```text
(F^u)^p = F^{up} = F^{1 + km} = F.
```

Однако это работает только при условии, что `F^m = 1`. Нужно явно доказать, когда это условие выполнено для pairing-equation случая.

### 3.4. Не хватает точной рекуррентной формулы для `F_num` и `F_den`

В документе дана идея:

```text
F = F_num / F_den,
c^p * F_den = F_num.
```

Но для реализации нужны точные рекуррентные формулы по всем шагам Miller loop.

Нужно отдельно описать:

```text
doubling:
    F_num <- F_num^2 * l_dbl(P)
    F_den <- F_den^2 * v_dbl(P)

addition:
    F_num <- F_num * l_add(P)
    F_den <- F_den * v_add(P)
```

Если используется signed NAF, нужно также описать:

```text
addition with +Q
addition with -Q
```

и дать формулы для хорды в обоих случаях.

### 3.5. Не хватает проверяемых тест-векторов

В документе есть Sage-подобный код, но для реализации недостаточно:

- нет конкретного `P`;
- нет конкретного `Q0`;
- нет `Q' = psi(Q0)`;
- нет prepared-line массива;
- нет `F_num`;
- нет `F_den`;
- нет `c`;
- нет проверки положительного и отрицательных случаев.

Нужны воспроизводимые тест-векторы, желательно с Rust-реализацией, а не только Sage/Python.

## 4. Что нужно исследовать и выдать

### 4.1. Строгая математическая спецификация

Нужно подготовить финальную спецификацию `Ehat/Fq2` verifier:

1. Параметры:

```text
x, p, q,
Fq2,
Fq6,
rho,
Ehat,
G1,
G2,
GT.
```

2. Distortion map:

```text
psi(x,y) = (w^2 * x^q, rho * y^q)
```

с доказательством, что:

```text
psi(G1) subset G2,
G1 and G2 are paired non-degenerately.
```

3. Корректный pairing:

```text
a(Q',P) = f_{s,Q'}(P)^((q^6 - 1)/p)
```

или другой корректный вариант, если `s = x - 1` окажется неверным.

4. Полное доказательство корректности `s`.

Если `s = x - 1` неверен или недостаточен, нужно указать правильный параметр, например `p`, `q-1`, `q^2-1`, или другой параметр из optimal Ate decomposition.

### 4.2. Точные line formulas

Для точки:

```text
T = (w^2 c, d), c,d in Fq2
```

нужно дать формулы:

- doubling;
- addition with `Q'`;
- addition with `-Q'`, если используется signed representation;
- numerator line value at `P in Fq2`;
- denominator/vertical line value at `P in Fq2`.

Нужный формат:

```text
l(P) = A + B*w + C*w^2
v(P) = A' + B'*w + C'*w^2
```

с указанием, какие коэффициенты равны нулю и почему.

### 4.3. Точный prepared-line формат

Нужно определить сериализуемый формат:

```text
struct EhatPreparedLine {
    uint8 op;       // DOUBLE или ADD
    Fq2 L;
    Fq2 nu;
    Fq2 c_vert;
}
```

или предложить более экономичный формат.

Нужно указать:

- сколько line records получается;
- сколько `Fq` элементов в одном record;
- общий размер calldata/blob/code-shards;
- какие поля можно не хранить и восстановить дешевле on-chain.

### 4.4. Точная residue relation

Нужно строго вывести проверку:

```text
c^p * F_den = F_num.
```

Для pairing-equation из двух сопряжений:

```text
a(Q1', P1) * a(Q2', P2) = 1
```

нужно указать:

```text
F_num = F_num_1 * F_num_2
F_den = F_den_1 * F_den_2
```

и доказать:

```text
FinalExp(F_num / F_den) = 1
<=> exists c: c^p * F_den = F_num.
```

Также нужно отдельно указать, как обрабатывается `F_den = 0`. Для корректного сопряжения вычисления в таких точках должны исключаться выбором входов или обработкой особых случаев.

### 4.5. Исполняемые тест-векторы

Нужен быстрый reference-код, предпочтительно Rust.

Минимальный набор:

1. Сгенерировать `P in G1`.
2. Сгенерировать `Q0 in G1`.
3. Вычислить `Q' = psi(Q0)`.
4. Построить prepared lines для loop parameter `s`.
5. Вычислить `F_num`, `F_den`.
6. Вычислить `c`, такое что:

```text
c^p * F_den = F_num.
```

7. Проверить равенство.
8. Испортить:

- одну line;
- `P`;
- `Q0`;
- `c`;
- `F_den`;

и показать, что проверка падает.

### 4.6. Оценка стоимости

Нужно дать оценку в терминах операций:

- сколько doubling steps;
- сколько addition steps;
- сколько `Fq6.square`;
- сколько dense/sparse `Fq6.mul`;
- сколько `Fq2.mul`;
- сколько `Fq6.mul` нужно для `c^p`.

Оценка должна быть связана с текущими измерениями проекта:

```text
Fq.mul 2-limb после F2: ~1018 gas/op
Fq.square 2-limb после F2: ~940 gas/op
Fp2.mul stack: ~4171 gas/op
Fp4.mul full-stack: ~14675 gas/op
```

Для `Fq6` нужно либо вывести ожидаемую стоимость из `Fq2`, либо предложить отдельный micro-benchmark.

## 6. Важное ограничение

Не нужно предлагать Weil-ratio как финальное решение. Weil-ratio уже реализован как fallback и стоит около `200M gas` только для `Ehat/Fq2`. Цель исследования — понять, можно ли заменить его на prepared Tate/Ate + residue verifier аналогично MNT4/MNT6.

Если окажется, что для `Ehat/Fq2` denominator elimination невозможен, это не блокирует решение автоматически. Но нужно строго показать, что раздельное накопление:

```text
F_num, F_den
```

и проверка:

```text
c^p * F_den = F_num
```

полностью сохраняют корректность и soundness.
