# L12. Безопасная привязка кэша линий lollipop-305

## 1. Задача

Предыдущий исследовательский контракт принимал подготовленные линии как обычный
blob. Этого достаточно для измерения арифметики, но недостаточно для безопасного
fixed-cache режима: пользователь мог заменить множители цикла Миллера.

Добавлены два режима, которые связывают вычисление с заранее зарегистрированным
кэшем:

1. `calldata + commitment`;
2. `fixed code-shards + EXTCODECOPY`.

Оба режима используют одно и то же арифметическое ядро. Различается только способ
доставки и проверки подготовленных линий.

## 2. Что именно фиксируется

| Часть lollipop pipeline | Размер blob |
|---|---:|
| stick | `57,312` байт |
| `E_cycle` | `115,488` байт |
| `Ehat` prepared-Ate | `99,840` байт |

### 2.1. Calldata + commitment

В конструкторе задается неизменяемый commitment:

```text
C = keccak256(domain || blob.length || keccak256(blob)).
```

При каждом вызове пользователь передает blob. Контракт сначала пересчитывает
commitment и только затем выполняет цикл Миллера. Разделение доменов исключает
подстановку blob одной части lollipop pipeline вместо другой.

Контракт сравнения:

```text
research_variants/Lollipop305CommittedCacheVerifier.sol
```

### 2.2. Fixed code-shards

Крупный blob разбивается на части и хранится в runtime-коде отдельных
data-контрактов. Адреса этих контрактов задаются конструктору один раз. При
проверке пользователь не передает ни blob, ни адреса shards. Verifier читает
runtime-код по зафиксированным адресам инструкцией:

```text
EXTCODECOPY(address, destination, offset, size).
```

Повторный `keccak256` не нужен: пользователь не может выбрать другой источник
данных во время вызова. Новый кэш требует нового развертывания verifier-а.

Основной контракт:

```text
src/Lollipop305FixedShardsVerifier.sol
```

## 3. Граница безопасности

Оба режима решают задачу неизменности зарегистрированного кэша. Они не доказывают
on-chain, что blob был математически выведен из произвольной точки `Q`. Для
fixed-cache постановки кэш проверяется до развертывания, после чего контракт
гарантирует, что пользователь не заменит его в отдельном вызове.

Для произвольной меняющейся `Q` нужен другой слой: proof корректности кэша или
прямой пересчет линий.

## 5. Gas-сравнение

### 5.1. Execution gas

| Часть | Calldata + commitment | Fixed code-shards | Разница |
|---|---:|---:|---:|
| stick | `8,712,264` | `8,723,296` | `+11,032` |
| `E_cycle` | `18,363,298` | `18,384,272` | `+20,974` |
| `Ehat` | `106,512,227` | `106,528,482` | `+16,255` |
| Сумма | `133,587,789` | `133,636,050` | `+48,261` |

Чтение runtime-кода немного дороже хеширования уже переданного blob. Но это
сравнение еще не учитывает стоимость доставки blob в транзакции.

### 5.2. Calldata

Для calldata действует правило EVM: нулевой байт стоит `4 gas`, ненулевой
байт -- `16 gas`.

| Часть | Commitment bytes | Code-shards bytes | Commitment calldata gas | Code-shards calldata gas |
|---|---:|---:|---:|---:|
| stick | `57,892` | `516` | `418,660` | `3,960` |
| `E_cycle` | `116,068` | `516` | `839,068` | `3,960` |
| `Ehat` | `100,548` | `644` | `1,071,468` | `7,184` |
| Сумма | `274,508` | `1,676` | `2,329,196` | `15,104` |

### 5.3. Суммарная оценка вызовов

Без одинаковой для обоих режимов базовой стоимости трех транзакций:

```text
calldata + commitment = 133,587,789 + 2,329,196 = 135,916,985 gas;
fixed code-shards      = 133,636,050 +    15,104 = 133,651,154 gas.
```

Экономия fixed code-shards:

```text
135,916,985 - 133,651,154 = 2,265,831 gas.
```

## 6. Выбор

Для основного fixed-cache режима выбран `Lollipop305FixedShardsVerifier`.
Execution gas двух подходов близок, но code-shards исключают повторную передачу
примерно `273 KB` подготовленных линий и уменьшают суммарную оценку примерно на
`2.27M gas`.

Commitment-вариант сохранен только в `research_variants/`: он нужен для
воспроизводимого сравнения и демонстрирует безопасную альтернативу, если кэш
нельзя зафиксировать в data-контрактах.
