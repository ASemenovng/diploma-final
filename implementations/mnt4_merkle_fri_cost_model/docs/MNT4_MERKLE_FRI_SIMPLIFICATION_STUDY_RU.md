# Исследование упрощения Merkle/FRI-проверки цикла Миллера для MNT4-753

## 1. Вопрос исследования

Нужно проверить, можно ли заменить последовательное исполнение цикла Миллера
в EVM более короткой прозрачной проверкой:

\[
e(P,Q)e(-R,S)=1,
\]

где \(Q,S\in G_2\) фиксированы для развернутого verifier-а, а
\(P,R\in G_1\) передаются при вызове.

Текущий оптимизированный ориентир:

| Реализация | Gas |
|---|---:|
| Article640 fixed-shards verifier | около \(93.9\) млн |

Цель этого документа — отделить:

1. математически корректные способы уменьшения работы verifier-а;
2. технические оптимизации EVM;
3. варианты, которые только выглядят дешевыми, но теряют soundness.

## 2. Итоговый вывод

Формально доказать невозможность всего класса `Merkle + FRI` нельзя.
Предыдущая row-major формализация была корректной, но неудачной: она
раскрывала слишком широкую строку трассы для каждого случайного запроса.
Ее нижняя оценка только по calldata уже превышала стоимость текущего
Article640 fixed-shards verifier-а.

Существует проверенное упрощение, которое имеет смысл реализовать как
экспериментальный модуль:

1. использовать сжатые делители для блоков шагов Миллера;
2. не раскрывать полную трассу вычисления значений этих делителей;
3. применить column-wise efficient evaluation из статьи *On Proving
   Pairings*;
4. объединить проверки арифметики расширенного поля одним случайным
   коэффициентом;
5. коммититься только к небольшому числу столбцов;
6. применять Merkle-коммитменты и FRI к узкому набору полиномов;
7. хранить неизменяемые коэффициенты для фиксированных \(Q,S\) в
   `code-shards`, а не повторять их в calldata.

Для этой модели предварительная нижняя оценка не запрещает выигрыш относительно
\(93.9\) млн gas. Это не доказательство того, что итоговый Solidity verifier
будет дешевле. Это достаточное основание написать Rust cost-model и только
после него принимать решение о реализации verifier-а.

## 3. Почему предыдущая формализация была слишком дорогой

В документе
`MNT4_BLOCK_COMPRESSED_ORDINARY_FRI_FORMAL_SPEC_RU.md`
проверка строилась как явная трасса Горнера. Для каждого случайно выбранного
положения раскрывались:

- коэффициенты четырех полиномов делителя;
- состояния четырех вычислений Горнера;
- состояния аккумулятора цикла Миллера;
- вспомогательные столбцы;
- quotient-значения;
- селекторы.

После разбиения трассы на три сегмента одно первичное раскрытие содержало не
менее \(156\) элементов \(\mathbb F_q\). Элемент поля MNT4-753 занимает
\(96\) байт:

\[
156\cdot 96=14\,976\text{ байт}.
\]

Для консервативного production-профиля обычного FRI требовалось
\(576\) запросов:

\[
14\,976\cdot576=8\,626\,176\text{ байт}.
\]

В худшем случае ненулевой calldata это дает:

\[
8\,626\,176\cdot16
=
138\,018\,816\text{ gas}.
\]

В эту оценку не входят:

- соседние раскрытия;
- Merkle-frontier;
- слои FRI;
- Keccak;
- арифметика verifier-а.

Следовательно, такую row-major реализацию писать на Solidity бессмысленно:
она проигрывает Article640 fixed-shards verifier-у еще до исполнения
арифметики.

## 4. Проверенные источники упрощения

### 4.1. Сжатие нескольких шагов Миллера

Статья *On Proving Pairings* показывает, что несколько раундов цикла Миллера
можно объединить. Для двух последовательных раундов:

\[
f_{i+2}(P)
=
f_i(P)^4
\cdot
\ell_i(P)^2
\cdot
\ell_{i+1}(P).
\]

Для блока из \(d\) раундов:

\[
f_{i+d}(P)
=
f_i(P)^{2^d}
\cdot
g_i(P),
\]

где:

\[
g_i
=
\prod_{j=0}^{d-1}
\ell_{i+j}^{\,2^{d-1-j}}.
\]

В MNT4-753 используется \(d=5\):

\[
f_{b+1}
=
f_b^{32}
\cdot
g_b^Q(P)
\cdot
g_b^S(-R)
\cdot
c^{\kappa_b}.
\]

Здесь \(c\) — witness residue-проверки финальной экспоненты,
а \(\kappa_b\) — заранее определенный показатель для данного блока.

### 4.2. Замена линий сжатыми делителями

Для фиксированной точки второго аргумента \(U\in\{Q,S\}\) блоковый делитель
имеет вид:

\[
g_b^U(x,y)
=
a_b^U(x)
+
y\,b_b^U(x).
\]

Коэффициенты \(a_b^U,b_b^U\) вычисляются один раз Rust-бэкендом для
фиксированных \(Q,S\). Они являются конфигурацией verifier-а, а не
динамическим witness.

Для блока из пяти раундов статья дает верхнюю границу числа коэффициентов:

\[
n_b<2^{d+2}+1=129.
\]

Это не означает, что все \(129\) коэффициентов нужно передавать при каждом
запросе. Они могут храниться в неизменяемых `code-shards`.

### 4.3. Column-wise efficient evaluation

Главное упрощение находится в разделе `Efficient Polynomial Evaluation`
статьи *On Proving Pairings*.

Пусть коэффициенты всех требуемых многочленов последовательно записаны в
один полином:

\[
C(T).
\]

Для вычисления значения конкретного полинома в точке
\(\xi\in\{P_x,R_x\}\) prover строит столбец Горнера:

\[
E_\xi(T).
\]

Пусть \(H\) — домен трассы, \(Z_H(T)\) — его vanishing polynomial,
\(g\) — генератор домена, а \(h(T)\) — фиксированный selector начала
каждой группы коэффициентов.

Вместо раскрытия каждой строки трассы verifier проверяет два полиномиальных
отношения:

\[
h(T)\bigl(C(T)-E_\xi(T)\bigr)
=
q_{0,\xi}(T)Z_H(T),
\]

\[
\bigl(1-h(T)\bigr)
\Bigl(
E_\xi(T)
-
\bigl(E_\xi(g^{-1}T)\xi+C(T)\bigr)
\Bigr)
=
q_{1,\xi}(T)Z_H(T).
\]

Первое отношение задает начало вычисления Горнера. Второе проверяет переход:

\[
e_{j+1}=e_j\xi+c_j.
\]

После challenge \(\gamma\) два отношения объединяются:

\[
B_\xi(T)
=
\frac{
h(T)\bigl(C(T)-E_\xi(T)\bigr)
+
\gamma(1-h(T))
\Bigl(
E_\xi(T)
-
\bigl(E_\xi(g^{-1}T)\xi+C(T)\bigr)
\Bigr)
}{
Z_H(T)
}.
\]

Verifier открывает только:

\[
B_\xi(v),\quad
h(v),\quad
C(v),\quad
E_\xi(v),\quad
E_\xi(g^{-1}v)
\]

в случайной точке \(v\).

Это заменяет раскрытие широкой строки трассы постоянным числом полевых
элементов.

### 4.4. Единый quotient для арифметики расширенного поля

Пусть расширение задано:

\[
\mathbb F_{q^4}
\simeq
\mathbb F_q[X]/(X^4-13).
\]

Для каждого проверяемого умножения prover строит многочлены
\(A_i(X),B_i(X),C_i(X)\) и quotient:

\[
A_i(X)B_i(X)-C_i(X)
=
Q_i(X)(X^4-13).
\]

Статья предлагает не проверять каждый \(Q_i\) отдельно. После challenge
\(\beta\) отношения объединяются:

\[
\sum_i
\beta^i
\bigl(A_i(X)B_i(X)-C_i(X)\bigr)
=
Q(X)(X^4-13).
\]

В случайной точке \(\alpha\) verifier проверяет:

\[
\sum_i
\beta^i
\bigl(A_i(\alpha)B_i(\alpha)-C_i(\alpha)\bigr)
=
Q(\alpha)(\alpha^4-13).
\]

Так число quotient-раскрытий не растет линейно с числом проверяемых
умножений.

## 5. Итоговая компактная модель

### 5.1. Неизменяемая конфигурация verifier-а

Для фиксированных \(Q,S\) один раз вычисляются:

- сжатые делители \(g_b^Q,g_b^S\);
- лента коэффициентов \(C(T)\);
- selector \(h(T)\);
- доменные параметры;
- корни Merkle-деревьев или адреса `code-shards`;
- digest конфигурации.

При вызове эти данные не должны повторяться в calldata.

### 5.2. Динамический witness

Rust prover для входов \(P,R\) строит:

- столбцы вычисления значений \(E_{P_x},E_{R_x}\);
- значения сжатых делителей \(g_b^Q(P),g_b^S(-R)\);
- состояния блокового аккумулятора Миллера;
- residue-witness \(c,c^{-1}\);
- один объединенный quotient арифметики расширенного поля;
- композиционные quotient-полиномы;
- FRI-доказательство малой степени.

### 5.3. Что проверяет Solidity verifier

Контракт:

1. проверяет конфигурацию фиксированных \(Q,S\);
2. воспроизводит Fiat-Shamir challenges;
3. проверяет Merkle multiproof;
4. проверяет column-wise отношения вычисления сжатых делителей;
5. проверяет объединенное отношение арифметики \(\mathbb F_{q^4}\);
6. проверяет блоковые переходы аккумулятора Миллера;
7. проверяет residue-отношение вместо полной финальной экспоненты;
8. проверяет FRI proof малой степени.

Контракт не должен:

- исполнять все раунды цикла Миллера;
- повторно вычислять фиксированные делители;
- принимать произвольные значения делителей без PCS-opening;
- выполнять полную финальную экспоненту;
- раскрывать полную широкую строку трассы.

## 6. Ordinary FRI и DEEP-FRI

### 6.1. Обычный FRI

FRI доказывает, что закоммиченная таблица близка к таблице значений
многочлена ограниченной степени, то есть к коду Рида-Соломона.

Преимущества:

- прозрачность;
- отсутствие trusted setup;
- простая базовая модель;
- хороший reference baseline.

Недостаток для EVM: при строгой консервативной оценке надежности число
запросов может быть большим. Для параметров старой модели
\(N=32768,\rho=1/8\) production-профиль содержал \(576\) запросов.

### 6.2. DEEP-FRI

DEEP-FRI использует раскрытия вне исходного evaluation domain. Авторы
DEEP-FRI доказывают улучшение proximity gap примерно от
\(1-\sqrt{\rho}\) в сторону \(1-\rho\).

DEEP-FRI проверяет то же low-degree утверждение. Он не меняет математику
MNT4-сопряжения.

Потенциальное преимущество:

- меньше запросов при той же надежности.

Цена:

- OOD-значения;
- дополнительная композиция;
- дополнительные инверсии и умножения над \(\mathbb F_q\).

Поэтому нельзя заранее утверждать, что DEEP-FRI будет дешевле в EVM.
Нужно измерить оба профиля на одной и той же узкой column-wise трассе.

## 7. Предварительная gas-модель узкой схемы

### 7.1. FRI-раскрытия

Рассмотрим ordinary FRI:

\[
32768\to16384\to\ldots\to16.
\]

Это \(11\) слоев свертки.

Для каждого запроса verifier открывает пару значений на каждом слое.
Компактный Merkle multiproof дедуплицирует совпадающие позиции и frontier.

Детерминированная модель для \(576\) запросов дает:

| Показатель | Значение |
|---|---:|
| Уникальные FRI field values | \(6994\) |
| Frontier hashes | \(13352\) |
| FRI payload | \(1\,098\,688\) байт |
| Верхняя оценка calldata FRI payload | \(17\,579\,008\) gas |

Эта оценка включает только FRI field values и Merkle frontier. В нее не
входят source openings и исполнение verifier-а.

Оценка получена небольшой детерминированной моделью multiproof: для каждого
слоя берутся уникальные пары раскрываемых позиций, после чего считается
минимальный frontier бинарного Merkle-дерева. Она не является измерением
готового Solidity-контракта.

### 7.2. Арифметика свертки FRI

Одна свертка требует ориентировочно двух умножений в \(\mathbb F_q\).
Консервативный verifier может обработать \(576\) путей на всех
\(11\) слоях независимо:

\[
576\cdot11\cdot2
=
12\,672
\]

умножений в \(\mathbb F_q\).

Измеренная стоимость оптимизированного MNT4-умножения:

\[
G_{\mathrm{mul},q}=2959\text{ gas}.
\]

Только умножения свертки дают:

\[
12\,672\cdot2959
=
37\,496\,448\text{ gas}.
\]

При multiproof-дедупликации одинаковых позиций достаточно вычислить
\(3497\) уникальных сверток:

\[
3497\cdot2
=
6994
\]

умножения. Для оптимизированного потокового verifier-а это дает:

\[
6994\cdot2959
=
20\,695\,246\text{ gas}.
\]

### 7.3. Предварительный диапазон стоимости

Консервативная сумма двух крупных компонентов без дедупликации арифметики:

\[
17\,579\,008
+
37\,496\,448
=
55\,075\,456\text{ gas}.
\]

При дедупликации арифметики предварительная сумма уменьшается:

\[
17\,579\,008
+
20\,695\,246
=
38\,274\,254\text{ gas}.
\]

Обе величины не являются итоговой стоимостью и не являются формальной
нижней границей для всего verifier-а. Это предварительный диапазон двух
крупных компонентов для смоделированного multiproof. Еще нужны:

- source openings;
- Keccak;
- проверка column-wise relations;
- проверка арифметики расширенного поля;
- residue relation;
- ABI и control-flow overhead.

Но это существенно лучше предыдущей row-major оценки:

\[
138\,018\,816\text{ gas}
\]

только на первичную calldata.

### 7.4. Сравнение профилей

| `query_count` | FRI calldata, worst-case | Умножения без дедупликации | Gas без дедупликации |
|---:|---:|---:|---:|
| 32 | \(2\,229\,248\) | \(704\) | \(2\,083\,136\) |
| 64 | \(3\,812\,352\) | \(1408\) | \(4\,166\,272\) |
| 96 | \(5\,061\,632\) | \(2112\) | \(6\,249\,408\) |
| 128 | \(6\,292\,480\) | \(2816\) | \(8\,332\,544\) |
| 256 | \(10\,141\,696\) | \(5632\) | \(16\,665\,088\) |
| 576 | \(17\,579\,008\) | \(12\,672\) | \(37\,496\,448\) |

Эта таблица показывает основную проблему обычного FRI: строгий
production-профиль может приблизиться к текущему fixed-shards verifier-у
из-за числа запросов. Поэтому DEEP-FRI имеет смысл исследовать не как
обязательный слой, а как способ уменьшить `query_count`.

## 8. Какие упрощения допустимы

| Упрощение | Корректно | Почему |
|---|---|---|
| `code-shards` для фиксированных коэффициентов \(Q,S\) | Да | Данные неизменяемы и входят в конфигурацию verifier-а |
| Вычислять selector из номера позиции | Да | Selector детерминирован доменом |
| Column-wise evaluation вместо полной трассы Горнера | Да | Это IOP из статьи *On Proving Pairings* |
| Один quotient после challenge \(\beta\) | Да | Это случайная линейная комбинация отношений расширенного поля |
| Merkle multiproof | Да | Меняется только формат доставки раскрытий |
| Batch нескольких oracle-полиномов одной случайной линейной комбинацией | Да | Требуется domain separation и Fiat-Shamir challenge после commitments |
| Ordinary FRI вместо DEEP-FRI | Да | При наличии численной soundness-оценки и достаточного числа запросов |
| DEEP-FRI вместо ordinary FRI | Да | При полной спецификации OOD-композиции и численной soundness-оценке |

## 9. Какие упрощения недопустимы

| Идея | Почему некорректна |
|---|---|
| Проверять только Merkle-root | Root связывает verifier с таблицей, но не доказывает малую степень |
| Принимать \(g_b^Q(P),g_b^S(-R)\) без opening proof | Prover сможет подогнать множители под ложный переход |
| Проверять несколько произвольных строк без FRI/PCS | Это эвристическая spot-check, а не формальный low-degree proof |
| Объявить 32 или 64 запроса production-профилем без soundness-теоремы | Число запросов должно следовать из доказанной оценки |
| Оставить широкую row-major строку | Calldata проигрывает текущему verifier-у до исполнения арифметики |

## 10. Альтернативы с другим trade-off

### 10.1. KZG over BN254

KZG дает короткие on-chain openings благодаря precompile BN254. Но проверка
MNT4-отношений внутри BN254-circuit снова требует non-native арифметики.

Преимущество:

- короткая EVM-проверка.

Недостаток:

- рост constraints в off-chain prover-е.

Это полезный сравнительный режим, но не прозрачная замена Merkle/FRI.

### 10.2. Рекурсивное сжатие STARK-proof

Можно построить прозрачное доказательство MNT4-трассы, а затем рекурсивно
сжать его до короткого proof для EVM.

Преимущество:

- дешевый on-chain endpoint.

Недостаток:

- дополнительный рекурсивный слой;
- рост off-chain сложности;
- отдельная задача выбора внешней proof system.

### 10.3. Прямая проверка fixed-shards

Текущий Article640 verifier читает фиксированный prepared cache и выполняет
арифметику цикла Миллера непосредственно.

Преимущество:

- простая формальная модель;
- отсутствие PCS-слоя;
- измеренный результат около \(93.9\) млн gas.

Недостаток:

- стоимость все еще выше лимита блока.

## 11. Реализации, на которые можно ориентироваться

### 11.1. Stone Prover

Официальный репозиторий StarkWare содержит STARK prover и verifier для AIR
программ Cairo. Он подтверждает практическую архитектуру:

- вычислительная трасса;
- полиномиальные ограничения;
- Merkle-коммитменты;
- FRI;
- проверка случайных раскрытий.

Репозиторий:

https://github.com/starkware-libs/stone-prover

### 11.2. Winterfell

Winterfell содержит отдельные Rust-модули:

- `air`;
- `fri`;
- `crypto` с хешами и Merkle-деревьями;
- `prover`;
- `verifier`.

Это удобный ориентир для структуры Rust cost-model и тестового prover-а.
Сам репозиторий предупреждает, что проект не проходил аудит и не является
production-ready.

Репозиторий:

https://github.com/facebook/winterfell

### 11.3. Ограничение применимости примеров

Эти реализации нельзя механически подключить к MNT4-verifier-у и получить
дешевый EVM-контракт. Они показывают проверенную структуру прозрачного
доказательства. В настоящей задаче каждое умножение verifier-а выполняется
над трехсловным полем MNT4-753, поэтому требуется отдельная gas-модель.

## 12. Решение о следующем шаге

Не нужно сразу писать Solidity verifier. Сначала нужен Rust cost-model
компактной column-wise схемы.

### 12.1. Обязательные режимы Rust-прототипа

1. `ordinary-fri-strict`:
   - доказанная надежность не хуже \(2^{-128}\);
   - консервативный production baseline.
2. `ordinary-fri-benchmark`:
   - несколько значений `query_count`;
   - только для построения кривой стоимости.
3. `deep-fri-strict`:
   - точная OOD-композиция;
   - численная soundness-оценка;
   - сравнение с ordinary FRI.

### 12.2. Что должен вывести cost-model

- число oracle-полиномов;
- степени всех quotient-полиномов;
- размер LDE-домена;
- число FRI-слоев;
- `query_count`;
- итоговую soundness-оценку;
- число source openings;
- число FRI openings;
- размер Merkle frontier;
- размер calldata;
- число умножений, возведений в квадрат и инверсий в \(\mathbb F_q\);
- число Keccak;
- нижнюю оценку gas;
- оценку gas с ABI и control-flow overhead.

### 12.3. Stop/go критерий

Solidity verifier имеет смысл реализовывать, только если Rust cost-model
показывает:

\[
G_{\mathrm{expected}}
<
93\,879\,746
\]

и желательно:

\[
G_{\mathrm{expected}}
<
60\,000\,000.
\]

Если даже оптимистичная нижняя оценка компактной схемы превышает
\(93.9\) млн gas, прозрачное направление для MNT4-753 следует закрыть
как отрицательный результат.

## 13. Что можно доказать уже сейчас

### Утверждение 1

Предыдущая row-major Merkle/FRI-схема не может быть существенно дешевле
текущего Article640 fixed-shards verifier-а.

### Обоснование

Ее calldata lower bound:

\[
138\,018\,816\text{ gas},
\]

а измеренная стоимость Article640 fixed-shards verifier-а:

\[
\approx93.9\text{ млн gas}.
\]

### Утверждение 2

Нельзя утверждать, что любой Merkle/FRI-подход для MNT4-753 непрактичен.

### Обоснование

Column-wise evaluation из статьи уменьшает число раскрываемых динамических
значений с широкой строки трассы до постоянного числа значений на
полиномиальное отношение. Предварительный диапазон двух крупных компонентов
узкой схемы:

\[
\approx38.3\text{--}55.1\text{ млн gas}
\]

без учета оставшихся компонентов. Предварительная модель уже не исключает
выигрыш.

### Утверждение 3

Обычный FRI не гарантирует практический результат для MNT4-753.

### Обоснование

При консервативном strict-профиле `query_count = 576` основная стоимость
возникает из повторяющихся умножений над трехсловным полем MNT4-753.
Окончательный ответ требует Rust cost-model. DEEP-FRI необходимо проверить
как альтернативный профиль с меньшим числом запросов.
