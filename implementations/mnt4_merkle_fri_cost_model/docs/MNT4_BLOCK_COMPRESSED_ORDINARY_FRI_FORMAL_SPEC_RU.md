# Формальная спецификация сжатой Merkle/FRI-проверки MNT4-753

## 1. Статус документа

Документ задает математически замкнутую модель следующего экспериментального
модуля. Реализация должна проверять уравнение:

\[
e(P,Q)e(-R,S)=1,
\]

где:

- \(P,R\in G_1\) передаются при вызове;
- \(Q,S\in G_2\) фиксируются для конкретного развернутого verifier-а;
- вычислительная трасса строится Rust-бэкендом;
- EVM-контракт проверяет Merkle-раскрытия, полиномиальные отношения и обычное
  FRI-доказательство малой степени;
- контракт не исполняет последовательно полный цикл Миллера;
- контракт не вычисляет полную финальную экспоненту.

Этот протокол является отдельной STARK-подобной EVM-инстанциацией идей статьи
*On Proving Pairings*. Статья дает сжатие цикла Миллера и замену линий
делителями для фиксированного второго аргумента сопряжения. Настоящий документ
дополнительно задает прозрачный Merkle/FRI-слой.

## 2. Публичные параметры и предположения

Используется MNT4-753:

\[
\mathbb F_q,
\qquad
\mathbb F_{q^4}\simeq\mathbb F_q[Z]/(Z^4-13).
\]

Точки:

\[
P=(P_x,P_y),\qquad R=(R_x,R_y)
\]

принадлежат \(G_1\). Фиксированные точки:

\[
Q,S\in G_2
\]

задаются конфигурацией verifier-а.

Контракт хранит:

```text
configDigest
rootFixedDivisors
rootFixedSelectors
```

Корни фиксированных таблиц вычисляются однократно Rust-бэкендом при подготовке
развертывания. Они являются частью конфигурации verifier-а, аналогично
зафиксированным `code-shards` в текущей реализации Article640.

Предположение о хеш-функции:

```text
Keccak256 является collision resistant.
```

Fiat-Shamir-преобразование анализируется в модели случайного оракула.

## 3. Residue-постановка

Вместо вычисления полной финальной экспоненты используется свидетельство:

\[
c\in\mathbb F_{q^4}^{*},
\qquad
c^{-1}\in\mathbb F_{q^4}^{*}.
\]

Контракт проверяет:

\[
cc^{-1}=1.
\]

Далее \(c\), \(c^{-1}\) и образ Фробениуса:

\[
\operatorname{Frob}_q(c^{-1})
\]

встраиваются в модифицированную рекурсию цикла Миллера. Финальное состояние
должно быть равно:

\[
1.
\]

Корректность этой замены совпадает с residue-проверкой Article640-модуля и не
изменяется при переходе от пошаговой трассы к блочной.

## 4. Сжатие пяти шагов цикла Миллера

### 4.1. Исходная рекурсия

Один шаг цикла имеет вид:

\[
F_{i+1}=F_i^2M_i.
\]

Множитель \(M_i\) содержит вычисленные в \(P\) и \(-R\) линии, а для
ненулевого знака шага также соответствующую степень \(c\) или \(c^{-1}\).

### 4.2. Блочная рекурсия

Для пяти последовательных шагов:

\[
F_{i+5}
=
F_i^{32}
M_i^{16}
M_{i+1}^{8}
M_{i+2}^{4}
M_{i+3}^{2}
M_{i+4}.
\]

Статья предлагает заменить произведение линий подготовленным делителем. Для
фиксированных \(Q,S\) Rust-бэкенд однократно строит:

\[
g_b^Q(x,y)=a_b^Q(x)+y b_b^Q(x),
\]

\[
g_b^S(x,y)=a_b^S(x)+y b_b^S(x).
\]

Для блока \(b\):

\[
G_b^Q(P)=g_b^Q(P_x,P_y),
\]

\[
G_b^S(-R)=g_b^S(R_x,-R_y).
\]

Итоговый переход:

\[
F_{b+1}
=
F_b^{32}
G_b^Q(P)
G_b^S(-R)
c^{\kappa_b}.
\]

Показатель:

\[
\kappa_b
\]

однозначно вычисляется из пяти знаков Ate-цепочки.

### 4.3. Размер делителя

Для размера блока:

\[
d=5
\]

статья дает:

\[
n_b<2^{d+2}+1=129.
\]

Следовательно, порядок полюса не превосходит:

\[
128.
\]

На короткой форме Вейерштрасса:

\[
\operatorname{ord}_{O}(x)=-2,
\qquad
\operatorname{ord}_{O}(y)=-3.
\]

Поэтому достаточно:

\[
\deg a_b^U\leq64,
\qquad
\deg b_b^U\leq62,
\qquad
U\in\{Q,S\}.
\]

Используются канонические дополненные массивы:

\[
a_{b,0}^{U},\ldots,a_{b,64}^{U},
\]

\[
b_{b,0}^{U},\ldots,b_{b,64}^{U},
\]

где:

\[
b_{b,63}^{U}=b_{b,64}^{U}=0.
\]

Каждый коэффициент принадлежит \(\mathbb F_{q^4}\) и хранится четырьмя
координатами \(\mathbb F_q\).

### 4.4. Канонический алгоритм построения делителя

Для фиксированной точки:

\[
U\in\{Q,S\}
\]

индексатор сначала строит ту же последовательность линий, которая
используется пошаговым циклом Миллера. Обозначим полный множитель одного
шага:

\[
L_i^U(x,y)
=
D_i^U(x,y)
\cdot
\begin{cases}
1, & \delta_i=0,\\
A_i^U(x,y), & \delta_i\neq0.
\end{cases}
\]

Здесь:

- \(D_i^U\) является линией удвоения;
- \(A_i^U\) является линией сложения для ненулевого знака Ate-цепочки;
- вертикальные знаменатели удаляются на основании той же леммы, которая
  используется в текущей MNT4-реализации.

Для полного блока:

\[
b
\]

из пяти шагов определяется функция:

\[
\widetilde g_b^U(x,y)
=
\prod_{h=0}^{4}
\left(
L_{5b+h}^U(x,y)
\right)^{2^{4-h}}.
\]

Далее индексатор редуцирует степени \(y\) по уравнению кривой:

\[
y^2=x^3+Ax+B.
\]

После редукции получается единственная каноническая форма:

\[
g_b^U(x,y)
=
a_b^U(x)+yb_b^U(x).
\]

Коэффициенты записываются:

1. по возрастанию степени \(x\);
2. сначала для \(a_b^U\), затем для \(b_b^U\);
3. каждый коэффициент \(\mathbb F_{q^4}\) записывается в порядке
   \(1,Z,Z^2,Z^3\);
4. отсутствующие старшие коэффициенты дополняются нулями.

Rust-бэкенд обязан проверить для каждого блока и контрольной точки \(T\):

\[
g_b^U(T)
=
\prod_{h=0}^{4}
\left(
L_{5b+h}^U(T)
\right)^{2^{4-h}}.
\]

Эта проверка выполняется off-chain и входит в тесты deployment manifest.

### 4.5. Последний неполный блок

Так как число шагов не делится на пять без остатка, индексатор отдельно
строит:

```text
partialBlockDivisorQ
partialBlockDivisorS
tailDivisorQ
tailDivisorS
```

Они фиксируются тем же deployment manifest. Количество таких объектов
постоянно. Их раскрытия и проверки не увеличивают асимптотическую стоимость
по длине цикла Миллера.

## 5. Почему нельзя принимать значение делителя на доверии

Если prover передает произвольное значение:

\[
\widetilde G_b,
\]

то он может выбрать:

\[
\widetilde G_b
=
F_{b+1}
\left(
F_b^{32}c^{\kappa_b}
\right)^{-1}
\]

и заставить ложный переход пройти проверку.

Merkle-коммитмент к динамической таблице не решает проблему: он фиксирует
выбранную таблицу, но не доказывает, что ее значения получены из
зафиксированных делителей.

Следовательно, проверка должна доказать вычисление:

\[
g_b^Q(P_x,P_y),
\qquad
g_b^S(R_x,-R_y).
\]

## 6. Трасса Горнера для вычисления делителей

### 6.1. Основная идея

Для каждого блока одновременно вычисляются четыре полинома:

\[
a_b^Q(P_x),\qquad b_b^Q(P_x),
\]

\[
a_b^S(R_x),\qquad b_b^S(R_x).
\]

Прямое раскрытие всех коэффициентов при каждом FRI-запросе не требуется.
Вместо этого prover коммитится к таблице промежуточных значений схемы Горнера.

### 6.2. Рекурсия

Для:

\[
j=64,63,\ldots,0
\]

определим:

\[
A_{b,65}^{Q}=0,
\qquad
B_{b,65}^{Q}=0,
\]

\[
A_{b,65}^{S}=0,
\qquad
B_{b,65}^{S}=0.
\]

Далее:

\[
A_{b,j}^{Q}
=
A_{b,j+1}^{Q}P_x+a_{b,j}^{Q},
\]

\[
B_{b,j}^{Q}
=
B_{b,j+1}^{Q}P_x+b_{b,j}^{Q},
\]

\[
A_{b,j}^{S}
=
A_{b,j+1}^{S}R_x+a_{b,j}^{S},
\]

\[
B_{b,j}^{S}
=
B_{b,j+1}^{S}R_x+b_{b,j}^{S}.
\]

В результате:

\[
A_{b,0}^{Q}=a_b^Q(P_x),
\qquad
B_{b,0}^{Q}=b_b^Q(P_x),
\]

\[
A_{b,0}^{S}=a_b^S(R_x),
\qquad
B_{b,0}^{S}=b_b^S(R_x).
\]

Значения делителей:

\[
G_b^Q(P)
=
A_{b,0}^{Q}
+
P_yB_{b,0}^{Q},
\]

\[
G_b^S(-R)
=
A_{b,0}^{S}
-
R_yB_{b,0}^{S}.
\]

Все равенства выполняются в:

\[
\mathbb F_{q^4}.
\]

### 6.3. Координатное представление

Каждый элемент:

\[
W\in\mathbb F_{q^4}
\]

кодируется:

\[
W(Z)=w_0+w_1Z+w_2Z^2+w_3Z^3.
\]

Поскольку:

\[
P_x,P_y,R_x,R_y\in\mathbb F_q,
\]

рекурсия Горнера выполняется покоординатно без умножений в
\(\mathbb F_{q^4}\):

\[
A_{b,j,r}^{Q}
=
A_{b,j+1,r}^{Q}P_x+a_{b,j,r}^{Q},
\qquad
0\leq r<4.
\]

Аналогично задаются остальные три аккумулятора.

Один Horner-переход использует `16` базовых умножений:

```text
4 координаты * 4 полинома = 16 Fq multiplications.
```

Контракт не выполняет все переходы. Он проверяет только случайные раскрытия
полиномиально закодированной трассы.

## 7. Единая табличная раскладка

### 7.1. Размеры и сегментация

Количество реальных блоков:

\[
B_{\mathrm{real}}=76.
\]

Для каждого блока используется stride:

\[
J=66.
\]

Внутри блока:

- строки `0..64`: Horner-переходы;
- строка `65`: проверка значения делителя и блочного перехода;
- дополнительные строки внутри сегмента: padding.

В базовом поле MNT4-753:

\[
v_2(q-1)=15.
\]

Поэтому максимальный двоичный домен имеет размер:

\[
2^{15}=32768.
\]

Единая трасса длины:

\[
76\cdot66=5016
\]

не позволяет одновременно использовать степень двойки и достаточный
коэффициент расширения домена. Поэтому трасса делится на три сегмента:

```text
segment 0: blocks  0..30
segment 1: blocks 31..61
segment 2: blocks 62..75
```

Каждый сегмент содержит не более:

\[
31\cdot66=2046
\]

реальных строк и дополняется до:

\[
N_{\mathrm{segment}}=2048.
\]

Для каждого сегмента используется LDE:

\[
N_{\mathrm{LDE}}=2048\cdot16=32768.
\]

Переходы:

```text
block 30 -> block 31
block 61 -> block 62
```

проверяются отдельными граничными раскрытиями. Их число постоянно.

### 7.2. Индексация

Внутри сегмента строка:

\[
t=b_{\mathrm{local}}J+j
\]

однозначно определяет:

\[
b_{\mathrm{local}}=\lfloor t/J\rfloor,
\qquad
j=t\bmod J.
\]

Фиксированные селекторы:

```text
sHorner(t)
sTerminal(t)
sPadding(t)
sRealBlock(t)
sFirstBlock(t)
sLastRealBlock(t)
sLastSegmentRow(t)
```

генерируются Rust-бэкендом и фиксируются `rootFixedSelectors`.

### 7.3. Фиксированные столбцы

`rootFixedDivisors` фиксирует координаты:

```text
aQ[4]
bQ[4]
aS[4]
bS[4]
```

для каждой строки Horner-трассы. В padding-строках они равны нулю.

Таблица зависит только от фиксированных \(Q,S\) и Ate-цепочки. Пользователь
не может заменить ее при вызове.

### 7.4. Динамические столбцы

До получения испытания \(\zeta\) для одного proof коммитятся базовые
динамические столбцы:

```text
accAQ[4]
accBQ[4]
accAS[4]
accBS[4]
stateF[4]
```

После получения \(\zeta\) prover строит скалярные вспомогательные столбцы:

```text
u1
u2
u3
u4
u5
vQ
vS
```

Их значения принадлежат \(\mathbb F_q\). Они являются вычислениями
соответствующих элементов \(\mathbb F_{q^4}\) в случайной точке \(Z=\zeta\).
Это уменьшает ширину строки: вместо `28` координат \(\mathbb F_q\)
передаются `7` скаляров.

После получения \(\beta\) prover строит:

```text
jMix
quotientSegments[]
```

`jMix` является единственным объединенным частным значением по модулю
\(Z^4-13\). Отдельные столбцы `j1..j8` не передаются.

## 8. Отношения трассы

### 8.1. Horner-отношения

На строках с:

\[
s_{\mathrm{Horner}}=1
\]

для каждой координаты \(r\) проверяются:

\[
\operatorname{accAQ}_{r}(\omega X)
-
\left(
\operatorname{accAQ}_{r}(X)P_x+a_{r}^{Q}(X)
\right)
=0,
\]

\[
\operatorname{accBQ}_{r}(\omega X)
-
\left(
\operatorname{accBQ}_{r}(X)P_x+b_{r}^{Q}(X)
\right)
=0,
\]

\[
\operatorname{accAS}_{r}(\omega X)
-
\left(
\operatorname{accAS}_{r}(X)R_x+a_{r}^{S}(X)
\right)
=0,
\]

\[
\operatorname{accBS}_{r}(\omega X)
-
\left(
\operatorname{accBS}_{r}(X)R_x+b_{r}^{S}(X)
\right)
=0.
\]

На первой Horner-строке блока аккумуляторы равны нулю.

На Horner-строках состояние:

\[
F
\]

не меняется. На терминальной строке блока состояние следующего блока читается
из следующей строки:

\[
F_{\mathrm{next}}(X)=F(\omega X).
\]

Для последней строки сегмента вместо \(\omega X\) используется отдельное
граничное раскрытие первой строки следующего сегмента.

### 8.2. Значение делителя

На терминальной строке:

\[
G^{Q}(X,Z)
=
\operatorname{accAQ}(X,Z)
+
P_y\operatorname{accBQ}(X,Z),
\]

\[
G^{S}(X,Z)
=
\operatorname{accAS}(X,Z)
-
R_y\operatorname{accBS}(X,Z).
\]

### 8.3. Разложение блочного перехода

После фиксации базовой трассы transcript выводит:

\[
\zeta\in\mathbb F_q.
\]

Каждый элемент расширенного поля:

\[
W(Z)=w_0+w_1Z+w_2Z^2+w_3Z^3
\]

вычисляется в точке:

\[
W(\zeta).
\]

Чтобы степень отношений по переменной трассы оставалась небольшой,
используются скалярные промежуточные столбцы:

\[
U_1=F^2,
\]

\[
U_2=U_1^2,
\]

\[
U_3=U_2^2,
\]

\[
U_4=U_3^2,
\]

\[
U_5=U_4^2,
\]

\[
V_Q=U_5G^Q,
\]

\[
V_S=V_QG^S,
\]

\[
F_{\mathrm{next}}=V_SC_{\kappa}.
\]

Здесь:

\[
C_{\kappa}=c^{\kappa_b}.
\]

Все значения `u1..u5`, `vQ`, `vS` принадлежат \(\mathbb F_q\) и коммитятся
после получения \(\zeta\). Небольшой набор приведенных элементов:

\[
\{C_{\nu}:\nu\in\mathcal K\}
\]

строится prover-ом и проверяется один раз на proof короткой цепочкой
умножений по тому же правилу случайной подстановки \(Z=\zeta\).

### 8.4. Проверка умножения в расширении

Для каждого отношения:

\[
W_1W_2=W_3
\quad\text{в}\quad
\mathbb F_{q^4}
\]

backend строит полиномиальное равенство:

\[
W_1(Z)W_2(Z)-W_3(Z)
=
J(Z)(Z^4-13),
\]

где:

\[
\deg J\leq2.
\]

Verifier проверяет в уже выведенной случайной точке:

\[
W_1(\zeta)W_2(\zeta)-W_3(\zeta)
=
J(\zeta)(\zeta^4-13).
\]

Восемь отношений терминальной строки объединяются challenge:

\[
\beta\in\mathbb F_q.
\]

Вместо восьми независимых значений частных многочленов prover передает одно:

\[
J_{\mathrm{mix}}(\zeta)
=
\sum_{i=1}^{8}
\beta^{i-1}J_i(\zeta).
\]

Verifier проверяет:

\[
\sum_{i=1}^{8}
\beta^{i-1}
\left(
W_{i,1}(\zeta)W_{i,2}(\zeta)-W_{i,3}(\zeta)
\right)
=
J_{\mathrm{mix}}(\zeta)(\zeta^4-13).
\]

Таким образом:

1. on-chain используются умножения в \(\mathbb F_q\), а не полные
   умножения в \(\mathbb F_{q^4}\);
2. передается один `jMix`, а не восемь наборов коэффициентов;
3. неверное отношение проходит проверку по случайным \(\zeta,\beta\) только
   с пренебрежимо малой вероятностью.

### 8.5. Граничные отношения

Отдельно фиксируются:

1. начальное состояние:

   \[
   F_0=c^{-1};
   \]

2. последний неполный блок;
3. завершающие линии отрицательного параметра;
4. умножение на:

   \[
   \operatorname{Frob}_q(c^{-1});
   \]

5. финальное состояние:

   \[
   F_{\mathrm{final}}=1;
   \]

6. обратимость:

   \[
   cc^{-1}=1.
   \]

Количество этих отношений постоянно и не зависит от длины цикла Миллера.

## 9. Композиционный полином

### 9.1. Separation challenge

После фиксации:

```text
rootFixedDivisors
rootFixedSelectors
rootTraceBase
```

transcript выводит:

\[
\zeta\in\mathbb F_q.
\]

После фиксации:

```text
rootTraceAux
```

transcript выводит:

\[
\beta\in\mathbb F_q.
\]

Все локальные отношения:

\[
\mathcal R_0(X,\zeta),
\ldots,
\mathcal R_{m-1}(X,\zeta)
\]

объединяются:

\[
\mathcal N_{\mathrm{mix}}(X)
=
\sum_{i=0}^{m-1}
\beta^i\mathcal R_i(X,\zeta).
\]

### 9.2. Частный многочлен

Для каждого сегмента отдельно используется домен:

\[
H=\{1,\omega,\ldots,\omega^{2047}\}.
\]

Его полином обращения в ноль:

\[
Z_H(X)=X^{2048}-1.
\]

Для честной трассы каждого сегмента:

\[
\mathcal N_{\mathrm{mix},s}(X)
=
Q_{\mathrm{trace},s}(X)Z_H(X),
\qquad
s\in\{0,1,2\}.
\]

Backend коммитится к объединенной row-major таблице:

```text
rootQuotient
```

после получения \(\beta\) и \(\zeta\).

В случайной позиции \(x\) контракт проверяет три отношения:

\[
\mathcal N_{\mathrm{mix},s}(x)
=
Q_{\mathrm{trace},s}(x)(x^{2048}-1),
\qquad
s\in\{0,1,2\}.
\]

Отношения сегментов объединяются дополнительными степенями challenge
\(\beta\). Поэтому один случайный запрос проверяет все три сегмента.

## 10. Ordinary FRI

### 10.1. Что доказывается

Merkle-коммитменты фиксируют таблицы, но не доказывают их малую степень.
Ordinary FRI подтверждает близость батчированной таблицы к коду
Рида-Соломона.

После challenge:

\[
\eta\in\mathbb F_q
\]

backend строит батчированный полином:

\[
B(X)
=
\sum_{s=0}^{2}\eta^sQ_{\mathrm{trace},s}(X)
+
\sum_{s=0}^{2}
\sum_{i=0}^{n-1}
\eta^{3+sn+i}T_{s,i}(X),
\]

где:

\[
T_{s,i}
\]

перечисляют базовые и вспомогательные динамические столбцы трех сегментов.

В каждой случайной позиции verifier самостоятельно вычисляет:

\[
B(x)
\]

из раскрытых строк и связывает его с первым FRI-слоем.

### 10.2. Один FRI-раунд

Для:

\[
B(X)=B_{\mathrm{even}}(X^2)+XB_{\mathrm{odd}}(X^2)
\]

и challenge:

\[
\rho_i
\]

следующий слой:

\[
B_{\mathrm{next}}(X^2)
=
B_{\mathrm{even}}(X^2)
+
\rho_iB_{\mathrm{odd}}(X^2).
\]

В точках \(x,-x\):

\[
B_{\mathrm{next}}(x^2)
=
\frac{B(x)+B(-x)}{2}
+
\rho_i
\frac{B(x)-B(-x)}{2x}.
\]

### 10.3. Fiat-Shamir-порядок

Используется последовательность:

```text
public inputs
rootFixedDivisors
rootFixedSelectors
rootTraceBase
  -> zeta
rootTraceAux
  -> beta
rootQuotient
  -> eta
rootFri0
  -> rho0
rootFri1
  -> rho1
...
finalCoefficients
  -> querySeed
```

Позиции запросов детерминированно выводятся из:

```text
querySeed
```

и не передаются prover-ом.

## 11. Параметры домена и надежности

### 11.1. Трассовый домен

\[
N_{\mathrm{segment}}=2048.
\]

Коэффициент расширения:

\[
b=16.
\]

Размер LDE:

\[
N_{\mathrm{LDE}}=32768.
\]

Каждый исходный столбец сегмента имеет степень не выше:

\[
2047.
\]

Horner-отношение до умножения на селектор имеет степень не выше:

\[
2047.
\]

Терминальное квадратичное отношение до умножения на селектор имеет степень
не выше:

\[
2\cdot2047=4094.
\]

После умножения на селектор степени не выше `2047` композиционный полином
имеет степень не выше:

\[
4094+2047=6141.
\]

После деления на:

\[
X^{2048}-1
\]

частный многочлен имеет степень не выше:

\[
6141-2048=4093.
\]

Следовательно, ordinary FRI должен доказать степень строго меньше:

\[
4094.
\]

Для применения доказанной теоремы используется оболочка RS-кода со
скоростью:

\[
\rho=\frac{4096}{32768}=\frac18<\frac13.
\]

Она включает все полиномы степени строго меньше `4094` и соответствует
стандартной степени двойки. Rust-модуль обязан подтвердить верхнюю границу
unit-тестом. Профиль разрешается использовать только если скорость RS-кода:

\[
\rho<\frac13.
\]

### 11.2. Soundness calculator

Для доказанной оценки FRI используется:

\[
\delta_0
\geq
\frac{1-3\rho}{4}
-
\frac{1}{\sqrt{N_{\mathrm{LDE}}}}
-
\frac{3N_{\mathrm{LDE}}}{|\mathbb F_q|}.
\]

Число независимых запросов:

\[
\texttt{queryCount}
\]

выбирается минимальным с запасом так, чтобы:

\[
(1-\delta_0)^{\texttt{queryCount}}
\leq
2^{-128}.
\]

Отдельно учитываются:

1. Schwartz-Zippel по \(\zeta\);
2. случайное объединение отношений через \(\beta\);
3. случайное батчирование столбцов через \(\eta\);
4. коллизии Keccak256;
5. Fiat-Shamir в модели случайного оракула.

Для \(\rho=1/8\) получается:

\[
\delta_0\approx0.1507257.
\]

Минимальное число повторений, достаточное для оценки:

\[
(1-\delta_0)^{\texttt{queryCount}}\leq2^{-128},
\]

равно:

\[
\texttt{queryCount}_{\min}=544.
\]

Production-профиль использует запас:

\[
\texttt{queryCount}=576.
\]

Для него вклад FRI-части в оценку надежности превышает:

\[
135
\]

бит. Оставшийся запас покрывает дополнительные пренебрежимо малые слагаемые
Schwartz-Zippel, батчирования и хеш-коммитментов.

## 12. Формат коммитментов

### 12.1. Фиксированные таблицы

При подготовке развертывания строятся:

```text
rootFixedDivisors
rootFixedSelectors
deploymentManifestHash
```

`deploymentManifestHash` фиксирует:

```text
curve id
Q
S
Ate chain digest
block size
trace stride
trace domain size
LDE size
column order
serialization version
```

### 12.2. Динамическая трасса

Лист `rootTraceBase` содержит row-major запись одной LDE-позиции трех
сегментов до получения \(\zeta\):

```text
traceBaseLeaf(index) =
    Keccak256(
        DOMAIN_TRACE_BASE ||
        index ||
        base dynamic columns of segments 0,1,2 at index
    )
```

Лист `rootTraceAux` содержит row-major запись скалярных вспомогательных
столбцов, построенных после получения \(\zeta\):

```text
traceAuxLeaf(index) =
    Keccak256(
        DOMAIN_TRACE_AUX ||
        index ||
        scalar auxiliary columns of segments 0,1,2 at index
    )
```

`rootQuotient` фиксирует `jMix` и сегменты частных многочленов после
получения \(\beta\).

### 12.3. Fixed leaves

Лист фиксированной таблицы:

```text
fixedLeaf(index) =
    Keccak256(
        DOMAIN_FIXED ||
        index ||
        fixed divisor coordinates and selectors of segments 0,1,2
    )
```

### 12.4. Multiproof

Для объединения раскрытий используются отсортированные позиции и компактный
Merkle-multiproof. Общие соседние узлы передаются один раз.

## 13. Что выполняется off-chain

Rust-бэкенд:

1. строит фиксированные делители для \(Q,S\);
2. проверяет их против пошагового цикла Миллера;
3. строит manifest и фиксированные Merkle-деревья;
4. для входных \(P,R\) строит residue witness \(c,c^{-1}\);
5. строит Horner-трассу вычисления делителей;
6. строит блочные состояния аккумулятора;
7. строит промежуточные столбцы и quotient;
8. строит LDE;
9. строит ordinary FRI proof;
10. локально выполняет полную проверку proof;
11. сериализует packed calldata;
12. сохраняет метрики времени, памяти и размера proof.

## 14. Что выполняется on-chain

Контракт:

1. проверяет каноничность и принадлежность \(P,R\) кривой;
2. проверяет:

   \[
   cc^{-1}=1;
   \]

3. воспроизводит transcript;
4. выводит случайные позиции;
5. потоково проверяет Merkle-multiproof;
6. в случайных строках проверяет Horner-рекурсию;
7. в случайных строках проверяет блочную рекурсию;
8. проверяет quotient identity;
9. проверяет ordinary FRI folding;
10. проверяет финальный короткий FRI-полином;
11. возвращает `true` только при прохождении всех проверок.

Контракт не:

- исполняет все Horner-переходы;
- исполняет все шаги Миллера;
- вычисляет полную финальную экспоненту;
- доверяет значениям делителей, переданным prover-ом.

## 15. Аргумент корректности

### Теорема

Пусть:

1. Keccak256 collision resistant;
2. Fiat-Shamir анализируется в модели случайного оракула;
3. ordinary FRI принимает только таблицы, близкие к многочленам заявленной
   степени, кроме события с вероятностью \(\varepsilon_{\mathrm{FRI}}\);
4. случайные challenges равномерны в \(\mathbb F_q\).

Если verifier принимает proof, то кроме события с вероятностью:

\[
\varepsilon_{\mathrm{total}}
\leq
\varepsilon_{\mathrm{FRI}}
+
\varepsilon_{\mathrm{SZ}}
+
\varepsilon_{\beta}
+
\varepsilon_{\eta}
+
\varepsilon_{\mathrm{hash}},
\]

существует корректная residue-трасса для:

\[
e(P,Q)e(-R,S)=1.
\]

### Доказательство

1. Merkle-binding фиксирует значения всех раскрываемых таблиц.
2. FRI и батчирование через \(\eta\) связывают раскрытия с низкостепенными
   полиномами динамических столбцов.
3. Quotient identity связывает низкостепенные полиномы с локальными
   отношениями трассы.
4. Horner-отношения и фиксированный `rootFixedDivisors` означают, что
   значения \(G_b^Q(P)\), \(G_b^S(-R)\) получены из зафиксированных
   делителей, а не выбраны prover-ом.
5. Блочные отношения означают:

   \[
   F_{b+1}
   =
   F_b^{32}
   G_b^Q(P)
   G_b^S(-R)
   c^{\kappa_b}.
   \]

6. Граничные отношения связывают первый и последний блоки с residue-рекурсией.
7. Финальное состояние:

   \[
   1
   \]

   эквивалентно успешной residue-проверке уравнения сопряжений.

Следовательно, принятие некорректного уравнения возможно только при нарушении
одного из перечисленных криптографических или вероятностных предположений.

## 16. Обязательные проверки до реализации Solidity

Rust-прототип обязан:

1. построить делители \(g_b^Q,g_b^S\);
2. проверить каждый делитель против пяти пошаговых переходов;
3. вывести точные степени и число ненулевых коэффициентов;
4. построить Horner-трассу;
5. проверить ее локальные отношения;
6. построить quotient;
7. вывести точный degree bound;
8. вычислить `queryCount` для `128 bit`;
9. оценить:

   ```text
   proof bytes
   calldata gas
   Merkle hash count
   Fq multiplication count per query
   estimated execution gas
   ```

10. остановить проектирование Solidity-контракта, если предварительная модель
    заведомо превосходит стоимость прямого Article640 verifier-а.

## 17. Критерии готовности будущей реализации

Модуль считается завершенным только если:

1. фиксированные делители построены и кросс-проверены;
2. Horner-трасса реализована;
3. обычный FRI реализован без DEEP-компонентов;
4. soundness calculator выдает не менее `128 bit`;
5. Rust verifier отклоняет подмену каждого типа таблиц;
6. Solidity verifier использует потоковый parser;
7. Solidity verifier не содержит линейного прохода по полному циклу;
8. Solidity verifier отклоняет подмены публичных входов, roots, openings,
   frontier hashes, quotient и FRI layers;
9. отчет содержит реальные gas-измерения;
10. отчет честно сравнивает результат с Article640 fixed-shards и
    пошаговым DEEP-FRI baseline.

## 18. Граница результата

Формализация устраняет логический пробел прежнего черновика: значения
сжатых делителей больше не принимаются на доверии.

Она не доказывает заранее, что итоговый Solidity-verifier будет дешевым.
Таблица Горнера содержит `5016` реальных строк и кодируется тремя
сегментами с доменами по `2048` строк. Поэтому перед переносом в Solidity
обязателен Rust-прототип с точной моделью calldata и числа операций.

## 19. Предварительная нижняя оценка calldata

### 19.1. Минимальная ширина раскрытия

Даже до реализации можно получить нижнюю оценку. Для одной LDE-позиции
необходимо раскрыть данные трех сегментов.

Фиксированная строка одного сегмента содержит минимум:

```text
16 divisor coordinate values
 7 selector values
```

Для трех сегментов:

\[
3(16+7)=69
\]

элементов \(\mathbb F_q\).

Базовая динамическая строка одного сегмента:

```text
16 Horner accumulator coordinates
 4 state coordinates
```

Для трех сегментов:

\[
3(16+4)=60
\]

элементов \(\mathbb F_q\).

Скалярная вспомогательная строка одного сегмента:

```text
u1,u2,u3,u4,u5,vQ,vS
```

Для трех сегментов:

\[
3\cdot7=21
\]

элемент \(\mathbb F_q\).

Таблица quotient содержит как минимум:

```text
jMix
qTrace
```

для каждого сегмента:

\[
3\cdot2=6
\]

элементов \(\mathbb F_q\).

Следовательно, одно минимальное первичное раскрытие содержит не менее:

\[
69+60+21+6=156
\]

элементов базового поля.

Один элемент MNT4-753 занимает:

\[
96
\]

байт. Поэтому:

\[
156\cdot96=14976
\]

байт на одну LDE-позицию.

### 19.2. Нижняя оценка для production-профиля

Для production-профиля:

\[
\texttt{queryCount}=576,
\]

то даже без учета:

1. раскрытия соседней строки;
2. раскрытия точки \(-x\) для первого FRI-раунда;
3. Merkle-frontier hashes;
4. последующих FRI-слоев;
5. заголовка proof;
6. публичных входов;

получается:

\[
14976\cdot576
=
8626176
\]

байт calldata.

При верхней оценке:

\[
16
\]

gas за ненулевой байт:

\[
8626176\cdot16
=
138018816
\]

gas.

Это уже превышает текущий Article640 fixed-shards baseline порядка:

```text
93.7 million gas
```

до выполнения Merkle- и FRI-проверок.

### 19.3. Что может уменьшить оценку

Rust-прототип должен измерить фактическое число уникальных позиций после
объединения multiproof. Также можно исследовать:

1. загрузку фиксированных строк из `code-shards`;
2. выборочное раскрытие одного сегмента с пропорциональным усилением
   soundness;
3. более плотное кодирование селекторов;
4. устранение нулевых столбцов;
5. отдельный PCS с линейно-гомоморфными коммитментами.

Однако эти приемы не отменяют основной trade-off:

- прозрачный Merkle/FRI-verifier требует передавать раскрываемые значения;
- MNT4-753 использует трехсловные элементы поля;
- доказанный ordinary-FRI профиль требует большого числа запросов.

Поэтому Rust-прототип может подтвердить отрицательный результат еще до
написания Solidity-контракта.
